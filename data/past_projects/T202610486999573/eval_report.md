# TCC 操作系统内核技术评估报告

## 项目基本信息

- **项目名称**：TCC（根据内核 `uname` 系统调用返回 `"TCC"` 推断）
- **架构**：RISC‑V 64 位（Sv39）为主架构，LoongArch 64 位为辅架构
- **实现语言**：C（内核主体）、RISC‑V 汇编（进入/退出、上下文切换、陷阱入口）、LoongArch 汇编（仅入口）
- **生态归属**：独立实现的类 Unix 教学内核，借鉴 xv6 设计
- **特点**：双架构策略，包含比赛评测直通模式（内核态直接输出硬编码测试结果），三套文件系统并存（EXT4 只读层、自定义 inode 文件系统、内存文件系统），完整的 RISC‑V 陷阱与上下文切换机制

## 实现的子系统与功能

1. **启动子系统**：多核同步初始化，OpenSBI 下 S‑mode 进入，外设逐一初始化。
2. **内存管理**：物理页分配器（内核/用户区域分离），Sv39 三级页表管理，用户地址空间（含 mmap/munmap、堆增长/收缩），内核页表建立与切换，跨空间数据拷贝。
3. **进程管理**：进程结构体（含文件描述符表、当前目录、mmap 链表），进程创建/复制（fork）、退出、等待、简单轮询调度、基于睡眠/唤醒的阻塞机制。
4. **陷阱与异常处理**：用户态/kernel态两级陷阱入口，trampoline 页切换机制，时钟中断、外部中断（PLIC 派发）、系统调用分派，内核态异常诊断与进程终止。
5. **系统调用**：35 个 Linux RISC‑V ABI 兼容的系统调用，覆盖文件、进程、内存、时间、信息类。
6. **文件系统**：EXT4 只读解析器（支持 extent tree），自定义类 Unix inode 文件系统（含超级块、位图、间接块、目录），内存文件系统（RAMFS），虚拟目录（ramdir），管道（环形缓冲区）。
7. **设备驱动**：UART 16550a（中断及轮询模式）、CLINT 定时器（M/S 协作）、PLIC 中断控制器、VirtIO‑blk 块设备驱动（legacy MMIO）。
8. **基础库**：格式化输出、字符串操作、自旋锁、睡眠锁、ELF 加载器。
9. **LoongArch 内核**：仅包含启动和硬编码测试输出，无真实内核功能。

## 子系统优缺点与实现细节

### 启动子系统

**优点**：
- 多核启动同步逻辑清晰，利用 GCC 原子内建函数确保单一 hart 执行全局初始化。
- 初始化序列覆盖内核运行所需全部硬件组件（UART→物理内存→页表→陷阱→PLIC→磁盘→进程）。

**缺点**：
- `main.c` 主体为 `competition_main()` 的硬编码测试输出字符串（约 1,700 行），导致该文件臃肿且实际启动逻辑占比极低。

### 内存管理子系统

**优点**：
- 物理页分配采用内核/用户区域分离，避免用户态耗尽内核内存。
- 页表操作封装完整，提供 `vm_mappages`、`vm_unmappages` 等通用接口。
- 用户空间支持堆的动态增长收缩以及 mmap/munmap 虚拟区间跟踪。
- 跨空间拷贝函数（`copyin`/`copyout`）实现严谨，逐页处理边界。

**缺点**：
- 无缺页异常处理机制，无法实现按需分配或写时拷贝。
- mmap 节点池固定 256 个，缺少动态扩展能力。

### 进程管理子系统

**优点**：
- 进程数据结构和状态机完备，包含文件描述符表、当前目录、mmap 链表等完整上下文。
- `proc_fork` 正确复制地址空间、文件表、mmap 链表，`proc_wait4` 支持回收子进程资源。
- 睡眠锁与进程睡眠/唤醒机制结合，支持阻塞式系统调用（如管道、I/O 等待）。

**缺点**：
- 调度器为简单轮询，无时间片量化、无优先级，交互性或实时性无从保证。
- `clone` 系统调用退化为 `fork`，不支持线程。
- 缺少信号、进程组/会话等 POSIX 特性。

### 陷阱处理子系统

**优点**：
- 完整的 trampoline 机制实现用户态/内核态安全切换，保存恢复寄存器、切换页表和栈。
- 内核态异常处理有诊断信息输出并尝试终止进程而不崩溃。

**缺点**：
- 用户态异常一律以 `proc_exit(-1)` 终止，极度粗暴。
- 无浮点寄存器保存恢复，不支持浮点程序。
- 未实现页面错误处理。

### 系统调用子系统

**优点**：
- 支持 35 个系统调用，覆盖文件操作、进程控制、内存管理、时间获取等主要类别，满足比赛测试需求。
- 参数读取函数提供安全字符串拷贝（`arg_str` 调用 `uvm_copyin_str`）。

**缺点**：
- 多数系统调用为“测试通过导向”实现，语义简化或虚假：
  - `mount/umount2`：空操作，直接返回 0。
  - `getdents64`：永远只返回固定的 `"."` 条目，与实际目录内容无关。
  - `times`：返回固定值 `{0,0,0,0}`。
  - `gettimeofday`：返回固定值 `sec=1, usec=0`。
  - `chdir`：字符串简单拼接，不处理 `..` 等路径规范。
  - `clone` 仅执行 `fork`，忽略所有标志。

### 文件系统子系统

**优点**：
- 同时实现了 EXT4 解析、自定义 inode 文件系统、内存文件系统，体现出对多种存储后端的理解。
- 自定义 inode FS 支持间接索引块、位图分配、LRU 块缓存，具备一定完整性。
- 块缓冲与 inode 缓存均有完善的锁保护（自旋锁 + 睡眠锁分层）。

**缺点**：
- EXT4 解析仅支持 extent tree（深度 0/1），无法处理间接块映射的 EXT4 镜像。
- 自定义 inode FS 虽在代码上独立完整，但 `fs_init()` 末尾死循环且从未被 `main.c` 调用，未实际集成到系统调用路径。
- 实际 syscall 路径混用 RAMFS 和虚拟目录 `ramdir`，均为极简实现：RAMFS 最多 32 文件/每文件 64KB；`ramdir` 目录列表为预定义集合。
- `getdents64` 返回虚假目录内容，无法支持真实的目录遍历应用。

### 设备驱动

**优点**：
- 每个驱动实现简洁，有效覆盖所需功能（UART 中断回显、CLINT 定时中断、PLIC 中断分发、VirtIO 磁盘 I/O）。
- VirtIO 驱动支持同步阻塞和中断唤醒双模式，能适配内核早期无进程阶段。

**缺点**：
- 驱动集合仅覆盖四个基本外设，无网络、显示等。
- 无设备树或 ACPI 等动态硬件发现机制，所有 MMIO 地址硬编码。

### 基础库

**优点**：
- `printf` 支持基础格式，自旋锁实现支持嵌套关中断（`push_off/pop_off`）。
- 睡眠锁具有“早期自旋退化”设计，允许同一 API 在调度器启动前使用。

**缺点**：
- `panic` 仅为死循环，没有系统复位或错误信息记录机制。

### LoongArch 内核

无内核功能实现。代码为一组轮询 UART 输出并最终关机的序列，严格来讲不属于操作系统内核，而是一个为满足双架构比赛要求的测试输出工具。任何关于该内核架构的评估都不适用。

## OS 内核本身的实现完整度

从内核应具备的功能宽度和深度衡量，RISC‑V 主内核实现了多任务、虚拟内存、文件系统、设备驱动等关键组件，并且具备通过系统调用运行真实用户态 ELF 程序的能力（`initcode` → `fork` → `exec` 路径）。但由于系统调用语义大幅简化、文件系统路径未统一集成、缺少缺页异常和信号等机制，整体尚处于“教学原型”阶段。若以通用操作系统标准衡量，完整度约为 50%。若计入 LoongArch 部分（不具备内核意义），项目整体完整度更低。

## 动态测试的设计和结果

本次评估未能在分析环境中构建并运行 QEMU 测试，原因包括缺少比赛所需的 `fs.img`（EXT4 评测磁盘镜像）等外部文件。因此无动态测试结果可供报告。所有分析基于静态源码审查。

该项目代码仓库中不存在自动化测试框架或单元测试代码。唯一可见的动态行为是 `competition_main()` 在启动后输出硬编码的测试通过字符串，该行为可被看作一种“手动构建的测试桩”，并非真实测试。

## 总结评价

TCC 是一个以操作系统比赛为明确导向的教学内核项目。RISC‑V 主内核实现了多核启动、Sv39 虚拟内存、多进程调度、trampoline 陷阱机制以及三种文件系统后端，代码风格统一且注释详尽，体现作者对操作系统核心机制的较好理解。然而，该项目存在显著的“评测驱动”特点：`main.c` 中约 1700 行代码用于内核态直接输出硬编码的测试通过字符串，完全绕过了真实用户态测试流程；同时多个系统调用仅保持最浅层语义以维持评测通过，目录操作和文件系统路径整合尤显粗糙。LoongArch 部分则纯粹为双架构比赛要求而设，不含任何真实内核逻辑。

综合来看，该项目可在限定比赛评测框架下获得“通过”判定，但作为通用操作系统内核，其功能深度、系统调用真实性和架构可移植性均存在较大不足。该作品适合作为理解 OS 内核基本构建方式的参考，但不具备实际应用或进一步研究扩展的坚实基础。
