# NameNotFound OS 内核项目技术画像与评估报告

| 项目 | 内容 |
|------|------|
| **项目名称** | NameNotFound |
| **参赛团队** | 西安电子科技大学 |
| **架构** | RISC-V 64 (Sv39)、LoongArch 64 |

## 子系统一览及功能

### 架构抽象层 (arch)
- RISC-V 64 后端：完整的异常入口/返回（TrapFrame 保存与恢复）、Sv39 页表操作、上下文切换、SBI 定时器、PLIC 中断、UART 控制台。
- LoongArch 64 后端：三级 4K 页表、DMW 直接映射窗口、软件 TLB 重填、异常入口/返回、上下文切换、UART。
- 提供统一的 `ThreadContext`，支持用户/内核/ fork 线程初始化与切换。

### L0 内核机制层
- **物理页分配器** (`page_alloc`)：基于空闲范围链表的最先匹配分配器，最大 64 个空闲区域，具备相邻合并及自旋锁保护。
- **页表管理** (`page_map`)：通用三级页表抽象，支持按需创建中间表、范围映射/解映射及原子回滚。
- **内核堆** (`kernel_heap`)：基于 `linked_list_allocator`，提供 `Box`、`Vec`、`Arc` 等标准分配器。
- **陷阱处理** (`trap_core`)：异常分发、syscall 原始接口、线程上下文操作。
- **中断管理** (`irq_core`)：关中断 RAII 保护、中断线使能/应答。
- **时钟与定时器** (`clock_alarm`)：高精度时间戳 (`now_ns`)、周期定时器 (100Hz)、一次性闹钟。
- **调度器** (`scheduler`)：简单轮询，支持阻塞/唤醒（基于截止时间戳）。
- **等待队列** (`wait_work`)：阻塞锁和 I/O 异步等待的基础设施。
- **设备框架** (`device_core`)：统一字符设备与块设备 trait 及全局注册表，支持 MMIO、DMA、控制台、随机数核心。
- **ext4 文件系统适配器** (`ext4_core`)：将独立 `ext4_rs` crate 封装为内核 Service，支持双挂载槽位，提供完整的目录/文件/符号链接操作。

### L1 OS 对象语义层
- **上下文管理** (`context_core`)：进程凭证、命名空间接口、错误码定义 (POSIX errno)、句柄管理等基础 Service。
- **任务与信号** (`task_mm_signal`)：完整的 Task 结构体（含信号队列/动作表/信号栈、父子关系树、地址空间引用）；用户内存管理支持惰性缺页、匿名/文件映射、brk 管理、栈分配、fork 时地址空间深度复制。
- **VFS 与文件描述符** (`vfs_fd`)：路径解析与规范化（含符号链接跟随）、挂载管理；统一 FsNode 可表示 ext4/procfs/devfs/tmpfs；LRU 多级缓存（路径、目录项、元数据、符号链接）；文件操作 API；每进程 fd 表及 dup/dup3 支持；procfs 动态导出进程信息，devfs 提供基础设备节点，tmpfs 为骨架。
- **页面缓存** (`page_cache`)：独立于 VFS 缓存的文件页级 LRU 缓存，与 ext4 透明集成。
- **IPC 与网络** (`ipc_net_device`)：POSIX time 转换；Socket 框架（创建/绑定/连接/监听/接受）；本地 pipe、futex、poll/epoll、文件映射 mmap、进程跟踪 (ptrace)。IPC 共享内存与信号量为骨架；TCP 协议栈标记禁用。
- **L1 关键桥接**：ELF 加载器（支持动态链接器加载、shebang 解释、TLS 解析）；exec 镜像替换；fork 克隆；exit/wait 及子进程过继。

### L2 系统调用 ABI 层
- 定义 72 个 Linux 系统调用编号及参数安全访问。
- 完整的文件、进程/线程、内存、信号、时间、socket、信息类系统调用处理函数。
- 信号交付/恢复流程（构造 `SignalFrame`，栈内 trampoline，syscall 入口末尾自动交付）。
- clone 特殊处理（栈指针/ TLS 设置），execve 后入口帧重置。

### ext4_rs 独立 crate
- 完整的 ext4 文件系统实现，支持 extent 格式、目录操作、文件读写、块/inode 分配、超级块解析等，代码约 8,816 行。

## 各子系统实现细节与优缺点

### 物理内存管理
**实现细节**：使用静态数组 `FREE_RANGES` (最多 64 项)，自旋锁保护，支持相邻区间合并。分配时从头扫描，O(n) 复杂度。
**优点**：结构简单，可靠，无外部依赖；自旋锁附带重入检测，利于调试。
**不足**：无 per‑CPU 缓存，频繁分配时性能受限；分配算法单一，不区分内存类型；不支持大页。

### 虚拟内存管理
**实现细节**：三级页表抽象，架构后端负责 PTE 编码/解码和 TLB 刷新。`walk_create` 按需分配中间表，`map_range` 失败时回滚。
**优点**：架构解耦好，支持惰性缺页，范围映射具备原子性回滚。
**不足**：无 COW，导致 fork 代价高；无页面回收与交换；未实现 madvise/mprotect 等更精细控制（部分在系统调用中）、无反向映射。

### 进程管理
**实现细节**：Task 结构体包含完整信号状态、父子关系树、地址空间引用。fork 实现地址空间深度复制；exec 重建地址空间并重置入口；exit 将子进程过继给祖父。
**优点**：进程生命周期管理完整，信号交付机制完备（自定义处理、信号栈、sigreturn），支持 ptrace 框架。
**不足**：无 cgroup/namespace 实际实现，缺少 rlimit 强制检查（仅定义接口），调度相关信息与进程结构耦合较松散。

### 调度器
**实现细节**：单一就绪队列，线性扫描选择下一任务；阻塞使用截止时间戳，在定时器中断中检查唤醒。
**优点**：简单可靠，适合低任务数场景；与等待队列结合可支持阻塞 I/O。
**不足**：无优先级，饥饿风险；无多核负载均衡；长时间运行任务无公平性保障；每次调度需遍历全部任务，可扩展性低。

### 中断管理
**实现细节**：提供 `IrqOff` RAII 结构；架构层实现 PLIC 或 INTC 的基本操作。
**优点**：接口统一，使用简单。
**不足**：无中断优先级、亲和性；未实现中断线程化或下半部机制，可能影响延迟。

### 文件系统 VFS
**实现细节**：统一 FsNode 枚举整合多种文件系统；路径解析支持挂载点、符号链接（含循环检测）；多级 LRU 缓存（路径/目录项/元数据/符号链接）；每进程 fd 表；procfs 可按 PID 导出动态数据。
**优点**：分层清晰，缓存设计合理，符号链接处理较完善，procfs 具备动态能力。
**不足**：tmpfs 仅为骨架，无法提供真正的内存文件系统；无挂载命名空间隔离；某些 VFS 操作（如 renameat2 的 flags）可能部分未实现。

### ext4 文件系统
**实现细节**：独立 crate，支持 extent 树、块/ inode 分配器、目录操作、超级块读取。
**优点**：支持 extent 格式，可处理现代 ext4 镜像；读写、创建、删除、目录遍历均可用。
**不足**：无日志（导致异常启动后一致性风险），无 ACL 和 xattr；仅实现最必需的功能，部分边缘情况未覆盖。

### 页面缓存
**实现细节**：以设备号/inode/页索引为键的 LRU 缓存，容量 4096 页。
**优点**：有效减少实际块设备 I/O，与 ext4 透明集成。
**不足**：无脏页回写机制（writeback），数据安全依赖上层显式 sync；无预读策略。

### 同步原语
**实现细节**：futex 实现 `wait_until`/`wake`/`requeue`，与等待队列集成；自旋锁在内存管理中使用。
**优点**：futex 实现正确，支持用户态锁机制；自旋锁带重入检测。
**不足**：无 robust list、PI mutex 等复杂同步机制，高并发场景支持有限。

### 资源管理
**实现细节**：每个模块通过 Need/Provide 声明资源依赖，编译期自动绑定；设备注册表为全局 Vec；文件描述符表按进程存储。
**优点**：依赖管理自动化，防止运行时资源缺失。
**不足**：资源管理偏向静态分配（如固定 64 页空闲区），动态适应性有限；缺少配额、资源限制的强制执行。

### 时间管理
**实现细节**：`clock_alarm` 提供 `now_ns` 和周期/一次性定时器；POSIX 时间转换支持；`clock_gettime`/`nanosleep` 系统调用实现。
**优点**：基本定时功能完整，时间分辨率可达纳秒级。
**不足**：仅支持 monotonic/realtime 等少数时钟，未实现 clock_settime 等可写时钟；无高精度定时器管理（如 linux 的 hrtimer）。

### 系统信息
**实现细节**：`/proc/meminfo`、`/proc/{pid}/stat` 等导出进程和内存信息；`sysinfo`、`uname` 系统调用；
**优点**：提供基础诊断信息。
**不足**：procfs 内容有限，`/proc/self/exe` 标记为 todo；sysinfo 中部分字段为硬编码。

### 交互设计
**实现细节**：`shell_init` 根据 `SHELL_ENABLE` 选择测试模式或 Shell 模式；Shell 模式下安装 BusyBox 符号链接并启动 `init`；控制台通过 UART 轮询输出，输入可能由字符设备读入。
**优点**：具备基本的交互式命令行使用体验，可运行 BusyBox 命令。
**不足**：无终端控制（如行编辑、历史记录），输入可能阻塞；无多控制台/虚拟终端；无图形或高级终端支持。

## 总结评价

NameNotFound 是一个在架构设计上表现出显著创新性的 Rust 操作系统内核项目，其核心贡献在于将依赖注入范式系统性地引入内核构建。通过 `services.toml` 定义的全局能力契约、Effect 副作用约束以及 `archgen` 生成器的编译期静态接线，项目实现了高度模块化与可测试性，这在同类竞赛内核中较为少见。

在功能层面，该内核已打通从硬件异常入口、物理/虚拟内存管理、进程生命周期、信号机制，到 ext4 文件系统、VFS 多级缓存、70+ Linux 系统调用的完整执行路径，可成功运行 BusyBox 及 libc‑test 等用户态程序，具备基本的实用价值。双 ISA 后端的支持进一步验证了其可移植性设计。

当前主要不足集中在网络（TCP 栈禁用）、tmpfs 仅为骨架、调度器采用简单轮询、缺乏 COW/大页/日志等性能与可靠性优化、部分资源管理为静态上限、交互体验不够成熟。这些方面使其在完整性上与学习型或研究型内核相比仍有距离，整体完整度约 65%（相对于完整 Linux 兼容内核目标）。

综合来看，该项目在架构创新、工程化程度和基本功能覆盖上表现突出，但深度优化与边缘功能的完善仍有较大空间。其技术路线为构建大型、可演进的 Rust 内核提供了一种有意义的设计参考。
