# noob is not RISV-V OS 内核项目技术评估报告

## 一、 项目基本信息

- **项目名称**：noob is not RISV-V
- **目标架构**：RISC-V 64 位（rv64imac）
- **实现语言**：C 语言为主，RISC-V 汇编辅助
- **生态归属**：OS 竞赛内核赛道项目（吉利学院）
- **运行平台**：QEMU virt 虚拟机（配置为 2 CPU，128MB 内存）
- **代码规模**：内核包含 25 个 C 源文件、3 个汇编文件、28 个头文件；SBI 固件包含 3 个 C 文件、2 个汇编文件
- **构建系统**：GNU Make 结合 RISC-V 交叉编译工具链（riscv64-unknown-elf-gcc）
- **项目特点**：采用进程与线程分离的数据结构设计；使用协程（Coroutine）机制实现内核态的上下文切换与调度；自行实现了极简的 M-mode SBI 固件。

## 二、 实现的子系统与功能

该项目实现了一个包含 M-mode 固件、S-mode 内核与 U-mode 用户态的三层操作系统架构，具体包含以下子系统与功能：

1. **SBI 固件（M-mode）**：实现基础的硬件初始化、PMP 配置、异常委托以及基于 CLINT 的定时器中断转发。
2. **引导与陷阱处理**：实现内核态与用户态的陷阱向量（trap vector），支持系统调用（ecall）、缺页异常与外部设备中断的分发处理。
3. **物理内存管理（PMM）**：实现基于二叉树的 Buddy 分配器与基于链表的 Slab 分配器。
4. **虚拟内存管理（VMM）**：实现 RISC-V Sv39 三级页表管理、内核与用户地址空间映射、VM 区域（VMA）链表管理以及基础的缺页异常处理。
5. **进程与线程管理**：实现进程与线程分离的控制块设计、基于静态池的分配机制、`fork`、`exec`、`exit`、`wait` 等生命周期管理。
6. **协程调度器**：基于 `setjmp`/`longjmp` 思想实现的协程上下文切换，提供基础的轮询调度循环。
7. **文件系统**：实现 VFS 抽象层与 FAT32 文件系统，支持 BPB 解析、FAT 链遍历、长短文件名解析、文件与目录的创建及读写。
8. **块 I/O 与设备驱动**：实现基于 LRU 策略的块缓存、VirtIO MMIO 块设备驱动、UART 串口驱动与 PLIC 中断控制器驱动。
9. **系统调用接口**：实现约 20 个系统调用，涵盖进程控制、文件操作与内存管理。

## 三、 各个子系统的实现完整程度

- **SBI 固件**：部分实现。仅完成引导与定时器转发，未实现 SBI 规范中的控制台服务与多核唤醒（hart 1 直接进入死循环）。
- **引导与陷阱处理**：基本完整。支持主要的用户态陷阱，但内核态陷阱处理缺乏容错机制（未知陷阱进入死循环）。
- **物理内存管理（PMM）**：较完整。实现了 Buddy 与 Slab 两级分配，但缺乏并发锁保护。
- **虚拟内存管理（VMM）**：基本完整。实现了页表与 VM 区域管理，缺页异常仅支持用户栈的延迟分配，不支持写时复制（COW）。
- **进程与线程管理**：基本完整。实现了核心的生命周期系统调用，但 `fork` 未实现 COW，父子进程直接共享物理页。
- **协程调度器**：部分实现。实现了协程切换机制，但缺乏时间片轮转与抢占式调度能力，且硬编码为单核运行。
- **文件系统（FAT32）**：较完整。支持文件的创建、读写与目录遍历，但未实现文件删除（unlink）与 FAT 链释放。
- **块 I/O 缓存**：基本完整。实现了 LRU 替换策略，但缓存容量固定且较小（30 个块）。
- **VirtIO 块设备驱动**：基本完整。实现了描述符链与 I/O 提交，但采用忙等待（Busy-waiting）同步阻塞机制，未利用中断。

## 四、 各个子系统的优缺点及实现细节

### 4.1 SBI 固件
- **实现细节**：链接于 `0x8000000`，配置 PMP 允许 S-mode 访问全部物理内存，将异常与中断委托至 S-mode。定时器中断间隔设定为 1,000,000 个时钟周期，通过触发 S-mode 软件中断（SSIP）进行转发。
- **优点**：代码精简，直接切入内核引导，无冗余依赖。
- **缺点**：不符合标准 SBI 规范，缺乏多核支持（非 0 号 hart 直接死循环），缺乏控制台打印等基础调试服务。

### 4.2 物理与虚拟内存管理
- **实现细节**：PMM 使用静态数组（128KB）维护 Buddy 树，Slab 支持 8B 至 2KB 的 8 种对象大小。VMM 采用 Sv39 页表，每个进程维护 `vm_t` 链表记录 6 种类型的虚拟内存区域。每个线程拥有独立的 `stack_pagetable`。
- **优点**：PMM 采用成熟的 Buddy+Slab 组合；VMM 的 VM 区域分类设计（如 `VM_LAZY_ALLOC`, `VM_NO_FORK`）为内存管理提供了细粒度控制。
- **缺点**：PMM 的 `palloc`/`pfree` 未使用自旋锁保护，存在并发安全隐患；`kfree` 依赖地址是否页对齐来区分分配器，逻辑脆弱。VMM 中大量 `sfence.vma` 被注释，可能引发 TLB 一致性问题；堆管理器无锁保护。

### 4.3 进程与线程管理
- **实现细节**：进程与线程控制块分离，各自维护大小为 256 的静态池。`fork` 复制 VM 链表并标记 `VM_PA_SHARED`；`exec` 清除 VM 列表并重新加载 ELF。
- **优点**：进程与线程分离的设计在 OS 竞赛项目中较为少见，结构清晰，有利于后续扩展多线程支持。
- **缺点**：`fork` 未实现写时复制（COW），父子进程直接写入同一物理页会导致数据损坏；`sys_exec` 中对 `epc -= 4` 的处理逻辑缺乏明确依据；进程退出时的资源释放深度固定为 1，可能遗漏嵌套资源。

### 4.4 协程调度器
- **实现细节**：为每个线程分配一个管理协程，通过汇编实现的 `coro_setjmp`/`coro_longjmp` 保存和恢复 `ra`、`sp` 及 `s0-s11` 寄存器。调度器在主循环中遍历线程池，切换至 READY 状态的线程协程。
- **优点**：使用协程进行上下文切换是一种创新尝试，减少了传统汇编级上下文切换的复杂度。
- **缺点**：调度策略退化为简单的轮询，无优先级与时间片机制；定时器中断处理函数为空，无法实现抢占式调度；硬编码单核运行，无法利用 QEMU 配置的双核环境。

### 4.5 文件系统与块 I/O
- **实现细节**：VFS 层定义 superblock、inode、file 抽象。FAT32 驱动解析 BPB，将 FAT 表整体读入内存，支持 SFN 与 LFN。块缓存维护 30 个块的 LRU 链表。
- **优点**：FAT32 实现度较高，支持长文件名解析与跨簇读写；VFS 层支持文件系统挂载机制（如 initrd 挂载）。
- **缺点**：FAT 表全量读入内存，对大容量磁盘扩展性差；未实现 `unlink` 系统调用与 FAT 链回收，导致文件系统只能增不能删；块缓存容量仅 15KB，对频繁 I/O 场景性能受限。

### 4.6 VirtIO 块设备驱动
- **实现细节**：使用 VirtIO MMIO legacy v1 接口，配置 8 个描述符的队列。I/O 操作构建包含请求头、数据、状态的三描述符链。
- **优点**：正确实现了 VirtIO 队列的初始化与描述符链的构建。
- **缺点**：I/O 提交后采用 `while(b->disk == 1)` 忙等待，导致磁盘操作完全同步阻塞，浪费了 CPU 资源，且未实际利用已实现的 VirtIO 中断处理函数。

## 五、 OS 内核本身的实现完整度

该 OS 内核具备了现代操作系统的基本骨架，涵盖了内存管理、进程调度、文件系统与设备驱动四大核心模块。从代码结构来看，内核包含了 25 个 C 源文件，模块划分较为清晰。然而，在核心机制的深度与健壮性上存在明显缺失：缺乏抢占式调度、缺乏写时复制（COW）、缺乏文件删除机制、缺乏多核支持，且多处关键路径（如 PMM、堆管理）缺乏并发锁保护。整体而言，该项目实现了一个能在单核环境下运行基础用户态程序的原型系统，但在应对并发、资源回收与复杂 I/O 场景时完整度不足。

## 六、 动态测试的设计和结果

**测试设计**：
原计划通过构建外部测试套件 `riscv-syscalls-testing`，制作包含用户态程序的 FAT 镜像（`sdcard.img`）与初始内存盘（`initrd.img`），在 QEMU virt 机器上启动内核，验证系统调用、进程创建、文件读写及调度器的实际运行表现。

**测试结果**：
- **静态构建**：内核（25 个 C 文件 + 3 个汇编文件）与 SBI 固件均成功编译并链接，生成 319KB 的 `kernel-qemu` ELF 文件与 200KB 的 `sbi-qemu` 裸二进制镜像。
- **动态运行**：由于镜像制作流程强依赖 `sudo mount` 挂载 loop 设备，在当前沙箱环境中受限于权限无法执行。尝试使用 `mcopy` 替代方案制作 FAT 镜像，但因完整的 QEMU 启动流程需要所有特定格式的镜像就绪，最终未能完成端到端的动态运行测试。因此，缺乏运行时状态下的功能验证与性能数据。

## 七、 细则评价表格

| 评价条目 | 是否实现及完整度 | 关键发现 | 评价 |
| :--- | :--- | :--- | :--- |
| **内存管理** | 较完整。实现了 Buddy+Slab 物理分配与 Sv39 虚拟内存管理。 | PMM 缺乏锁保护；`kfree` 依赖页对齐区分分配器；缺页异常仅支持用户栈延迟分配；无 COW 机制。 | 基础框架完备，VM 区域分类设计合理，但并发安全性差，高级内存特性（如 COW）缺失。 |
| **进程管理** | 基本完整。实现了进程/线程分离、fork、exec、exit、wait。 | 进程/线程池静态分配（各 256 个）；fork 直接共享物理页；exec 中 `epc -= 4` 逻辑不明。 | 数据结构设计有亮点，但核心系统调用（如 fork）的实现存在致命的数据一致性缺陷。 |
| **文件系统** | 较完整。实现了 VFS 抽象与 FAT32 驱动，支持挂载。 | FAT 表全量加载至内存；支持长短文件名；未实现文件删除（unlink）与簇回收。 | 读写与创建功能可用，但缺乏资源回收机制，长期使用会导致磁盘空间耗尽。 |
| **交互设计** | 部分实现。实现了 UART 输出与控制台初始化。 | 仅支持串口字符输出，无输入处理；SBI 层未实现标准控制台服务。 | 仅满足基础的内核日志打印需求，缺乏完整的用户态终端交互能力。 |
| **同步原语** | 基本完整。实现了自旋锁（spinlock）与等待队列（waitqueue）。 | 自旋锁已定义并在部分模块使用，但 PMM、VMM 堆管理等核心模块未使用锁保护。 | 提供了基础的同步工具，但在内核全局范围内的应用不一致，存在竞态条件风险。 |
| **资源管理** | 部分实现。实现了文件描述符表（256 个）与进程/线程静态池。 | 进程退出时 VM 列表释放深度固定为 1；FAT32 无簇回收；块缓存固定为 30 个块。 | 资源分配机制存在，但回收机制严重不完善，易导致内存与磁盘资源泄漏。 |
| **时间管理** | 不完整。实现了 M-mode 定时器转发，但 S-mode 处理为空。 | 定时器中断间隔 1,000,000 周期；S-mode 的 `clock_intr()` 为空（TODO）；无系统时间获取。 | 仅实现了硬件层面的中断触发，内核层未消费时间事件，导致无法进行时间片调度与时间统计。 |
| **系统信息** | 未实现。 | 未发现 `uname`、`sysinfo` 等获取内核版本、硬件信息的系统调用或接口。 | 缺乏标准的系统信息暴露机制。 |
| **设备驱动** | 基本完整。实现了 VirtIO 块设备、UART、PLIC。 | VirtIO 采用忙等待同步阻塞，未利用中断；PLIC 初始化完整但仅服务于单核。 | 驱动能完成基本 I/O 任务，但 VirtIO 的忙等待设计严重降低了 I/O 并发效率。 |
| **异常与中断** | 基本完整。实现了内核/用户态陷阱向量与分发。 | 支持 ecall、缺页、外部中断；内核态未知陷阱进入死循环；`sfence.vma` 被注释。 | 异常分发逻辑清晰，但缺乏内核态容错处理，且 TLB 刷新机制被禁用，存在隐患。 |

## 八、 总结评价

“noob is not RISV-V” 是一个结构清晰、具备一定设计创新的 RISC-V 操作系统原型。项目在架构上做出了有价值的尝试，例如将进程与线程控制块分离、引入协程机制进行上下文切换，以及实现较为完整的 FAT32 文件系统与 VFS 挂载机制。这些设计表明开发者对操作系统核心概念有较深的理解。

然而，该项目在工程实现的严谨性与核心机制的深度上存在明显短板。首先，关键路径缺乏并发保护（如 PMM 和堆管理无锁），且 `fork` 系统调用未实现写时复制，这在多任务环境下会导致严重的数据损坏。其次，调度器缺乏时间片与抢占机制，VirtIO 驱动采用忙等待，使得系统无法有效处理并发 I/O 与计算任务。最后，文件系统缺乏删除与空间回收机制，SBI 固件与调度器硬编码为单核运行，未能利用现代多核硬件环境。

总体而言，该项目成功构建了一个能够引导并运行基础用户态程序的单核操作系统骨架，在文件系统与内存区域管理上表现出较好的完成度，但在并发安全、资源回收、抢占式调度等高级操作系统特性上仍需大幅完善。
