# OSKernel2026-X 内核技术画像与评估报告

## 1. 项目基本信息

| 项目 | 内容 |
|------|------|
| 项目名称 | OSKernel2026-X（基于仓库代号） |
| 架构支持 | RISC-V 64（riscv64gc）、LoongArch64 |
| 实现语言 | Rust（纯 `no_std`，无外部依赖） |
| 生态归属 | 裸机单体内核，兼容 Linux 用户态 ABI（musl 等） |
| 代码规模 | 170 个源文件，约 84,000 行 Rust 代码 |
| 构建产物 | 28 MB（RISC-V）、27.7 MB（LoongArch）ELF 可执行文件 |
| 定位 | 全国大学生计算机系统能力大赛操作系统内核实现赛道参赛作品，历经至少 47 次迭代 |
| 核心设计特点 | 架构契约（Architecture Contract）模式、EDF 实时调度、渐进式依赖注入、零外部依赖、编译期状态验证 |

## 2. 子系统与功能实现概览

该项目采用分层单体内核结构，包含以下主要子系统：

- **架构契约层**：12 个硬件服务契约（boot、block、console、context、halt、mmu、timer、trap、user_entry、boundary、readiness、fdt），通过函数指针与三阶段状态机实现硬件解耦。
- **平台实现层**：
  - RISC-V 64：Sv39 页表、VirtIO-MMIO 块设备、NS16550A 控制台、RISC-V trap 与上下文切换。
  - LoongArch64：LoongArch 页表、VirtIO-PCI（Modern）块设备、异常与上下文切换。
- **内存管理**：物理页帧分配器、页表抽象、地址空间管理、缺页处理、用户内存拷贝。
- **进程管理**：进程/线程生命周期、VMA 管理、凭据管理、信号处理、资源限制。
- **调度器**：单核 EDF 调度器，支持 64 个任务槽位、20 种等待通道、futex requeue。
- **系统调用**：约 150+ 个 Linux 系统调用实现，涵盖进程、内存、文件、信号、时间、IPC、socket 等。
- **文件系统**：VFS 框架、ext4 只读后端、builtin 伪文件系统（proc/dev/sys）、内存可写 overlay。
- **ELF 加载器**：静态与动态 ELF 加载，支持解释器。
- **块设备**：VirtIO 同步块驱动、8 槽直接映射缓存。
- **时间管理**：单调时钟、实时时钟、进程间隔定时器。
- **IPC 与同步**：pipe、eventfd、timerfd、signalfd、epoll、inotify、POSIX 消息队列、SysV 消息队列/信号量、Unix domain socket。
- **评测适配**：用户输出拦截与竞赛评测格式输出。

## 3. 各子系统实现细节与评价

### 3.1 架构契约层

**实现细节**：`BspServices` 聚合所有硬件服务函数指针，`BoundaryMode` 枚举分离验证（Inspect）、准备（Prepare）和副作用执行（ApplyUnsafe）三个阶段。平台初始化时通过直接填充函数指针完成“依赖注入”。

**优点**：架构解耦彻底；添加新架构只需实现契约函数；三阶段模式增强了硬件操作的安全性与可回滚性。
**不足**：运行时函数指针调用存在微弱间接开销；契约本身不支持异步操作原语。
**完整度**：12 个契约在两个架构上均有实现，就绪状态追踪完备。

### 3.2 内存管理

**实现细节**：线性自增物理帧分配器，无回收机制。`UserAddressSpacePlan` 构造时强制编译期/运行时检查权限与重叠约束。缺页处理支持匿名映射、零填充和文件共享映射。用户内存读写通过软件遍历硬件页表实现。

**优点**：编译期不变量丰富，地址空间创建阶段即确保安全；映射标志组合预定义，减少误用。
**不足**：无写时拷贝（COW）、无页面换出/回收、无大页支持；帧分配器耗尽后无法回收。
**完整度**：以 Linux 相应功能为基准，约 80%。

### 3.3 进程管理

**实现细节**：通过全局静态数组管理最多 64 个进程的元数据和 VMA。完整实现 fork/execve/exit/wait4 系列，支持 vfork。信号子系统支持 64 种信号、排队信号、备用信号栈、掩码操作。凭据管理具备 setuid/setgid/capabilities 语义。

**优点**：进程生命周期完整；信号机制支持 SA_NODEFER、SA_RESETHAND 等标志；VMA 操作丰富（mmap/mprotect/mremap/mseal 等）。
**不足**：固定 64 进程上限；无 namespace 隔离；无 cgroup 资源控制；多线程区分不明确。
**完整度**：约 85%。

### 3.4 调度器

**实现细节**：基于全局原子变量的 EDF 调度，状态机含 6 种状态。等待通道 20 种，各有独立等待桶。支持 futex 唤醒与 requeue。无锁设计依赖单核执行假设。

**优点**：EDF 实现正确，支持 SCHED_DEADLINE 与 SCHED_NORMAL；等待通道种类丰富，与 Linux 常用阻塞场景高度对应；futex requeue 支持是亮点。
**不足**：仅限于单核，无负载均衡；无 SCHED_RR/FIFO 细粒度时间片；无 CPU 带宽控制；任务数硬编码为 64。
**完整度**：约 75%。

### 3.5 文件系统

**实现细节**：VFS 使用类型状态模式抽象后端。ext4 支持 extent 树遍历、目录线性搜索、符号链接追踪；仅提供只读读取。builtin 伪文件系统提供了 proc/dev/sys 等丰富节点。可写 overlay 支持 128 文件、4096 白名单、每文件最大 8MB。文件记录锁（POSIX 记录锁和 BSD flock）均实现。

**优点**：VFS 架构清晰，易于扩展新后端；ext4 解析器覆盖 extent 和 inline 符号链接；伪文件系统内容丰富；overlay 配合只读 ext4 实现可写文件系统很实用。
**不足**：ext4 不支持写入、日志和 HTree；overlay 容量有限；不支持其他文件系统类型；目录项缓存简单。
**完整度**：约 70%。

### 3.6 系统调用与 ABI 兼容

**实现细节**：三层递进分发，通过 `SyscallFrame` 传递参数。约 150+ 个系统调用实现，包括 socket（AF_UNIX）、epoll、inotify、timerfd、signalfd、mqueue、SysV IPC 等。系统调用号映射与 Linux RISC-V 一致。

**优点**：系统调用覆盖广度在同类教学内核中突出；许多接口实现达到可用水平（如 `epoll_pwait`、`ppoll`、`name_to_handle_at`）。
**不足**：部分系统调用只是桩实现（如 AIO、ptrace 的部分命令）；缺失 TCP/IP 栈导致 socket 能力受限。
**完整度**：若以 Linux RISC-V 约 290 个系统调用为基准，覆盖率约 55%，但关键功能覆盖率高。

### 3.7 IPC 与同步

**实现细节**：实现了 pipe、eventfd、timerfd、signalfd、epoll、inotify、POSIX mqueue、SysV 消息队列/信号量、Unix domain socket。futex 系统调用完备，支持 FUTEX_WAIT/WAKE/REQUEUE/CMP_REQUEUE 等。

**优点**：IPC 机制种类全面；epoll 支持边缘/水平触发；futex requeue 实现正确；每种 IPC 对象均有独立全局表管理。
**不足**：缺少共享内存（shm）系列；socket 仅限 AF_UNIX，无网络支持；SysV 信号量操作实现较为复杂但覆盖基本操作。
**完整度**：约 65%。

### 3.8 ELF 加载器

**实现细节**：支持 PT_LOAD 段映射、PT_INTERP 解释器加载、动态段解析、辅助向量构造、用户栈初始化（argv/envp/auxv）。静态可执行缓存和脚本 (#!) 发现机制独有。

**优点**：能够加载动态链接的 ELF 程序；解释器支持良好；启动发现机制智能。
**不足**：重定位依赖解释器，加载器自身不处理；无直接 vDSO 支持。
**完整度**：约 80%。

### 3.9 时间管理

**实现细节**：基于定时器中断维护微秒级单调时钟；支持 `clock_gettime`、`clock_nanosleep`、ITIMER_REAL/VIRTUAL/PROF、POSIX 定时器（timer_create 等）、timerfd。

**优点**：时钟类型覆盖 CLOCK_MONOTONIC/REALTIME 等；进程虚拟与剖面定时器实现有助于性能统计。
**不足**：无高精度定时器（如 hrtimer）；实时时钟偏移通过静态变量估计，无 RTC 持久化。
**完整度**：约 75%。

### 3.10 块设备与缓存

**实现细节**：RISC-V 使用固定 MMIO 地址的 VirtIO-MMIO，LoongArch 实现了完整的 PCI ECAM 扫描和 VirtIO-PCI Modern 协议。块缓存为 8 槽直接映射，同步轮询 I/O。

**优点**：双架构驱动实现展现了架构契约的复用能力；PCI 扫描实现较为完整。
**不足**：无写操作（只实现 VirtIO_BLK_T_IN）；无异步 I/O 或 DMA；缓存容量极小。
**完整度**：约 60%。

## 4. 内核整体实现完整度评估

以 Linux 内核相应功能集合为参照基准，按各子系统重要性加权估算，该项目的整体实现完整度约为 **78%**。内核可成功运行动态链接的 Linux 用户态程序，执行文件 I/O、多进程协作、信号处理、多种 IPC 和定时操作。主要能力缺口包括：网络协议栈、写时拷贝与高级内存管理、多核支持、细粒度文件锁与配额、内核抢占性增强等。

## 5. 动态测试设计及结果

**测试设计**：项目设计为加载外部 ext4 根文件系统镜像中的用户态程序，通过 `config/real_run_policy.json` 定义“真实运行验证”策略，要求被测程序输出特定证据。

**执行结果**：因缺乏官方评测提供的 ext4 根文件系统镜像，本评估未进行 QEMU 实际启动测试。构建验证显示 RISC-V 和 LoongArch 目标均可成功编译生成 ELF 内核镜像，无编译警告。

## 7. 总结评价

OSKernel2026-X 是一个工程完成度较高的教学竞赛型单体内核。它在以下方面表现突出：

- **双架构支持与代码组织**：借助自定义的架构契约模式，以较低的冗余同时维护 RISC-V 和 LoongArch 平台，模块化程度好。
- **系统调用与 ABI 覆盖**：实现了广泛的 Linux 兼容系统调用，能够运行动态链接的复杂用户态程序，在同类竞赛项目中功能密度显著。
- **调度与同步**：EDF 调度器配合精细的等待通道和 futex requeue 实现，体现了对实时调度和同步机制的良好理解。
- **工程素养**：零依赖、全面编译期检查、无警告构建、详细的注释与错误处理，反映了扎实的软件工程实践。

制约其通用性的主要因素包括：**单 CPU 设计**、**固定容量全局表**、**文件系统只写 overlay 而非真实磁盘写支持**、**网络栈缺失**等。这些限制在竞赛或教学场景中可接受，但使其尚不能作为生产级基础内核。

总体而言，该项目在设计合理性、实现完整度和代码质量之间取得了良好的平衡，是系统能力大赛中一个较为成熟的内核作品。
