# OSakura 内核项目技术画像与评估报告

## 一、 项目基本信息

- **项目名称**：OSakura
- **目标架构**：RISC-V 64位 (RV64)
- **实现语言**：C语言（辅以RISC-V汇编）
- **生态归属**：教学/竞赛级操作系统内核（面向QEMU virt平台，采用OpenSBI引导）
- **代码规模**：约 9633 行（包含 `.c` 和 `.S` 源文件）
- **许可证**：GPLv3
- **核心特点**：
  - 支持 ext4 与 FAT32 双文件系统，并通过函数指针表实现抽象层。
  - 支持 ELF 动态链接程序的加载与执行。
  - 实现轻量级 procfs 虚拟文件系统。
  - 提供约 60 个 Linux 兼容系统调用。

## 二、 实现的子系统与功能

1. **启动引导**：OpenSBI M-mode 到 S-mode 的跳转，单核初始化，BSS段清零，内核栈设置。
2. **进程管理**：进程控制块(PCB)管理，fork/exec/exit/wait 生命周期管理，基于时间片的轮转调度，上下文切换。
3. **内存管理**：物理内存空闲链表分配（分内核池与用户池），SV39 三级页表管理，内核/用户虚拟地址空间映射，mmap/brk 动态内存分配。
4. **文件系统**：ext4 文件系统（支持 extent 树），FAT32 文件系统，多文件系统抽象层，轻量级 procfs，块设备缓冲层(LRU)，管道(pipe)。
5. **设备驱动**：16550A UART 驱动，VirtIO 块设备驱动(MMIO)，PLIC 中断控制器驱动，Goldfish RTC 驱动，基于 SBI 的定时器驱动。
6. **陷阱与异常处理**：Trampoline 机制，用户态/内核态陷阱分发，系统调用陷入，时钟与外部中断处理。
7. **系统调用**：约 60 个系统调用，涵盖文件 I/O、进程控制、内存管理、信号处理、时间获取等。
8. **信号机制**：31 种标准信号支持，信号掩码与处理程序注册。
9. **同步原语**：自旋锁(spinlock)，睡眠锁(sleeplock)。

## 三、 各个子系统的实现完整程度

- **启动引导**：实现了单核完整启动流程。多核启动代码框架存在，但相关逻辑被注释禁用。
- **进程管理**：实现了完整的进程生命周期和基础轮转调度。未实现多级反馈队列等高级调度算法，未实现内核线程。
- **内存管理**：实现了基础的物理页分配和 SV39 分页。物理页分配器仅支持单页分配，未实现 Copy-on-Write (COW) 和多页连续分配。
- **文件系统**：ext4 实现了超级块、inode、extent 树和目录操作，未实现日志(journal)机制；FAT32 实现了基础簇链和目录操作；procfs 仅实现了少数硬编码路径。
- **设备驱动**：实现了控制台和块设备核心驱动。未实现 VirtIO 网络驱动（尽管 QEMU 启动参数中配置了网络设备）。
- **陷阱与异常处理**：实现了完整的用户态/内核态陷入与返回机制。内核态异常直接触发 panic，未实现内核态异常恢复。
- **系统调用**：实现了 60 个系统调用接口。部分系统调用（如 `sys_times`）返回硬编码固定值，文件权限检查函数直接返回 true。

## 四、 各个子系统的优缺点及实现细节

### 4.1 启动引导
- **实现细节**：通过 `Entry.S` 根据 `hartid` 计算内核栈地址，`start.c` 关闭分页并设置临时 trap 处理，`main.c` 按序初始化各模块。
- **优点**：初始化顺序清晰，模块依赖关系明确。
- **缺点**：仅支持单核运行，多核扩展性受限。

### 4.2 进程管理
- **实现细节**：进程结构体同时维护 FAT32 和 ext4 两套文件指针；调度器遍历进程表寻找 RUNNABLE 状态进程；上下文切换保存 callee-saved 寄存器。
- **优点**：进程状态机完整，支持信号和 mmap 区域链表管理。
- **缺点**：调度算法为简单的遍历轮转，效率较低；抢占式调度代码被注释，当前依赖协作式或时钟中断唤醒；fork 时全量复制物理页，内存开销大。

### 4.3 内存管理
- **实现细节**：物理内存分为内核池(4MB)和用户池(至128MB)；用户空间使用双向链表管理 mmap 区域（静态数组限制 128 个区域）；采用 SV39 页表。
- **优点**：内核与用户内存池隔离，mmap 支持匿名映射和文件映射。
- **缺点**：物理页分配器通过断言限制仅能分配单页；缺少 COW 机制。

### 4.4 文件系统
- **实现细节**：通过 `FS_OP_t` 函数指针表实现多文件系统切换；ext4 使用 extent 树管理数据块；缓冲层使用 LRU 算法和睡眠锁；procfs 通过拦截特定路径分配虚拟文件描述符（起始 FD=1000）。
- **优点**：文件系统抽象层设计实用，ext4 extent 支持大文件和稀疏文件。
- **缺点**：ext4 目录项限制在单个 4KB 块中，限制了单目录文件数量；缺少日志机制，掉电易导致文件系统损坏；procfs 内容硬编码。

### 4.5 设备驱动
- **实现细节**：UART 使用中断驱动和 32 字节发送缓冲区；VirtIO 块设备使用 MMIO 和虚拟队列；PLIC 配置中断优先级。
- **优点**：核心外设驱动稳定，支持中断驱动的 I/O。
- **缺点**：缺少网络驱动实现，无法利用 QEMU 配置的网络设备。

### 4.6 陷阱与异常处理
- **实现细节**：Trampoline 映射在虚拟地址空间顶部，确保页表切换时指令执行连续；用户态异常导致进程被杀死。
- **优点**：Trampoline 机制标准且高效。
- **缺点**：内核态异常无恢复机制，直接 panic。

### 4.7 系统调用与信号
- **实现细节**：参数通过 trapframe 寄存器传递，提供辅助函数从用户空间安全读取数据；支持 31 种标准信号。
- **优点**：系统调用覆盖面广，支持动态链接程序加载（解析 `PT_INTERP` 段）和输出重定向。
- **缺点**：部分系统调用为桩实现或返回硬编码值；权限检查缺失。

## 五、 OS内核本身的实现完整度

以标准教学/竞赛操作系统（如 xv6 或 Linux 基础子集）为基准，OSakura 实现了操作系统的核心骨架。具备完整的进程调度、虚拟内存、文件系统和设备驱动。但在高级特性（如多核支持、COW、高级调度、网络协议栈、日志文件系统）方面存在缺失。整体功能覆盖度满足基础用户态程序（如 shell、基础命令行工具）的运行需求。

## 六、 动态测试的设计和结果

- **测试设计**：项目包含 `kernel/test/test_execve.c` 用于验证 execve 系统调用和程序加载。用户态包含 `initcode` 作为首个用户进程，负责执行 `/init`。
- **测试结果**：由于构建和运行依赖包含用户程序和 ext4 文件系统的 `sdcard.img` 镜像，且镜像制作脚本需要宿主机的 `sudo` 权限进行 mount 操作，在当前沙箱环境中无法完成动态运行测试。基于静态代码分析，系统调用分发逻辑和 ELF 加载逻辑代码路径完整，理论上可支持动态链接程序的执行。

## 七、 细则评价表格

| 条目 | 是否实现及完整度 | 关键发现 | 评价 |
| :--- | :--- | :--- | :--- |
| **内存管理** | 已实现基础功能。支持 SV39 分页、mmap、brk。 | 物理内存分内核/用户池；物理页分配器断言限制仅分配单页；未实现 COW。 | 满足基础内存分配需求，但单页分配限制降低了大块内存分配效率，无 COW 导致 fork 开销大。 |
| **进程管理** | 已实现核心生命周期。支持 fork/exec/exit/wait。 | 调度器为纯轮转遍历；抢占式调度代码被注释；进程结构体冗余维护双文件系统指针。 | 进程状态机完整，但调度算法过于简单，缺乏优先级和多级反馈机制。 |
| **文件系统** | 已实现 ext4/FAT32 及 procfs。 | ext4 支持 extent 树；目录项限制在单 4KB 块；无 journal 机制；procfs 为硬编码拦截。 | 文件系统抽象层设计良好，ext4 extent 支持是亮点，但单块目录限制和无日志机制是明显短板。 |
| **交互设计** | 已实现基础控制台 I/O。 | UART 驱动支持中断和流控；控制台支持行编辑和回显；支持 shell 输出重定向（硬编码实现）。 | 满足了基本的终端交互需求，但输出重定向硬编码在 `proc_exec` 中，设计不够优雅。 |
| **同步原语** | 已实现自旋锁和睡眠锁。 | 自旋锁支持中断嵌套计数；睡眠锁基于自旋锁和进程休眠实现。 | 实现标准且完整，能够有效保护临界区并避免长时间持有自旋锁导致的 CPU 浪费。 |
| **资源管理** | 已实现基础文件描述符和内存区域管理。 | 进程打开文件表限制为 `NOFILE`；mmap 区域限制为 128 个；虚拟文件描述符起始于 1000。 | 资源上限采用静态数组硬编码，缺乏动态扩展能力，但在竞赛/教学场景下足够使用。 |
| **时间管理** | 已实现定时器与 RTC 驱动。 | 基于 SBI 设置下一次时钟中断；Goldfish RTC 提供实时时钟；`sys_times` 返回固定值 100。 | 硬件时间获取完整，但部分时间相关系统调用返回硬编码值，未实现精确的进程时间统计。 |
| **系统信息** | 已实现基础信息查询。 | 提供 `uname`、`sysinfo` 系统调用；procfs 提供 `/proc/meminfo`（硬编码值）。 | 能够响应基础的系统信息查询请求，但信息内容为硬编码，无法反映系统实时真实状态。 |
| **安全与权限** | 未实现有效权限控制。 | `check_flags` 函数直接返回 true；未实现用户/组权限校验；内核态异常直接 panic。 | 缺乏基本的文件访问权限控制，系统安全性较弱，仅适用于单用户教学/竞赛环境。 |

## 八、 总结评价

OSakura 是一个结构清晰、功能相对完整的 RISC-V 教学/竞赛级操作系统内核。项目成功构建了操作系统的核心骨架，其代码规模适中，模块边界清晰，注释充分。

**主要优势**：
1. 实现了完整的 ext4 文件系统，并支持 extent 树，这在同级别竞赛内核中具有较高的技术含量。
2. 提供了约 60 个系统调用，并支持 ELF 动态链接程序的加载，用户态程序兼容性较好。
3. 采用函数指针表实现的多文件系统抽象层设计实用，展现了良好的软件工程思维。

**主要不足**：
1. 调度算法仅为简单的遍历轮转，且禁用了抢占式调度，多任务处理效率受限。
2. 内存管理缺乏 Copy-on-Write 优化，且物理页分配器不支持多页连续分配。
3. 部分系统调用和虚拟文件系统内容为硬编码或桩实现，缺乏真实的系统状态反馈。
4. 仅支持单核运行，且缺失网络驱动实现，限制了其在更复杂场景下的应用。

总体而言，OSakura 达到了操作系统竞赛和高级教学实验的预期目标，具备较好的代码可读性和架构扩展基础，但在高级调度、内存优化、安全权限及多核支持等方面仍有较大的改进空间。
