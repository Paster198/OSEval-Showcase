# OS内核项目技术画像与评估报告：冰清玉洁YWD

## 一、 项目基本信息

- **项目名称**：冰清玉洁YWD
- **参赛单位**：合肥工业大学
- **目标架构**：RISC-V 64位（支持 QEMU virt 虚拟机与 Kendryte K210 开发板）
- **实现语言**：C语言（内核主体与用户态程序）、汇编语言（启动引导与上下文切换）、Perl（构建脚本）
- **生态归属**：基于 xv6-riscv 教学操作系统架构进行深度改造与扩展
- **代码规模**：总计约 29,206 行（内核C/汇编源码 16,526 行，内核头文件 7,076 行，用户态程序 5,604 行）
- **核心特点**：引入每进程独立内核页表、物理页引用计数机制、FAT32文件系统支持、按需页面加载（Demand Paging）以及兼容部分 Linux RISC-V ABI 的 52 个系统调用。

## 二、 实现的子系统与功能

1. **启动引导子系统**：支持 QEMU 和 K210 双平台入口，依赖 RustSBI 固件完成 M-mode 到 S-mode 的特权级跳转。
2. **内存管理子系统**：包含带引用计数的物理页分配器、Sv39 三级虚拟内存管理、写时复制（CoW）机制以及 `mmap` 内存映射功能。
3. **进程管理子系统**：基于动态空闲链表的进程控制块（PCB）分配，支持 `fork`/`clone`/`exec`，实现基础轮转调度与时间片机制。
4. **中断与异常处理子系统**：区分用户态与内核态陷阱，实现指令、读、写三种缺页异常处理，支持延迟分配与按需代码加载。
5. **文件系统子系统**：实现 FAT32 文件系统规范，支持长/短文件名、目录遍历、文件读写，以及基于哈希链表的缓冲区缓存和 `mount`/`umount` 挂载机制。
6. **设备驱动子系统**：包含 VirtIO MMIO 块设备驱动、K210 外设驱动（SPI、GPIO、FPIOA、SD卡、DMA）以及基于 SBI 的控制台字符驱动。
7. **系统调用子系统**：实现 52 个系统调用，涵盖进程控制、文件操作、内存管理、IPC、系统信息获取及调试跟踪。
8. **同步机制子系统**：提供基于原子指令的自旋锁和基于睡眠/唤醒机制的睡眠锁。
9. **用户态交互环境**：包含支持管道、重定向和环境变量的 Shell，以及基础 Unix 工具和系统测试程序。

## 三、 各个子系统的实现完整程度

- **设备驱动**：80%（VirtIO 驱动完整，K210 驱动受限于硬件未作运行时验证）
- **物理内存管理**：75%
- **启动引导**：75%
- **用户态程序**：75%

- **进程管理**：65%

## 四、 各个子系统的优缺点及实现细节

### 1. 内存管理子系统
- **实现细节**：物理分配器引入 `mm_table` 数组记录每个物理页的引用计数；虚拟内存采用每进程独立内核页表设计，将内核栈映射到固定的虚拟地址 `VKSTACK`。
- **优点**：引用计数机制为写时复制和共享映射提供了必要的基础设施；独立内核页表简化了内核栈的虚拟地址管理。
- **缺点**：`mm_table` 自身占用物理内存，减少了可用页数；`kfree` 中的边界检查使用 `kernel_end` 而非 `pa_addr_start`，存在越界隐患；独立内核页表导致每次进程切换需执行两次 `satp` 写入与 `sfence.vma`，增加了 TLB 刷新开销；缺页处理中修改页表后未调用 `sfence.vma`，可能导致 TLB 不一致。

### 2. 进程管理与调度子系统
- **实现细节**：PCB 通过动态空闲链表分配，每个 PCB 占用独立物理页；调度器采用基础轮转算法，在定时器中断中递减时间片。
- **优点**：引入时间片概念，并在无就绪进程时使用 `wfi` 指令降低功耗。
- **缺点**：空闲链表分配逻辑存在混乱，新创建的进程被加入链表后又被立即取出；结构体中定义了优先级字段并提供了系统调用，但调度器中的优先级队列操作代码被注释，功能未实际生效；时间片默认值为 1，调度粒度过粗。

### 3. 文件系统与缓存子系统
- **实现细节**：解析 FAT32 BPB 结构，使用 50 个条目的双向链表缓存目录项；缓冲缓存采用 4 个哈希桶的双链表设计；支持最多 8 个挂载点。
- **优点**：FAT32 的实现提升了与主机系统交换文件的实用性；挂载机制扩展了文件系统的组织能力。
- **缺点**：FAT 簇分配采用线性扫描，缺乏空闲位图，效率低下；FAT 表本身无缓存机制；挂载实现中存在指针使用错误；缓冲缓存缺少 LRU 淘汰策略，且 `bget` 中在未持有锁的情况下调用 `sleep`，存在竞态条件风险。

### 4. 中断与异常处理子系统
- **实现细节**：通过 `scause` 区分异常类型，分别调用 `do_store_fault`、`do_load_fault` 和 `do_ins_fault` 处理写、读和指令缺页。
- **优点**：实现了按需加载 ELF 段和延迟分配，优化了进程启动时的内存占用。
- **缺点**：CoW 路径中先分配新页再释放旧页，在内存紧张时易导致分配失败；指令缺页处理要求虚拟地址必须页对齐，限制了 ELF 段映射的灵活性；对非法地址访问的检查不够严密。

### 5. 系统调用子系统
- **实现细节**：实现 52 个系统调用，采用 Linux RISC-V ABI 编号，支持基于掩码的 `trace` 跟踪功能。
- **优点**：接口数量丰富，覆盖了主流操作系统的核心功能，向 Linux 标准靠拢。
- **缺点**：部分实现存在参数索引错误（如 `sys_waitpid` 中 `argaddr` 索引错误）；`sys_times` 和 `sys_uname` 直接写入用户态地址，未通过 `copyout` 进行地址合法性验证，存在安全隐患。

## 五、 OS内核本身的实现完整度

整体完整度约为 **70%**（基准：一个可启动、可交互、能运行基本用户程序且各子系统无致命逻辑缺陷的操作系统内核）。
项目搭建了完整的操作系统核心框架，并在内存管理和文件系统上进行了有意义的架构扩展。然而，多处关键子系统（如进程调度、缺页处理、系统调用安全）存在未完成的实验性代码或逻辑缺陷。此外，构建脚本中的拼写错误和未定义的系统调用号表明代码在提交前未经过严格的编译审查。

## 六、 动态测试的设计和结果

- **测试设计**：计划使用 QEMU virt 机器（2核，8M内存）加载内核二进制文件与 FAT32 磁盘镜像，通过 RustSBI 固件引导，运行用户态 Shell 及 `ostest.c` 等综合测试程序，验证系统调用、文件系统与进程调度功能。
- **测试结果**：**启动失败**。RustSBI (v0.1.1) 在跳转至内核入口地址 `0x80200000` 后，hart 0 触发 `InstructionFault` 异常（`mepc: 0x80004282`），导致系统 panic。
- **原因分析**：预编译的旧版 SBI 固件与当前内核入口代码（栈空间计算方式）存在兼容性问题。由于该固件为仓库预编译的二进制文件，无法在当前环境中重新编译验证，导致未能获取完整的动态运行测试结果。

## 八、 总结评价

“冰清玉洁YWD”项目是一个基于教学操作系统架构进行深度改造的内核作品。项目在内存管理和文件系统方面进行了显著的架构扩展，引入的物理页引用计数、每进程独立内核页表、FAT32 文件系统以及按需页面加载机制，展示了开发团队对操作系统核心概念的深入理解与一定的工程探索能力。系统调用数量扩展至 52 个，体现了向主流操作系统 ABI 兼容的努力。

然而，该项目在工程严谨性和功能闭环上存在明显短板。代码中残留了大量未完成的实验性逻辑（如优先级调度、进程空闲链表），构建脚本中存在拼写错误，部分系统调用缺乏用户态地址安全校验。此外，由于 SBI 固件兼容性问题导致内核无法在当前环境中成功启动，暴露出项目在环境适配与集成测试方面的不足。总体而言，该项目具备较好的技术广度与设计思路，但在代码质量控制、边界条件处理以及系统稳定性方面仍有较大的提升空间。
