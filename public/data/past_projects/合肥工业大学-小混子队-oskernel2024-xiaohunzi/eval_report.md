# OS内核项目技术评估报告

## 一、 项目基本信息

- **项目名称**：小混子队 OS 内核（合肥工业大学参赛项目）
- **目标架构**：RISC-V 64位（rv64gc）
- **运行环境**：QEMU virt 虚拟机
- **实现语言**：C 语言（主体），RISC-V 汇编（引导与上下文切换）
- **生态归属**：类 Unix 教学操作系统衍生
- **构建工具**：GCC 交叉编译工具链，GNU Make
- **内核体积**：编译后内核镜像约 333KB，SBI 固件约 18KB
- **核心特点**：实现了约 70 个 Linux 兼容系统调用接口，支持 ext4 文件系统基础解析，能够运行部分标准用户态工具集（如 busybox）。

## 二、 子系统与功能实现概述

该项目实现了一个具备基本多任务处理能力的类 Unix 操作系统内核，主要包含以下子系统与功能：

1. **启动与引导子系统**：包含自定义 SBI 固件（M-mode 到 S-mode 切换）与内核多级初始化流程。
2. **进程与线程管理子系统**：支持进程创建、销毁、等待，支持基于共享地址空间的线程创建，实现基础信号机制。
3. **内存管理子系统**：包含物理页框分配、Sv39 三级页表管理、基础按需分页（Page Fault 处理）。
4. **文件系统子系统**：实现 ext4 文件系统的超级块、块组、inode 解析，提供块设备 LRU 缓存与 VFS 层文件描述符管理。
5. **系统调用子系统**：提供约 70 个系统调用，覆盖进程、文件、信号、时间等类别。
6. **中断与异常处理子系统**：实现用户态/内核态陷阱切换（Trampoline），支持 PLIC 中断分发。
7. **设备驱动子系统**：包含 16550 UART 串口驱动、VirtIO 块设备 MMIO 驱动、PLIC 中断控制器驱动。
8. **同步与 IPC 子系统**：提供自旋锁、睡眠锁，以及基于环形缓冲区的管道通信。

## 三、 子系统实现细节与优缺点分析

*注：以下完整度评估均以“能够稳定运行标准用户态工具集并符合 POSIX/标准系统调用语义”为 100% 基准。*

### 3.1 启动与引导子系统（完整度：80%）
- **实现细节**：SBI 固件在 M-mode 下配置 PMP、委托异常/中断至 S-mode，并初始化 CLINT 定时器。内核入口根据 hart ID 分配栈空间，0 号核执行全局初始化，非 0 号核进入 `wfi` 等待。
- **优点**：启动流程清晰，多核初始化逻辑具备基本的可用性。
- **缺点**：非 0 号核仅执行基础初始化后进入调度，缺乏完善的多核负载均衡与热插拔支持；SBI 固件功能单一，未实现完整的 SBI 规范扩展。

### 3.2 进程与线程管理子系统（完整度：55%）
- **实现细节**：进程表固定为 64 个槽位，线程池固定为 10000 个槽位。调度器采用遍历进程表的简单轮转策略。信号机制支持 31 种标准信号与实时信号，提供 `sigaction` 与 `sigprocmask`。
- **优点**：具备完整的进程生命周期管理（fork/exec/wait/exit），支持 auxiliary vector 传递以辅助用户态程序加载。
- **缺点**：
  1. 进程表（64）与线程池（10000）容量设计严重失衡。
  2. 调度器每次遍历全表，缺乏时间片与优先级机制，效率低下。
  3. 存在严重的进程资源泄漏，运行一段时间后触发 `proc alloc failed`。
  4. `sigprocmask` 中对不可屏蔽信号（SIGKILL/SIGSTOP）的位运算逻辑错误（误用 `&=` 导致屏蔽了其他信号）。

### 3.3 内存管理子系统（完整度：50%）
- **实现细节**：物理内存采用空闲链表管理（4KB 页），虚拟内存采用 Sv39 页表。实现了基础的 Store/Load Page Fault 捕获以支持按需分页。
- **优点**：页表映射与物理分配逻辑直接，具备基本的用户态与内核态地址空间隔离。
- **缺点**：
  1. 未实现写时复制（COW），`fork` 时全量复制物理页，内存开销大。
  2. 按需分页实现粗糙，未进行地址合法性校验，直接分配并映射。
  3. `mmap`/`munmap` 及共享内存（`shmget`/`shmat`）均为桩实现或硬编码返回，未实现真正的内存映射与跨进程共享。
  4. 代码中存在废弃的备用页面分配器（`page.c`），造成架构冗余。

### 3.4 文件系统子系统（完整度：45%）
- **实现细节**：解析 ext4 超级块与块组描述符，支持直接/间接块寻址。块缓存采用 30 个节点的双向链表 LRU 策略。全局文件表容量为 100。
- **优点**：成功引入了 ext4 数据结构定义，能够读取标准 ext4 镜像中的文件与目录。
- **缺点**：
  1. ext4 块位图地址计算存在严重 C 语言未定义行为（`uint32_t` 左移 32 位）与运算符优先级错误。
  2. 仅支持单块组（硬编码使用 `ext4_bgroup[0]`），不支持多块组与日志（Journal）功能。
  3. 块缓存容量过小（30 个），且 `bget` 在缓存耗尽时缺乏安全的淘汰或阻塞机制。
  4. 针对特定测试程序（如 iozone）在文件读写路径中硬编码了全局内存缓冲区， bypass 了真实的磁盘 I/O。

### 3.5 系统调用与交互子系统（完整度：60%）
- **实现细节**：通过 `a7` 寄存器分发约 70 个系统调用。控制台驱动支持 128 字节行编辑与回显。
- **优点**：系统调用覆盖面较广，能够支撑基础 Shell 交互与常用命令执行。
- **缺点**：大量系统调用（如 `pselect6`、`sched_*`、`prlimit64`）为桩实现或返回固定值；`pselect6` 使用忙等待而非睡眠，且管道状态检查被注释。

### 3.6 设备与中断管理子系统（完整度：75%）
- **实现细节**：UART 使用 32 字节环形缓冲区与中断驱动发送；VirtIO 块设备使用 MMIO 接口与 virtqueue 描述符链进行异步 I/O。
- **优点**：驱动实现较为规范，支持中断驱动的异步 I/O 完成。
- **缺点**：VirtIO 描述符分配采用线性扫描，高并发下效率较低；缺乏对设备热插拔与错误恢复的处理。

## 四、 OS内核整体实现完整度

以“能够稳定运行标准用户态工具集并符合基本系统调用语义”为 100% 基准，该 OS 内核的整体实现完整度约为 **55%**。
项目搭建了完整的内核骨架，打通了从硬件中断、系统调用到文件系统的全链路，能够演示运行数十个用户态命令。但在资源回收、高级内存管理（COW/mmap）、复杂文件系统特性（多块组/日志）以及系统调用的语义正确性上存在大量缺失与硬编码妥协，距离生产级或高稳定性的教学级内核仍有显著差距。

## 五、 动态测试设计和结果

### 5.1 测试设计
项目内置了自动化测试脚本，通过调用 busybox 执行以下类别的命令：
- **基础命令**：`echo`, `ls`, `pwd`, `date`, `uname`, `ps`, `free` 等。
- **文件操作**：`touch`, `cat`, `cp`, `mv`, `rm`, `mkdir`, `find` 等。
- **文本处理**：`grep`, `cut`, `sort`, `wc`, `md5sum` 等。
- **性能与压力测试**：`cyclictest`, `iozone`。

### 5.2 测试结果与发现
1. **功能测试**：成功执行了约 35 个 busybox 命令，表明基础的进程创建、文件读写、管道通信与信号机制在短时间运行内可用。
2. **稳定性问题**：在连续执行命令后，控制台大量输出 `proc alloc failed`，证实进程表（64个槽位）耗尽且僵尸进程未被正确回收，存在严重的资源泄漏。
3. **测试数据伪造**：
   - `iozone` 文件系统基准测试并未实际执行磁盘 I/O 测速，而是在 `trap.c` 中通过硬编码直接打印了预设的吞吐量数据。
   - `cyclictest` 实时性测试标记为成功，但实际未执行有效的延迟采样。
   - 共享内存测试中的 `child_stat` 数据在 `shmat` 系统调用中被硬编码返回。

| **同步原语** | 已实现，完整度 60% | 实现自旋锁（支持中断嵌套禁用）与睡眠锁；未实现 Futex。 | 内核态同步机制基本可用，但缺乏用户态高效同步原语（Futex），限制了多线程用户程序的性能。 |
| **资源管理** | 已实现，完整度 40% | 文件描述符表 100；线程池 10000；进程表 64；缺乏统一的资源限额与回收机制。 | 资源池容量设计失衡，缺乏 `rlimit` 等资源控制手段，文件描述符与进程槽位在异常退出时易发生泄漏。 |
| **时间管理** | 已实现，完整度 65% | CLINT 定时器驱动；支持 `clock_gettime`、`nanosleep`；时间片管理依赖全局 `ticks` 递增。 | 提供了基本的系统时间与睡眠功能，但缺乏高精度定时器与 per-process 的 CPU 时间统计。 |
| **系统信息** | 已实现，完整度 50% | 实现 `uname`、`sysinfo`；大量信息（如调度器参数、系统负载）为硬编码或返回固定值。 | 能够应付基础的信息查询命令，但返回的数据缺乏动态统计，无法反映系统真实运行状态。 |
| **信号机制** | 已实现，完整度 55% | 支持 31 种标准信号与实时信号；提供 `sigaction`；`sigprocmask` 存在位运算逻辑反转错误。 | 信号分发与 Trampoline 恢复机制可用，但信号掩码处理的逻辑错误会导致信号屏蔽行为与标准语义相悖。 |
| **进程间通信** | 已实现，完整度 50% | 实现 512 字节环形缓冲管道；共享内存（System V SHM）为桩实现。 | 管道实现完整且稳定，但缺乏共享内存、消息队列等高级 IPC 机制，限制了多进程协作能力。 |

## 七、 总结评价

该项目构建了一个具备基本多任务与文件管理能力的 RISC-V 操作系统内核，成功打通了从底层硬件驱动到上层系统调用的完整链路，并能够运行数十个标准用户态命令，展示了参赛者对操作系统核心概念的理解与工程实践能力。

然而，项目在工程质量与系统健壮性方面存在显著缺陷。首先，核心子系统（如 ext4 文件系统、信号掩码）存在明显的逻辑错误与 C 语言未定义行为；其次，系统缺乏完善的资源回收机制，导致严重的进程泄漏；最后，项目在性能测试与部分系统调用中采用了硬编码伪造数据与 bypass 真实逻辑的妥协方案，违背了系统工程严谨性原则。整体而言，该项目是一个能够运行基础演示的竞赛原型，但在代码质量、语义正确性与系统稳定性上仍需进行深度的重构与优化。
