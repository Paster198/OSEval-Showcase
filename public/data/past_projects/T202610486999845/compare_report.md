# 对比分析报告

## 一、对比项目概览

本报告对以下六个 OS 内核项目进行多维对比分析：

| 编号 | 项目名称 | 所属队伍 | 语言 | 架构 | 生态归属 | Syscall 数 | 代码规模 |
|------|----------|----------|------|------|----------|-----------|----------|
| P1 | **alRED OS** | — | Rust | RV64+LA64 | 完全自研 | 282 | ~47,200 行 |
| P2 | TatlinOS | 华中科技大学-塔特林设计局 | Rust | RV64+LA64 | 完全自研 | 100+ | ~100+ .rs 文件 |
| P3 | Del0n1x | 哈尔滨工业大学（深圳） | Rust | RV64+LA64 | 完全自研 | 150+ | ~35,320 行 |
| P4 | ZeroOS (KeepOnOS) | 南开大学-萌新 | Rust | RV64 | ArceOS/Starry | 101 | ~61,441 行 |
| P5 | SC7 | 武汉大学-智核速启队 | C | RV64+LA64 | XV6 | 144 | ~56,662 行 |
| P6 | NoAxiom-OS | 杭州电子科技大学 | Rust | RV64+LA64 | 完全自研 | 115 | ~356 .rs 文件 |

## 二、架构设计对比

| 维度 | alRED OS | TatlinOS | Del0n1x | ZeroOS | SC7 | NoAxiom-OS |
|------|----------|----------|---------|--------|-----|------------|
| **内核类型** | 宏内核 | 宏内核 | 宏内核 | 宏内核 | 宏内核 | 宏内核 |
| **分层架构** | 隐式分层（模块化） | 明确分层 | HAL+内核二层 | 四层 crate 组件化 | HAL+HSAI+Kernel 三层 | HAL+Platform+Kernel 三层 |
| **模块化程度** | 中（syscall 层高度耦合） | 中高（子系统边界清晰） | 中高（HAL 隔离良好） | 高（~50 crate 组件化） | 中高（分层解耦良好） | 高（细粒度模块化） |
| **架构抽象** | 双架构共享 ~90% 代码 | 双架构通过 trait 抽象 | 条件编译隔离双架构 | 单一 RV64 架构 | 双架构 HAL/HSAI 分离 | trait 抽象双架构 |
| **并发模型** | 单核协作式 | 单核抢占式(1Hz) | 多核协程式(2核) | 多核异步执行器 | 单核轮询式 | 多核异步协程式 |
| **调度模型** | 协作线程模型 | 轮转调度(RR) | 优先级+协程 | FIFO/RR/CFS 可选 | 轮询遍历 | 多级优先级+时间片 |

**对比分析**：

- **alRED OS** 采用最务实的架构策略：其调度模型刻意选择了单核协作式，避免了多核同步的复杂性，从而将工程精力集中投放在系统调用广度上。但代价是 `syscall/fs.rs` 和 `syscall/process.rs` 中存在 150+ 个 `static mut` 变量，模块内部耦合度极高。
- **TatlinOS** 在分层设计上最为传统和规范：进程(Process)与线程(TCB)解耦、VFS trait 抽象清晰，但 1Hz 的时钟中断频率限制了调度精度。
- **Del0n1x** 和 **NoAxiom-OS** 共享"无栈协程异步调度"的技术路线，但 NoAxiom-OS 在协程基础设施的完善度上更胜一筹（实现了 O(1) 风格的 ExpiredScheduler、CPU 亲和性掩码），而 Del0n1x 的调度器仅分三级队列，缺乏时间片轮转。
- **ZeroOS** 的模块化程度最高——~50 个 crate 组件化架构使得子系统解耦非常彻底，但这也意味着其高度依赖 ArceOS/Starry 上游框架，自主可控性相对较低。
- **SC7** 是唯一的 C 语言项目，其 HAL/HSAI/Kernel 三层架构在双架构解耦方面设计最为严谨，但数据结构选型（静态数组池、链表 VMA）在可扩展性上不及 Rust 项目灵活。

## 三、子系统实现深度对比

### 3.1 内存管理

| 维度 | alRED OS | TatlinOS | Del0n1x | ZeroOS | SC7 | NoAxiom-OS |
|------|----------|----------|---------|--------|-----|------------|
| **页表机制** | Sv39 | Sv39 + LA64 | Sv39 + LA64 | Sv39 | Sv39 + LA64 | Sv39 + LA64 4级 |
| **物理分配器** | 静态帧池(线性扫描) | 伙伴系统 + PageCache | 栈式+LIFO回收 | Bitmap页分配+Slab/Buddy | 伙伴系统(0-10阶)+Slab | 全局自旋锁帧分配 |
| **COW** | 未实现 | 已实现 | 已实现 | 未实现(注释提及) | 已实现 | 已实现 |
| **懒分配** | 已实现 | 已实现 | 已实现 | 已实现(Demand Paging) | 已实现(PROT_NONE) | 已实现 |
| **mmap** | 已实现(MAP_SHARED) | 已实现(匿名+文件) | 已实现 | 已实现(匿名+文件+mremap) | 已实现(mmap/munmap) | 已实现(文件+匿名) |
| **共享内存** | MAP_SHARED 页共享 | System V SHM+GroupManager | System V SHM | IPC_PRIVATE SHM | System V SHM | System V SHM |
| **页面回收** | 无 | 无 | OOM 自动释放页缓存 | 无 | 无 | 超限全量遍历清理 |
| **Swap** | 无 | 无 | 无 | 无 | 无 | 无 |

**对比分析**：

- **TatlinOS** 的内存管理在所有项目中最为完整和现代：伙伴系统 + 页缓存（带高低水位线控制）+ COW + 懒分配 + GroupManager 共享页管理，构成了完整的现代内存管理链路。PageCache 的水位线设计（高水位 128 页、低水位 32 页、批量分配/回收）是明确的工程优化亮点。
- **alRED OS** 的内存管理策略最为特殊：Eager Fork 快照机制替代了 COW，静态帧池替代了动态分配器。这种设计在竞赛单核场景下功能正确且实现简单，但 4800 个快照帧（约 19MB）的内存开销显著大于 COW 方案。
- **Del0n1x** 的 OOM 自动降级机制（先释放 /tmp 页缓存，再释放 LTP 目录页缓存）具有实用价值，是在缺乏 Swap 条件下的一种务实补充。
- **SC7** 的伙伴系统 + Slab 分配器组合是六者中最接近传统 Unix 内核的设计，但 VMA 采用双向循环链表线性查找，在大规模地址空间下性能不佳。
- **NoAxiom-OS** 的 `msync` 被标记为 `unimplemented!()`，是一个明显的功能缺口。

### 3.2 进程与任务管理

| 维度 | alRED OS | TatlinOS | Del0n1x | ZeroOS | SC7 | NoAxiom-OS |
|------|----------|----------|---------|--------|-----|------------|
| **fork 策略** | Eager Fork 快照 | COW fork | COW fork | COW fork | COW fork | COW fork |
| **clone 标志位** | CLONE_VM/THREAD/VFORK 等 | 完整支持 | CLONE_VM/FILES/SIGHAND/THREAD 等 | CLONE_VM/FILES/THREAD 等 | 完整支持(含线程取消) | CLONE_VM/VFORK/THREAD 等 |
| **线程模型** | 协作线程(64+63) | TCB 与 PCB 分离 | 协程式线程 | 基于 async-task | POSIX 线程(静态池) | 基于 async-task |
| **信号机制** | 完整 64 信号+实时信号 | 完整 64 信号+实时信号 | 标准+实时信号 | 标准信号(缺STOP/CONT) | 完整 64 信号(含SA_SIGINFO) | 64 信号(缺严格排队) |
| **Futex** | 简化(无等待队列) | 完整(WAIT/WAKE/REQUEUE) | 完整(WAIT/WAKE/REQUEUE) | 完整(含 robust list) | 完整(含超时+WAITV) | 完整(私有/共享队列) |
| **调度策略** | 协作式(无抢占) | 轮转(RR), 1Hz | 三级优先级协程 | FIFO/RR/CFS 可选 | 轮询遍历 O(N) | 多级优先级+时间片 |
| **多核支持** | 无(单核) | 无(HART_NUM=1) | 有(2核 SMP) | 有(SMP) | 有(框架, 当前单核) | 有(多核) |

**对比分析**：

- **alRED OS** 的 Eager Fork 是六者中最独特的 fork 实现：在不支持 MMU 上下文切换的单核环境中，通过全量内存快照实现类 Unix fork 语义，并支持 partial fork（FORK_SKIP 跳过 I/O 缓冲区页）和 3 层嵌套。这是一个在受限环境下的创新工程方案，但 COW 的缺失意味着 fork 的内存开销在所有项目中最大。
- **NoAxiom-OS** 的进程管理数据结构设计最为精细——`Task` 结构体按并发访问模式将字段划分为 `Mutable`（锁保护）、`ThreadOnly`（无锁）、`Immutable`、`SharedMut` 四类，有效降低了锁竞争。
- **TatlinOS** 的 Process/TCB 分离设计在概念上最接近 Linux 内核，Futex 实现与定时器系统的深度集成（二叉堆管理超时）是其亮点。
- **SC7** 是唯一支持 POSIX 线程取消机制（PTHREAD_CANCEL_ENABLE/DEFERRED）的项目，在 POSIX 兼容性上最为深入。但其线性遍历调度器在进程数较多时效率明显低于其他项目的队列式调度器。
- **Del0n1x** 的文件描述符分配采用"位图 + 释放栈缓存"双重优化，在高频 open/close 场景下具备 O(1) 时间复杂度优势。
- **ZeroOS** 的硬编码内核栈上限（110 个）和 FD 上限（1025）是明显的可扩展性瓶颈。

### 3.3 文件系统

| 维度 | alRED OS | TatlinOS | Del0n1x | ZeroOS | SC7 | NoAxiom-OS |
|------|----------|----------|---------|--------|-----|------------|
| **VFS 层** | 无独立 VFS（Overlay 直接集成） | 有(Inode/File trait) | 有(Dentry/Inode/File) | 有(RootDirectory 多挂载) | 有(完整抽象层) | 有(trait 抽象) |
| **主文件系统** | EXT4(只读)+内存 Overlay | EXT4(lwext4 读写) | EXT4(lwext4 读写) | EXT4/FAT | EXT4(lwext4 移植)+VFAT | EXT4/FAT32/RamFS |
| **文件系统种类** | 1+Overlay | 1(EXT4) | 3(EXT4+devfs+procfs) | 4(EXT4+FAT+ramfs+devfs) | 2(EXT4+VFAT)+procfs | 5(EXT4+FAT32+RamFS+ProcFS+DevFS) |
| **页缓存** | 无（内存 Overlay 替代） | 无 | 有(PageCache+脏块追踪) | 无 | 无 | 有(MSI 协议+LRU 块缓存) |
| **写支持** | Overlay 层写 | 真实 EXT4 写 | 真实 EXT4 写 | 真实读写 | 真实读写 | 真实读写 |
| **符号链接** | 部分支持 | 支持 | 支持 | 内存 Map 模拟 | 支持 | 支持 |
| **procfs** | 最小 stub | 无 | 部分节点 | 空壳(ramfs 挂载) | 完整实现 | 详细实现 |

**对比分析**：

- **NoAxiom-OS** 的文件系统子系统在数量（5 种）和质量（MSI 页缓存协议、LRU 块缓存、ProcFS 详细节点）上都是六者中最强的。但 `fsync`/`fdatasync` 为空操作，存在数据丢失风险。
- **Del0n1x** 的页缓存 + 脏块追踪机制（DirtySet）+ Dentry 缓存（LRU 容量 20）构成了完整的文件系统性能优化链路，在 I/O 性能方面优于大多数项目。
- **alRED OS** 的 Overlay VFS 是独特的架构创新：通过在只读 EXT4 之上叠加内存文件层，使依赖写操作的测试程序能在只读镜像上透明运行，且支持跨 exec 的 overlay 保留模式（lmbench 多命令共享 /var/tmp）。但缺少独立的 VFS 抽象层，代码耦合在 syscall/fs.rs 中。
- **TatlinOS** 的 VFS 层（Inode/File trait）设计合理，但仅支持 EXT4 单一文件系统类型，缺乏 procfs/sysfs 等虚拟文件系统。
- **SC7** 支持 chroot 和扩展属性（xattr），在文件系统高级特性上最为深入，但其 ext4 核心依赖外部 lwext4 库移植，自主可控性受限。
- **ZeroOS** 针对 FAT32 不支持符号链接的问题，在用户态路径解析层通过 LINK_PATH_MAP 进行内存映射模拟，是一个巧妙的工程妥协。

### 3.4 网络支持

| 维度 | alRED OS | TatlinOS | Del0n1x | ZeroOS | SC7 | NoAxiom-OS |
|------|----------|----------|---------|--------|-----|------------|
| **协议栈** | 无（本地模拟） | 无（全局队列模拟） | smoltcp TCP/UDP | smoltcp TCP/UDP | 无（Socket 框架桩） | smoltcp TCP/UDP |
| **IPv6** | 模拟 loopback | 无 | 无 | 无 | 无 | 支持 |
| **Unix Socket** | 路径匹配模拟 | 管道组合模拟 | 未实现 | 未实现 | 未实现 | 未实现 |
| **网卡驱动** | 无 | 无 | VirtIO 网卡 | VirtIO/IXGBE | 无 | VirtIO 异步驱动 |
| **epoll** | 已实现 | 未实现 | 未实现 | 未实现 | 未实现 | 未实现 |
| **竞赛网络测试** | iperf/netperf harness 模拟 | 无 | 可用 | iperf 未通过 | 无 | iperf 排名第1 |

**对比分析**：

- **NoAxiom-OS** 的网络子系统在性能上遥遥领先——竞赛中 iperf 排名第 1、性能测试总分第 2，其异步驱动 + smoltcp 深度集成的效果得到了实际验证。
- **Del0n1x** 和 **ZeroOS** 均基于 smoltcp 实现了可用的 TCP/UDP 通信，但 ZeroOS 的 iperf 测试未通过。
- **alRED OS** 的网络"支持"本质上是本地缓冲区模拟，无真实协议栈，iperf/netperf 测试仅通过 harness 直接调用 socket API 完成形式上的输出。这在网络维度上是六者中最弱的——但这也是一种刻意的设计选择（优先保障其他子系统）。
- **SC7** 仅实现了 Socket 接口框架，底层无协议栈，网络功能为六者中最弱。
- **TatlinOS** 的套接字通过组合两个管道模拟，仅校验 127.0.0.1 地址。

- **SC7** 的 144 个系统调用在 C 语言项目中数量突出，且核心调用实现完整，与 POSIX 标准的兼容性在六者中最为深入。

## 四、技术亮点与创新对比

### alRED OS
1. **Eager Fork 快照机制**（高创新性）：在无 MMU 上下文切换的单核环境中，通过全量内存快照实现类 Unix fork 语义，支持 3 层嵌套和 partial fork。
2. **内存 Overlay VFS**（中高创新性）：在只读 EXT4 上叠加内存文件层，使测试程序可"写入"只读镜像。
3. **Compat Bridge 渐进迁移策略**（中等创新性）：9 类分类 bridge（约 59 个 case），保证评测流程不断的同时提供清晰的渐进增强路线。
4. **282 个系统调用的广度覆盖**（工程亮点）：AIO、xattr、inotify、timerfd、signalfd、eventfd、pidfd、BPF 等高级调用在其他项目中少见。

### TatlinOS
1. **页缓存 + 水位线控制**（中高创新性）：PageCache 的高/低水位线（128/32 页）和批量分配/回收机制，有效降低了物理页分配器的锁竞争。
2. **GroupManager 共享页管理**（中等创新性）：通过 BTreeMap 管理 MAP_SHARED 场景的共享物理页，避免重复分配。
3. **Futex 与定时器深度集成**（中等创新性）：基于二叉堆的定时器管理实现 Futex 超时唤醒。

### Del0n1x
1. **内核态无栈协程异步调度**（高创新性）：基于 async/await 实现内核异步任务调度，所有系统调用为 async 函数，I/O 等待零开销。
2. **FD 分配位图+缓存优化**（中等创新性）：位图 + 释放栈缓存在高频 open/close 场景下实现 O(1) 分配。
3. **OOM 分级释放策略**（中等创新性）：先释放 /tmp 页缓存，再释放 LTP 目录页缓存，在无 Swap 条件下提供实用降级方案。

### ZeroOS
1. **高度模块化组件架构**（高工程价值）：~50 个 crate 组件化设计，子系统边界清晰，通过 condition compilation 和 trait 实现灵活组合。
2. **双平台硬件适配**（中等创新性）：不仅支持 QEMU，还深度适配 VisionFive2 开发板，自研 PLIC、RTC、SD 卡驱动。
3. **FAT32 链接模拟**（中等创新性）：在用户空间层面通过 LINK_PATH_MAP 模拟符号链接，解决 FAT32 不支持链接的问题。

### SC7
1. **双架构 HAL/HSAI 分层解耦**（高工程价值）：HAL（硬件抽象）+ HSAI（硬件服务抽象）双层架构在双指令集解耦上最为系统和严谨。
2. **POSIX 线程取消机制**（中等创新性）：支持 PTHREAD_CANCEL_ENABLE/DEFERRED/ASYNCHRONOUS，在 POSIX 深度兼容上领先。
3. **UTS 命名空间支持**（中等创新性）：在六者中唯一实现了命名空间隔离（虽仅 UTS 维度）。

### NoAxiom-OS
1. **异步陷阱处理**（高创新性）：`user_trap_handler` 本身是 async 函数，允许在缺页异常等处理中直接进行异步 I/O，这是六者中最激进的异步集成。
2. **细粒度并发数据模型**（高创新性）：Task 结构体按 Mutable/ThreadOnly/Immutable/SharedMut 分类锁保护，死锁检测宏（`assert_no_lock!`）增强并发健壮性。
3. **竞赛性能验证**（高工程价值）：性能测试总分第 2、iperf 第 1，是六者中唯一有明确竞赛性能数据的项目。

## 五、不足与缺失对比

## 七、各项目总结评价

### alRED OS

alRED OS 是一个**系统调用广度见长、竞赛策略务实的 Rust 双架构内核**。其 282 个系统调用和 Overlay VFS 设计体现了"宽度优先"的开发理念。Eager Fork 快照机制在单核受限环境下是一种有创意的工程折衷，Compat Bridge 策略则为渐进式功能增强提供了清晰的路线图。主要短板在于 150+ 个 `static mut` 变量的全局状态耦合、网络功能的本地模拟性质、以及 LoongArch 端仅为框架的实现差距。该系统最适合需要快速覆盖大量系统调用接口的竞赛场景。

### TatlinOS

TatlinOS 是一个**内存管理设计现代、架构规范的传统宏内核**。其伙伴系统 + 页缓存（水位线控制）+ COW + 懒分配 + System V 共享内存的完整内存管理链路在六者中无人能及。VFS trait 抽象和 Process/TCB 分离设计展现了良好的架构素养。但 1Hz 时钟中断导致的调度精度问题、网络功能的缺失（全局队列模拟）、以及单核限制是其主要不足。如果提升时钟精度并引入多核支持，该项目有冲击第一梯队的潜力。

### Del0n1x

Del0n1x 是一个**核心功能扎实、协程架构有特色的竞技型内核**。其无栈协程异步调度降低了上下文切换开销，OOM 分级释放和 FD 分配优化体现了工程实用性。Dentry 缓存和脏块追踪机制使文件系统性能在同类项目中处于前列。决赛第 8 名的成绩验证了其综合能力。主要不足在于缺少 epoll、Unix Socket、以及调度策略相对简单（无时间片轮转）。该项目在保持当前架构优势的基础上，补充高级 I/O 多路复用和 Unix Socket 支持，竞争力将显著提升。

### ZeroOS (KeepOnOS)

ZeroOS 是一个**模块化程度最高、硬件适配范围最广的组件化内核**。~50 个 crate 的组件化设计使子系统边界最为清晰，双平台（QEMU + VisionFive2）硬件适配展现了扎实的底层开发能力。CFS 调度器的可选支持是其区别于多数项目的亮点。但单架构限制（仅 RV64）、FAT32 链接模拟的脆弱性、硬编码资源上限（110 个内核栈、1025 个 FD）以及空壳 procfs 等工程妥协，使其在通用性和鲁棒性上打了折扣。该项目最显著的价值在于证明了 ArceOS 组件化框架在大赛场景下的可行性。

### SC7

SC7 是一个**POSIX 兼容性最深、C 语言工程实现扎实的传统型内核**。UTS 命名空间、POSIX 线程取消机制、资源限制（rlimit）的完整实现、以及 HAL/HSAI 双层架构解耦，使其在标准化兼容性和架构严谨性上表现突出。56,000+ 行的 C 代码规模也显示了团队的工程投入。但 O(N) 调度器、链表 VMA 查找、静态池设计以及网络协议栈的完全缺失，使其在高并发和网络场景下受限。该项目作为 XV6 教学内核的深度扩展，在教学和研究价值上高于竞赛实用性。

### NoAxiom-OS

NoAxiom-OS 是六者中**综合能力最强的 Rust 异步内核**。其无栈协程异步架构不仅在调度和 I/O 路径上实现了零成本抽象，还通过异步陷阱处理将异步范式延伸至异常处理层面——这是所有项目中最激进的异步集成。5 种文件系统、MSI 页缓存协议、细粒度并发数据模型（Mutable/ThreadOnly/Immutable/SharedMut 分类）、以及竞赛性能测试总分第 2 和 iperf 第 1 的成绩，共同构成了一个在架构创新、功能完整度、实际性能三个维度上均表现突出的项目。主要改进空间在于 fsync/msync 的空实现、CFS 的废弃状态、以及 Unix Socket 的缺失。若补齐这些短板，该项目将达到接近生产级内核的完成度。

## 八、评审意见

本次对比的六个 OS 内核项目均达到了较高的工程完成度，代表了当前国内高校在操作系统内核领域的优秀水平。总体呈现以下格局：

**第一梯队（综合能力强）**：**NoAxiom-OS** 和 **Del0n1x**。两者均采用 Rust 无栈协程异步架构，在架构创新性、子系统完整度和实际性能三个维度上均表现突出。NoAxiom-OS 在文件系统丰富度和竞赛性能验证上更胜一筹，Del0n1x 在 OOM 处理和 FD 分配优化上有独到之处。

**第二梯队（特色鲜明但有明显短板）**：**TatlinOS**、**alRED OS**、**ZeroOS**。TatlinOS 的内存管理子系统为六者最强但调度精度过低；alRED OS 的系统调用广度无人能及但全局状态耦合度高且无真实网络栈；ZeroOS 的模块化架构最为优秀但受限于单架构和多个工程妥协。

**第三梯队（传统路线扎实但效率受限）**：**SC7**。C 语言 + XV6 派生路线在 POSIX 深度兼容性和架构严谨性上表现突出，但数据结构和算法的效率问题以及网络功能的缺失限制了其上限。

**对 alRED OS 的专项评审意见**：

alRED OS 采取的"宽度优先、策略务实"路线在竞赛场景中具有明确的合理性。282 个系统调用的注册数量、Eager Fork 快照机制的创新性、Overlay VFS 的实用设计以及 Compat Bridge 的渐进策略，构成了一个完整的"快速覆盖测试集"方案。然而，项目中 150+ 个 `static mut` 全局变量的使用是一个需要正视的工程质量问题——在单核协作场景下功能正确，但严重削弱了代码的可维护性和未来向多核/抢占调度演进的可能性。建议优先将高频使用的全局状态迁移到结构化的并发原语中。此外，LoongArch 端仅为框架（约 15% 完整度），与其声称的"双架构支持"存在较大差距；网络子系统依赖本地模拟而非真实协议栈，iperf/netperf 测试结果仅有形式意义。总体而言，alRED OS 在竞赛导向的"宽度"维度上表现出色，但在工程规范性和子系统"深度"上仍有显著的提升空间。
