# SystemNQB OS 内核项目技术报告

## 1. 项目基本信息

- **项目名称**: SystemNQB
- **目标架构**: RISC-V 64位 (riscv64gc)
- **实现语言**: C语言（约7,648行）、Rust语言（约1,200行）、汇编语言
- **生态归属**: 基于 xv6-riscv 框架进行二次开发
- **核心特点**: 
  - 采用 C 与 Rust 混合编程的内核架构。
  - 引入第三方 Rust 库（`ext4_rs` 用于文件系统，`buddy_system_allocator` 用于物理内存分配）。
  - 采用 Linux RISC-V 兼容的系统调用编号方案。
  - 使用内存磁盘（Memory Disk）替代传统块设备驱动，UART 输出依赖 SBI 调用。

## 2. 实现的子系统和功能

- **进程与线程管理**: 进程创建与销毁、上下文切换、轮转调度、基于 `clone` 系统调用的线程组支持。
- **内存管理**: Sv39 虚拟内存管理、基于伙伴系统的物理页分配、基础 `mmap`/`munmap` 内存映射。
- **文件系统**: ext4 文件系统读取与基础目录操作、LRU 块缓冲、内存磁盘后端。
- **系统调用接口**: Linux 兼容系统调用分发、进程/文件/内存/时间类系统调用。
- **中断与异常处理**: 用户态/内核态陷阱处理、Trampoline 跳转机制、PLIC 中断路由、时钟与 UART 中断。
- **设备驱动**: SBI UART 控制台、PLIC 中断控制器、内存磁盘（VirtIO 块设备驱动代码存在但已禁用）。
- **同步与 IPC**: 内核态自旋锁、睡眠锁、用户态管道（Pipe）。
- **可执行文件加载**: ELF 解析与加载、Shebang 脚本支持、辅助向量（Auxiliary Vector）传递。

## 3. 各个子系统的实现完整程度

*注：以下完整度评估以 xv6-riscv 原始功能为 100% 基础基准，以 Linux 基础 POSIX 功能为扩展基准。*

- **进程管理**: 基础功能 100% 覆盖 xv6 原始设计；扩展功能实现了 Linux 基础线程创建模型（`CLONE_THREAD` 等）。
- **内存管理**: 基础功能 100% 覆盖 xv6；扩展功能实现了基础匿名/文件映射，但缺乏按需分页和共享内存写回机制。
- **文件系统**: VFS 层 100% 覆盖 xv6；底层替换为 ext4，但缺乏日志系统，且存在初始化致命错误，实际可用完整度极低。
- **系统调用**: 实现了约 30 余项基础 Linux 系统调用，信号处理、IPC、网络等数十项系统调用仅返回 `-ENOSYS`。
- **陷阱与中断**: 100% 覆盖 xv6 基础中断与异常处理模型。
- **设备驱动**: 未直接操作硬件寄存器，依赖 SBI 和内存映射，硬件驱动完整度较低。
- **同步原语**: 100% 覆盖 xv6 的自旋锁与睡眠锁，未实现 POSIX 互斥锁/条件变量等高级用户态同步原语。

## 4. 各个子系统的优缺点以及实现细节

### 4.1 进程管理子系统
- **实现细节**: 最大支持 64 个进程（`NPROC=64`）和 8 个 CPU（`NCPU=8`）。每进程内核栈为 4 页。通过 `clone` 系统调用支持 `CLONE_THREAD`、`CLONE_PARENT_SETTID` 等标志位，实现线程组（`tgid`/`pid` 分离）。Rust 侧设计了基于对象池的 `Task` 分配机制，但未被 C 侧实际调用。
- **优点**: 在 xv6 基础上成功引入了线程组概念，支持基础的线程创建。
- **缺点**: 调度器仍为简单的 O(N) 轮转调度；Rust 侧的进程管理代码处于未集成状态，增加了代码冗余。

### 4.2 内存管理子系统
- **实现细节**: 物理内存布局将 1GB 内存划分为内核区、内存磁盘区（`0xA0000000`）和空闲区（`0xB0000000`）。物理页分配器由 C 侧转发至 Rust 侧的 `buddy_system_allocator`（64 阶）。虚拟内存采用 Sv39 三级页表。`mmap` 限制每进程最多 10 个映射（`NMMAP=10`），不支持 `MAP_SHARED` 写回。
- **优点**: 引入伙伴系统分配器，提升了物理内存分配的效率和灵活性。
- **缺点**: `mmap` 实现过于简化，缺乏缺页异常处理（Page Fault）和按需加载机制，所有映射在创建时即分配物理页。

### 4.3 文件系统子系统
- **实现细节**: VFS 层保留 xv6 的 `inode` 结构，底层通过 FFI 调用 Rust 侧的 `ext4_rs` 库。块缓冲层（`bcache`，容量 30）使用 LRU 策略，后端直接读写 `0xA0000000` 的内存磁盘。日志系统（`log.c`）为空实现。
- **优点**: 尝试集成更复杂的 ext4 文件系统以替代 xv6 原始的简单文件系统。
- **缺点**: `ext4_rs` 在解析超级块时因 `inodes_per_group` 为 0 导致除零 panic；缺乏日志系统导致无崩溃恢复能力；VirtIO 块设备驱动被禁用，脱离了真实块设备环境。

### 4.4 系统调用子系统
- **实现细节**: 通过 sed 脚本自动生成系统调用表和用户态桩代码。采用 Linux RISC-V 系统调用编号。
- **优点**: 提高了与 Linux 用户态程序（如 busybox）的兼容性潜力。
- **缺点**: 信号处理（`rt_sigaction` 等）、IPC、网络等大量系统调用未实现，限制了复杂用户态程序的运行。

### 4.5 陷阱与中断处理子系统
- **实现细节**: 使用 Trampoline 页进行用户态与内核态的页表切换和寄存器保存。通过 PLIC 管理外部中断，时钟中断通过 SBI 设置下一次触发。
- **优点**: 中断处理流程完整，内核态与用户态陷阱隔离良好。
- **缺点**: 属于 xv6 原生设计的直接继承，无额外创新。

### 4.6 设备驱动子系统
- **实现细节**: UART 输出直接调用 `sbi_console_putchar`。块设备通过 QEMU 的 `-device loader` 将镜像加载至物理内存，内核直接通过内存拷贝（`memmove`）进行读写。
- **优点**: 简化了硬件交互，降低了开发难度。
- **缺点**: 未实现真正的硬件驱动（如 16550 UART 或 VirtIO Block），无法在缺乏 SBI 和内存加载器的真实硬件上运行。

### 4.7 文件与管道子系统
- **实现细节**: 管道缓冲区大小为 512 字节（`PIPESIZE=512`）。`execve` 支持 ELF 解析，并实现了 Shebang 脚本委托执行。支持传递辅助向量（如 `AT_HWCAP`, `AT_PAGESZ`, `AT_RANDOM` 等）。
- **优点**: 辅助向量的支持提高了对现代 Linux 动态链接器的兼容性。
- **缺点**: 管道缓冲区较小，且由于文件系统崩溃，`execve` 无法实际加载外部程序。

## 5. OS 内核本身的实现完整度

基于 xv6-riscv 基础框架，内核的核心机制（进程、内存、中断、同步）完整度达到 100%。在扩展功能方面，系统调用接口实现了基础进程/文件/内存类调用，但文件系统因第三方库集成问题导致核心功能（`exec`、用户程序运行）不可用。整体而言，内核框架完整，但作为可运行的操作系统，其用户态支撑能力因文件系统初始化崩溃而严重受损。

## 6. 动态测试的设计和结果

- **测试设计**: 使用 QEMU 模拟 RISC-V 64 位环境（3 个 HART），加载 OpenSBI 固件，通过 `-device loader` 将 ext4 文件系统镜像加载至 `0xA0000000` 内存地址，启动内核。
- **测试结果**: 内核启动失败。
- **失败现象**: 内核在打印 `ext4 init ok` 和 `console ok` 后，触发 Rust panic：`panicked at third/ext4_rs/src/ext4_impls/inode.rs:22:21: attempt to divide by zero`。
- **原因分析**: `ext4_rs` 在计算 inode 所属块组时，除数 `inodes_per_group` 为 0。这表明传入的 ext4 镜像未被正确解析，或 `ext4_rs` 对特定块大小（如 1KB）的超级块解析存在缺陷。由于文件系统初始化失败，后续的 `init` 进程无法执行 `exec` 加载用户程序，导致系统无法进入用户态。

## 8. 总结评价

SystemNQB 是一个基于 xv6-riscv 框架进行二次开发的 OS 内核项目。该项目在架构设计上进行了 C 与 Rust 混合编程的尝试，引入了伙伴系统物理内存分配器和 ext4 文件系统，并采用了 Linux 兼容的系统调用编号方案，展现了团队在操作系统内核扩展方面的探索意愿。

然而，项目的实际可用性受到严重制约。核心问题在于文件系统子系统的集成失败：`ext4_rs` 库在初始化时因超级块解析缺陷触发除零 panic，导致内核无法完成启动流程，进而使得 `exec` 系统调用和用户态程序加载完全不可用。此外，项目大量依赖 SBI 调用和内存磁盘，禁用了 VirtIO 等真实硬件驱动，削弱了其在真实硬件环境下的适用性；信号处理、IPC 等关键系统调用的缺失也限制了其对复杂 Linux 用户态程序的兼容性。

总体而言，该项目在内核基础框架的扩展和混合语言架构设计上具有一定的工程价值，但在核心功能的稳定性、第三方库的集成调试以及硬件驱动的完整性方面存在明显不足，未能交付一个可稳定运行用户态程序的操作系统内核。
