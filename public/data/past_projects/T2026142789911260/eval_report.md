# weijun-eos-kernel 技术画像与评估报告

## 1. 项目基本信息

- **项目名称** : weijun-eos-kernel
- **目标架构** : RISC-V 64 (rv64gc, Sv39 分页，S-mode 运行)
- **实现语言** : Rust (edition 2024, 部分 unsafe 汇编)
- **生态归属** : 单体内核，OS 大赛赛道作品，未声称与任何现有内核兼容
- **主要特点** :
  - 手写 RISC-V 汇编陷阱入口及上下文切换
  - 基于伙伴系统的物理页帧分配器（176 MB 管理区域）
  - 完整进程模型（创建/调度/退出/回收），协作式轮询调度
  - 通过 `rsext4` crate 实现 ext4 文件系统读写
  - 基于 `virtio-drivers` crate 的 VirtIO-MMIO 块设备驱动
  - 管道 IPC，支持阻塞读/写同步
  - 约 50 余个 POSIX 系统调用，覆盖进程、文件、内存管理等
  - 嵌入引导 ELF 镜像，可无文件系统启动首个用户进程
- **代码规模** : 6 个 Rust 源文件，合计约 4155 行代码；链接脚本及构建文件另外

## 2. 子系统与功能实现概览

内核实现或部分实现了以下子系统：

- **启动与初始化** : 多阶段启动（汇编入口 → Rust 主函数 → 页表与 MMU 使能 → 设备初始化 → 首进程加载）
- **陷阱/中断处理** : 汇编级 TrapFrame 保存与恢复，支持用户态 ecall、断点及缺页异常（缺页处理为不可恢复）
- **内存管理** :
  - 基于伙伴系统的物理页帧分配器（18 阶，覆盖 4 KiB 至 512 MiB）
  - Sv39 三级页表操作（映射、取消映射、深拷贝）
  - 用户地址空间管理（堆、栈、mmap 区域）
- **进程管理** :
  - 进程控制块（PCB）含状态、PID、父子关系、文件描述符表、页表指针、TrapFrame 及内核上下文
  - 进程创建（clone 深拷贝页表）、执行（execve 重新加载 ELF）、退出与等待回收（wait4）
  - 协作式轮询调度器，依赖进程主动让出 CPU
- **文件系统与块设备** :
  - 基于 VirtIO-MMIO 的块设备驱动，完成扇区与 ext4 4KiB 块的映射
  - 通过 `rsext4` 库挂载并读写 ext4 文件系统（读写均已实现代码路径）
  - 文件描述符表（每进程 128 项），支持 stdin/stdout/stderr 及管道端点
- **管道 IPC** : 基于 `VecDeque` 的管道缓冲区（硬编码上限 4096 字节），支持阻塞读写及端点关闭检测
- **系统调用** : 实现了 50+ 个系统调用号的分发与处理，覆盖文件、进程、内存、系统信息等，时间与信号子系统为桩实现
- **ELF 加载器** : 支持静态链接 ELF 的 LOAD 段映射、用户栈构造（含辅助向量）及脚本 shebang 重定向

## 3. 各子系统实现完整度与细节分析

### 3.1 启动与初始化

**完整度** : 约 90%

**实现细节** : 从 OpenSBI 移交后的 `_start` 汇编开始，清零 BSS，设置栈指针，调用 `kmain` → `run`。`run` 中依次完成内核堆初始化（32 MiB 的 `LockedHeap`）、伙伴页分配器初始化、根页表构造（直接映射 DRAM 与 MMIO）、MMU 使能、陷阱向量设置、文件描述符表初始化、VirtIO 块设备挂载 ext4、内嵌引导 ELF 加载为首进程，最后 `sret` 进入用户态。

**优点** : 流程清晰，阶段划分合理；嵌入式引导 ELF 使内核可独立启动，降低对磁盘镜像的依赖。

**不足** : 未支持多核启动（仅单 hart）；栈顶定义在链接脚本中存在重复定义问题（初始定义覆盖为 `.stack` 段内的定义，虽最终一致但有歧义）。

### 3.2 陷阱与中断处理

**完整度** : 约 70%

**实现细节** : 汇编入口 `temporary_trap_entry` 保存完整的用户寄存器上下文（31 个 GPR + sepc/sstatus/scause/stval 等），之后调用 Rust 处理函数。支持 `UserEcall`（解析系统调用号并分发）、`Breakpoint`（跳过指令）。对于所有页面错误（指令/加载/存储）及非法指令，均触发 `unrecoverable()` 并关机。未注册定时器中断或外部中断处理函数。

**优点** : 寄存器保存/恢复行为规范，利用 `sscratch` 实现用户栈与内核栈的原子切换。

**不足** : 完全不具备按需分页能力，任何缺页异常均导致内核立即终止；无 Supervisor 定时器中断处理，无法实现抢占调度或时间统计；无外部中断处理。

### 3.3 内存管理

**完整度** : 约 60%（相对于支持按需分页、写时复制、页面回收等现代内核特性）

**实现细节** :
- 物理内存通过伙伴分配器 `MyPagingHandler` 管理，支持分配/释放，有基本的 buddy 拆分与合并逻辑。
- 虚拟内存通过 `PageTable64Cursor` 进行 4 KiB 粒度的映射/解映射，每次映射时分配物理页。
- `clone` 实现为页表深拷贝：遍历父进程所有 USER 页帧，分配新物理页并逐帧复制内容。
- `brk` 支持堆向上增长，`mmap` 支持匿名映射和基于文件的映射，但 `prot` 和 `flags` 参数被完全忽略，始终以 RW 权限映射。
- `munmap` 仅支持精确地址范围匹配释放，不支持部分释放；`mprotect` 为桩实现。

**优点** : 自实现的伙伴分配器整合了页表操作与 VirtIO DMA 的 `Hal` trait，避免了内存池分割。

**不足** :
- 不存在写时复制（fork 时深拷贝全量页表，效率低）。
- 不存在按需分页，所有映射在请求时立即分配物理页，且缺页处理为不可恢复。
- 伙伴分配器释放函数缺少地址对齐校验，已知会导致 `AlreadyMapped` 错误；order 存在理论上的溢出风险。
- 用户栈释放时地址非对齐问题加剧了分配器脆弱性。
- 无页面回收或交换。

### 3.4 进程管理

**完整度** : 约 75%

**实现细节** :
- 进程由 `Process` 结构表示，维护状态机（Ready/Running/Waiting/Zombie/Dead），通过父进程 `wait4` 回收僵尸进程资源。
- 调度器为简单的线性扫描轮询，选择下一个 `Ready` 状态进程，通过 `context_switch` 进行内核上下文切换（保存/恢复 `ra, sp, s0-s11`）。
- 进程间关系通过 `parent` 和 `children` 跟踪，支持基本的进程树。
- `clone` 支持创建子进程，但 `CLONE_VM` 标记会触发 `panic`，即无法创建线程。
- `execve` 会替换当前进程的用户态镜像（重新加载 ELF），并关闭带有 `CLOEXEC` 标志的文件描述符。

**优点** : 进程生命周期管理完整（创建、执行、退出、等待、回收），父子关系明确，资源在 `wait4` 时回收，防止僵尸泄露。

**不足** :
- **协作式调度**：完全依赖进程主动让出 CPU（管道阻塞、`wait4` 等待、主动退出）。没有抢占机制，任何用户态无限循环将导致系统挂起。
- 无进程组、会话等 POSIX 进程管理概念。
- `clone` 不支持共享地址空间，无法支持多线程。
- 根进程退出时直接关机，而非尝试重启或进入内核恐慌处理。

### 3.5 文件系统与块设备

**完整度** : 约 75%

**实现细节** :
- 块设备驱动扫描 MMIO 区域内的 VirtIO 设备，匹配 `Block` 类型后初始化 `VirtIOBlk`。通过 `VirtIoDisk` 适配器将 ext4 的 4 KiB 块操作转换为 512 字节扇区读写。
- ext4 文件系统通过 `rsext4` 库挂载，支持文件的打开、读写、目录遍历、创建/删除目录及文件等操作。
- 文件描述符层支持 stdin (UART 轮询)、stdout/stderr (SBI putchar)、普通文件和管道端点。
- 路径解析实现了 `.` 和 `..`，工作目录跟随进程。

**优点** : ext4 读写功能较完整，`getdents64` 手动构造 `linux_dirent64` 结构，兼容性良好；块设备扇区映射处理正确。

**不足** :
- 无 inode 缓存或目录项缓存，每次操作均直接通过 `rsext4` 访问磁盘；`total_blocks` 未实现。
- 未实现硬链接（`linkat` 仅返回 0）、挂载系统调用为桩、无权限检查。
- stdin 通过 UART MMIO 轮询实现，无中断驱动，会浪费 CPU 周期。
- 块设备使用全局可变变量 `BLK_DEVICE`，仅支持单设备。

### 3.6 管道 IPC

**完整度** : 约 85%

**实现细节** : 管道由 `Pipe` 结构管理，内部使用 `VecDeque<u8>` 作为缓冲区，硬编码最大 4096 字节。`pipe2` 创建一对文件描述符并记录读写端持有者的 pid。`read`/`write` 系统调用在管道上实现阻塞语义：当缓冲区空（读）或满（写）时，当前进程状态置为 `Waiting`，并协作调度到管道的另一端进程；当所有写端关闭时读返回 EOF，所有读端关闭时写返回 EPIPE。

**优点** : 阻塞逻辑正确，通过进程状态切换实现同步，符合 POSIX 管道基本语义；正确维护了端点引用计数。

**不足** : 仅支持阻塞模式，无 `O_NONBLOCK`；缓冲区大小固定为 4096 字节，无法调整；未集成 `select`/`poll`/`epoll` 等多路复用机制。

### 3.7 系统调用层

**完整度** : 约 55%（按实测功能比例）

**实现细节** : 定义 `SyscallOp` 枚举，通过 `from_sys` 将寄存器参数解析为具体操作。文件系统类系统调用（`openat`, `read`, `write`, `close`, `getdents64`, `fstat` 等）实现较为完整，多数能正确完成功能。进程管理类（`clone`, `execve`, `exit`, `wait4`）核心路径完整。内存管理类（`brk`, `mmap`, `munmap`）部分实现。时间相关（`clock_gettime`, `nanosleep` 等）和信号相关（`rt_sigaction` 等）均为桩实现。

**优点** : 系统调用分发架构清晰，可扩展性好；文件与进程相关调用的实现质量较高。

**不足** : 22 个系统调用为桩实现，覆盖了信号、时间、部分文件系统操作（`linkat`, `mount`, `umount2` 等），导致依赖这些调用的用户程序无法正常工作；`mmap` 忽略关键参数，功能受限。

### 3.8 构建与可移植性

**完整度** : 约 80%

**实现细节** : 使用 `Makefile` + `cargo` 构建，目标 `riscv64gc-unknown-none-elf`，依赖少量外部 crate。构建过程在给定环境中通过，输出约 1 MiB 的 ELF 内核镜像。可执行 `min_run_testcode` 通过 `include_bytes!` 嵌入，无需外部文件。

**优点** : 构建依赖清晰，编译过程顺利（仅有 240 个 Rust 新版 `static mut` 警告，无错误）。

**不足** : 使用大量 `static mut` 全局状态，致使内核无法适配多核或任何并发模型；链接脚本存在细微问题（栈顶重复定义）；未提供任何测试用例或 CI 配置。

## 4. OS 内核整体实现完整度

若以“能够稳定运行 musl-busybox 并通过基本 OS 大赛功能测试”为参考基准，该内核整体完整度约为 **65%**。核心进程与文件 I/O 路径已基本贯通，但时间子系统、信号子系统及抢占机制缺失，限制了用户态程序兼容性和实时性要求。

## 5. 动态测试的设计与结果

- **构建测试** :
  - 命令 : `make`
  - 结果 : **成功**，生成 `kernel-rv` ELF 文件
  - 编译器 : rustc 1.88.0
  - 二进制尺寸 : text 段约 409 KiB，data 段约 3 KiB，bss 段约 612 KiB
  - 警告 : 240 个（均与 `static mut` 相关）
  - 错误 : 0

- **运行时测试** : **未进行**。原因：缺少必需的 ext4 磁盘镜像文件 `sdcard-rv.img`，无法通过 QEMU 配合 VirtIO 块设备启动并验证文件系统操作。即使构造镜像，也需要确认与内核内嵌的块设备地址及分区布局一致。无测试结果可供分析。

## 6. 细则评价表格

| 评价条目 | 是否实现及完整度 | 关键发现 | 评价 |
|----------|------------------|----------|------|
| **内存管理** | 部分实现，约 60% | Sv39 页表与伙伴分配器结合，但无按需分页、无写时复制，缺页直接关机；`free` 缺乏对齐检查，存在已知 bug。 | 基础虚拟内存框架可用，但健壮性与高级特性缺失，影响多进程效率和可靠性。 |
| **进程管理** | 大部分实现，约 75% | 进程生命周期完整，支持创建、执行、退出与回收；调度为协作式轮询，无抢占，无法应对恶意循环。 | 模型清晰，资源回收逻辑较完善，但调度机制的缺位使得系统不具备公平性和实时性。 |
| **文件系统** | 大部分实现，约 75% | ext4 读写路径完整，`getdents64` 结构构造正确；但无缓存，`total_blocks` 未实现，部分文件系统调用为桩。 | 能满足基本文件操作需求，性能与完整性仍有较大提升空间。 |
| **交互设计** | 不适用 | 内核为参赛作品，无交互界面设计。 | 无评价。 |
| **同步原语** | 特殊实现 | 内核未提供互斥锁、信号量等传统同步原语；管道通过进程状态切换实现阻塞同步，协作式调度保证了内核数据结构的单核访问安全性。 | 单核下利用协作调度避免了数据竞争，但缺少通用同步机制，无法扩展到多核。 |
| **资源管理** | 大部分实现 | 文件描述符按进程管理，退出时回收 mmap、ELF 段、管道端点等；物理页通过伙伴分配器管理；但用户栈和内核栈在进程退出时未立即回收，推迟到父进程 wait4。 | 资源回收路径总体完整，延迟回收可能导致瞬时内存占用较高，但无泄漏。 |
| **时间管理** | 桩实现 | `clock_gettime` 返回固定值，`nanosleep` 等不实际睡眠；无定时器中断，无法提供时间片或真实时间服务。 | 完全未实现时间子系统，所有时间相关系统调用均为虚假实现，对用户程序造成误导。 |
| **系统信息** | 部分实现 | `uname` 正确返回内核名称、版本和架构；`prlimit64` 仅填充无上限限制值；`getpid`/`gettid` 正常，但 `getppid` 返回 0。 | 基本系统信息查询可满足，但 getppid、getuid 等返回虚假值，未与进程模型对齐。 |
| **中断管理** | 部分实现 | 仅处理了来自用户态的同步异常（ecall、断点、缺页）；无外部中断或定时器中断处理。 | 中断处理仅服务于系统调用，无法响应设备或计时事件，限制明显。 |
| **系统调用覆盖** | 部分实现，约 55% | 实现了 50+ 个系统调用，但近半数为桩；信号、时间相关调用完全无功能；`mmap` 忽略 `prot`/`flags`。 | 常见文件 I/O 和进程控制调用可用，但缺失的关键子系统导致复杂用户程序兼容性差。 |

## 7. 总结评价

weijun-eos-kernel 是一个结构清晰的 RISC-V 单体内核实验项目，在约 4000 行 Rust 代码内实现了包括伙伴分配器、Sv39 内存管理、完整进程生命周期、ext4 文件系统操作和管道 IPC 在内的核心功能。其汇编陷阱入口和上下文切换展现了开发者对 RISC-V 特权架构的良好理解；嵌入引导 ELF 的设计使内核部署较为便捷。

然而，该项目存在若干明显短板：协作式调度无法保证公平性；完全缺失按需分页和写时复制等现代内核基本技术；时间与信号子系统全部为桩实现，严重限制了用户态软件生态的兼容性；泛滥的 `static mut` 全局状态使得内核无法向多核演进，且由于 Rust 新版 lint 产生大量警告。此外，伙伴分配器的健壮性问题和部分系统调用参数被忽略等细节缺陷，说明测试与边界情况处理不足。

总体而言，该内核在单核、静态工作负载下具备了基本的用户程序托管能力，适合作为操作系统原理的展示作品；但若以通过 OS 大赛完整评测或承载通用用户程序为目标，仍需在内存管理、调度机制、中断处理以及系统调用完整性方面进行大量增强。
