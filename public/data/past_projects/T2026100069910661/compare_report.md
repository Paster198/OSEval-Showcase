## 七、各项目总结评价

### OSKernel2025-X（本项目）

本项目当前处于所有对比项目中最低的完成度水平，仅为 RISC-V S-mode 引导框架。代码量约 197 行（有效功能约 60 行），实现了内核栈设置、SBI 字符输出和内核主入口的占位框架。注释代码显示其设计参考了 MOS（MIPS Operating System）教学操作系统，规划了内存管理、进程管理（带优先级调度）、管道 IPC 和微内核风格的文件系统服务进程架构，但所有规划均未落地为实际代码。与其他项目动辄数千至数万行的代码规模相比，存在数个数量级的差距。唯一的优势在于代码简洁、引导路径正确，可作为进一步开发的起点。

### 宁波工程学院-xv6-riscv

该项目是对 xv6-riscv 向 QEMU virt 平台的标准移植，技术路线清晰且工程规范。21 个系统调用、Sv39 虚拟内存、SMP 多核支持、自旋锁/睡眠锁、管道 IPC 等核心机制完整，基于 SBI HSM 扩展的多核启动是值得参考的技术决策。当前主要缺失 FAT32 文件系统支持和 mmap 等高级系统调用。作为 xv6 生态的正统移植，与 OSKernel2025-X 同属"从教学内核出发"的技术路线，但其开发深度远超本项目。

### 长春理工大学-SystemNQB

该项目在架构设计上展现了最大的野心：C 与 Rust 混合编程、伙伴系统分配器、ext4 文件系统、Linux 兼容系统调用、线程组支持。自动化构建系统（sed 生成系统调用表、bindgen 生成 FFI）体现了优秀的工程素养。然而，项目因 ext4_rs 第三方库集成失败（超级块解析除零 panic）导致内核完全无法启动，属于高开低走的典型案例。这警示了依赖第三方复杂库（特别是 Rust ext4_rs 这类不够成熟的库）的集成风险。与 OSKernel2025-X 对比，SystemNQB 证明了快速堆叠复杂功能的策略存在极高的技术风险。

### 合肥工业大学-xv6-sifive

该项目是六个项目中唯一能够在 QEMU 中成功运行并执行数十个 busybox 命令的项目，功能覆盖面最广（约 70 个系统调用、ext4 文件系统、信号机制、线程支持）。硬件驱动较为完整（16550 UART、VirtIO MMIO、PLIC）。但其工程质量存在严重问题：进程资源泄漏、ext4 位图地址计算未定义行为、信号掩码逻辑错误、测试数据伪造（iozone 硬编码）。从竞赛视角看，该项目在"能跑"层面得分最高，但在"做对"层面存在明显缺陷。与 OSKernel2025-X 对比，xv6-sifive 展示了从基础内核到可运行系统的全路径，但同时也警示了追求功能广度而牺牲深度和诚信的代价。

### 上海交通大学-ChCore

ChCore 在所有对比项目中设计水平最高、架构最为完善。微内核架构与 Capability 安全模型的组合、迁移式 IPC 的 Shadow 线程优化、策略模式的可插拔调度器、用户态文件系统与网络栈的隔离设计，均体现了研究级操作系统的设计深度。Buddy+Slab 双层分配器、VMSpace 红黑树管理、COW 支持等细节实现扎实。主要不足在于构建环境依赖于特定工具链（Bootlin glibc 工具链），在当前环境中无法完成构建验证。与 OSKernel2025-X 对比，ChCore 代表的是一条完全不同的技术路线（自研微内核），其架构复杂度远超 xv6 生态项目，但相应地也需要更大的团队规模和更长的开发周期。

### 燕山大学-wCore OS

wCore OS 是 Rust 语言在 OS 内核开发中的优秀示范。利用 Rust 的类型安全和 RAII 机制，实现了严谨的物理页帧管理（FrameTracker）、PID 分配（PidHandle）和内核栈管理（KernelStack）。VFS 层的 trait 抽象设计清晰，支持 RamFS 与 FAT32 的多文件系统挂载。主要不足在于文件操作系统调用未暴露给用户态、brk 实现有逻辑缺陷、仅支持单核 FIFO 调度。与 OSKernel2025-X 对比，wCore OS 展示了另一种语言选择（Rust vs C）的技术路线，其类型安全和资源自动管理特性在中期开发阶段能有效减少内存相关 bug。

| 4 | 宁波工程学院-xv6-riscv | C | 移植路径规范、框架完整，但关键功能缺失 |
| 5 | 燕山大学-wCore OS | C | 代码质量好但功能断层，系统调用不完整 |
| 6 | OSKernel2025-X | D | 仅有引导存根，尚无任何子系统实现 |

### 分类评价

**架构创新梯队**：ChCore（微内核+Capability+迁移式IPC）> SystemNQB（C+Rust混合）> wCore OS（Rust全栈+RAII）

**功能广度梯队**：xv6-sifive（70+syscall, ext4, 信号, 线程）> ChCore（多FS, 网络, Capability）> SystemNQB > xv6-riscv > wCore OS > OSKernel2025-X

**工程质量梯队**：wCore OS（Rust类型安全, RAII）> ChCore（清晰分层, 策略模式）> xv6-riscv（xv6规范移植）> xv6-sifive（测试伪造扣分）> SystemNQB（核心崩溃）> OSKernel2025-X（尚无工程）

**可运行性梯队**：xv6-sifive（唯一可运行busybox）> 其余项目均无法在当前环境完整运行或未验证

## 九、评审意见

OSKernel2025-X 当前处于所有对比项目中最低的完成度水平，仅实现了一个能够输出 "Hello, RISC-V!" 的 RISC-V S-mode 引导存根，代码量约 60 行有效功能代码，整体完成度约 3%（以完整教学操作系统为基准）。与选中的五个对比项目存在数个数量级的差距——这些项目均基于成熟的教学内核框架（xv6 或 rCore）进行了数月至数年的深度开发，代码量在 6,000 行至 92,000 行之间，实现了完整的内存管理、进程调度、文件系统和系统调用接口。

从正面来看，OSKernel2025-X 的引导路径正确、SBI 调用封装基本规范、构建系统可完整复现。注释中规划的 MOS 移植架构（内存管理、优先级调度、IPC 管道、微内核风格文件系统服务进程）展现了清晰的设计意图。

然而，从竞赛角度审视，项目目前的工作量极不充分——仅有一个提交，所有核心子系统均未实现，无法支撑任何形式的用户态程序运行。与其他项目的对比清晰地展示了从"引导存根"到"可运行内核"之间的巨大鸿沟：即使是最基础的 xv6-riscv 移植（宁波工程学院），也需要实现 21 个系统调用、Sv39 页表、多核 SMP 和管道 IPC 等核心机制；而要达到可运行用户态工具的水平（合肥工业大学），还需进一步完成 ext4/FAT32 文件系统、信号机制和线程支持。建议项目团队明确当前所处的开发阶段与现实差距，优先完成内存管理和进程管理两大核心子系统，再逐步扩展文件系统和系统调用接口，方能在后续开发中缩小与对比项目的差距。
