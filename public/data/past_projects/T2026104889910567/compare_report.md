| **StarsOS** | 全局单队列FIFO调度（无时间片/无优先级）；文件系统全局粗粒度锁（mtx_file）削弱SMP优势；无页面置换；无网络协议栈；无ext4支持；孤儿进程处理不完整。 | **中高**：FIFO调度和全局锁设计使得SMP多核优势被严重削弱；文件系统虽精但不支持ext4限制了对现代磁盘的兼容性。 |

**分类评价**：

- **第一梯队（综合成熟度>80%）**：**cabbageOS**。在内存管理（Buddy+COW+Demand Paging）、进程管理（PCB/TCB分离+clone语义）、系统调用覆盖率（95%）等核心维度全面领先，架构设计最为现代化，是五个对比项目中成熟度最高的内核。

- **第二梯队（综合成熟度70-80%）**：**StarsOS**。虚拟内存创新（被动映射）和文件系统深度（FAT32+LFN+Socket）突出，但调度器和全局锁设计限制了整体表现，且不支持 ext4 是功能性短板。

- **第四梯队（综合成熟度<5%）**：**OSKernel2025-X**。与上述五个项目存在数个数量级的差距，不具备可比性。

## 七、各项目总结评价

### OSakura（武汉大学）

OSakura 是一个结构清晰、代码规范的教学/竞赛级宏内核。其核心优势在于自研 ext4 文件系统（含 Extent 树支持）和 ELF 动态链接程序的加载能力，这两项特性在同级别竞赛内核中具有较高的实现难度。函数指针表（FS_OP_t）实现的多文件系统抽象层展现了良好的软件工程思维。60个系统调用覆盖了进程、文件、信号等核心领域。主要短板包括：调度器为简单遍历轮转且禁用了抢占、fork 采用全量复制缺乏 COW 优化、ext4 无日志机制、仅支持单核。整体而言，OSakura 是一个"小而美"的内核作品，在关键子系统上自主钻研深入，但在性能和高级特性上仍有较大提升空间。

### TOYOS（华东师范大学）

TOYOS 与 OSakura 技术路线高度相似——均为自研 ext4（含 Extent 树）、支持 ELF 动态链接、采用 Trampoline 机制。VFS 三层架构设计清晰，锁机制包含死锁检测和错误释放检测，鲁棒性较好。然而，TOYOS 存在一个致命短板：信号机制的核心分发函数（sig_handle）未实现，导致定义了64个信号却无法实际触发，这使得信号子系统名存实亡。此外，调度器同样是遍历轮转且抢占被注释，内存管理无 COW。TOYOS 的设计框架良好但实现深度参差不齐，需要在信号处理和调度策略上进行集中补强。

### cabbageOS（华中科技大学）

cabbageOS 是五个对比项目中综合成熟度最高的内核。PCB/TCB 分离的进程/线程模型、基于 per-CPU Buddy 池的多核物理内存管理、COW 与按需分页、VFS 集成 FAT32 和 ext4 双文件系统——这些设计在架构层面展现了对现代操作系统内核的深刻理解。Futex 和 System V 共享内存的支持进一步扩展了用户态并发编程能力。Rust 驱动的引入是安全性和现代语言实践的有益尝试。主要不足在于进程组/会话管理的缺失影响 Shell 作业控制、ext4 依赖外部库限制深度优化。cabbageOS 是五个项目中唯一在多核、COW、多线程三个维度均达到较完整实现水平的内核。

### ASVOS（武汉科技大学）

ASVOS 的核心策略是"拿来主义"——通过移植 lwext4（约18,000行）快速获得包含 JBD 日志、Extent 树和 htree 目录索引的企业级 ext4 能力，同时支持 QEMU 和 VisionFive2 双平台。78个系统调用入口提供了较宽的功能覆盖面。然而，该项目在当前 GCC 13.2.0 环境下构建失败（类型定义冲突），这从根本上限制了对其功能优势的验证。此外，38%的系统调用为桩实现、FAT32 未集成 VFS、信号投递不完整、自主编写代码比例偏低（约23%），反映出移植优先于自主创新的策略在深度评估中暴露出短板。

### StarsOS（中国科学技术大学）

StarsOS 在设计创新上最为突出。"被动映射"机制按需分配物理页、initcall 自动注册机制借鉴 Linux 设计、O(1)复杂度定时睡眠事件队列、本地 Unix Domain Socket（TCP/UDP）——这些设计在竞赛内核中具有独创性。FAT32 实现支持 LFN 长文件名和硬链接，细节丰富。然而，全局单队列 FIFO 调度器和文件系统全局粗粒度锁（mtx_file）严重削弱了 SMP 多核架构的理论优势，使得并发性能大打折扣。不支持 ext4 文件系统也限制了对典型 Linux 磁盘的兼容性。StarsOS 是一个在局部子系统深度探索出色但系统级整合尚有矛盾的内核作品。

### OSKernel2025-X（本项目）

OSKernel2025-X 当前处于所有对比项目中最低的完成度水平。项目仅包含 197 行代码（有效功能代码约 60 行），实现了 OpenSBI 引导后输出 "Hello, RISC-V!" 并进入死循环的最简路径。代码注释中保留了源自 MOS 教学操作系统的架构蓝图（内存管理、进程管理、管道 IPC、文件系统服务进程），但均未实现。构建系统规范，QEMU 运行路径验证正确。与五个对比项目（代码规模从 9,633 到 38,254 行不等，均实现了完整的内核核心子系统）相比，OSKernel2025-X 在开发阶段上存在根本性的代际差距。当前项目可视为一个引导验证存根，而非操作系统内核。

## 八、评审意见

本报告对 OSKernel2025-X（本项目）与来自武汉大学、华东师范大学、华中科技大学、武汉科技大学、中国科学技术大学的五个竞赛级操作系统内核进行了系统的多维度对比分析。

**关于本项目**：OSKernel2025-X 当前处于极早期开发阶段，仅完成了 RISC-V S-mode 最小引导路径和 SBI 字符输出的验证。五大核心子系统（内存管理、进程管理、文件系统、设备驱动、系统调用）均未实现任何一个，整体完成度不足 2%。在所有对比项目中，OSKernel2025-X 与其他五个项目之间存在不可跨越的实现差距——这五个项目均实现了完整的内核核心链路（启动→内存管理→进程调度→文件系统→系统调用→用户态），代码规模从约 9,600 行到约 38,000 行不等。本项目若要以这些项目为参考目标，需要至少实现以下最低功能集：物理页分配器 + SV39 页表、进程控制块 + 基础调度器、至少一种文件系统（建议从 FAT32 或 Ramfs 起步）、中断/异常处理框架、以及 20 个以上的基础系统调用。

**关于对比项目**：五个对比项目代表了当前国内高校操作系统竞赛中 C 语言宏内核路线的不同技术策略。cabbageOS 在架构设计全面性和子系统实现深度上表现最优，是综合成熟度最高的参考标杆；StarsOS 在局部设计创新（被动映射、initcall、Socket）上最为突出但全局调度和锁设计存在结构性矛盾；OSakura 和 TOYOS 在自研 ext4 + ELF 动态链接的技术路线上展现了扎实的底层开发能力，是自主钻研深度的优秀代表；ASVOS 的外部库移植策略在文件系统功能完整度上有理论优势，但工程健壮性和自主创新比例是明显短板。

**分类排名**（基于综合成熟度）：

1. **综合最优**：cabbageOS（华中科技大学）—— 多核 + COW + 多线程 + 双文件系统，架构最完整。
2. **设计创新**：StarsOS（中国科学技术大学）—— 被动映射 + initcall + Socket + O(1)定时睡眠。
3. **自主深度**：OSakura（武汉大学）/ TOYOS（华东师范大学）—— 自研 ext4 Extent 树 + ELF 动态链接。
4. **功能移植**：ASVOS（武汉科技大学）—— lwext4 完整移植但工程健壮性和自主性不足。
5. **极早期阶段**：OSKernel2025-X（本项目）—— 仅有引导存根，各子系统均待从零构建。

综上，OSKernel2025-X 作为一个操作系统内核项目，目前完成的工作量仅相当于五个对比项目约 0.5%-2% 的代码规模和功能覆盖。建议在后续开发中优先补齐异常处理框架、物理内存管理和基础进程调度这三个最底层模块，在此基础上逐步向文件系统和系统调用层扩展。五个对比项目各自的技术路线和设计权衡可作为不同阶段的参考——从 OSakura/TOYOS 的自研精神、StarsOS 的设计创新，到 cabbageOS 的架构完整性，构成了一个多层次的技术进阶图谱。
