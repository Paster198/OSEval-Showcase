# OSKernel2026-X 技术画像与评估报告

## 1. 项目基本信息

| 属性 | 内容 |
|---|---|
| 项目名称 | OSKernel2026-X（基于 RT-Thread 5.x） |
| 目标架构 | RISC-V64（主要）、LoongArch64（初赛 stub） |
| 实现语言 | C（主要）、汇编、Rust（少量组件，未使用） |
| 内核基线 | RT-Thread 5.x（版本号 0x50200） |
| 生态归属 | RT-Thread 生态，竞赛扩展 |
| 构建系统 | SCons + Make，产物为 kernel-rv（RISC-V64 ELF）和 kernel-la（LoongArch64 ELF） |
| 竞赛自研代码位置 | applications/（约 9,000 行） |
| 运行平台 | QEMU RISC-V64 virt 机器（LoongArch64 为独立 stub） |
| 主要特征 | 基于 RT-Thread 内核，在其上构建用户态进程执行环境（“OS on OS”架构），实现 EXT4 只读扫描、ELF 加载、约 80+ Linux 兼容 syscall 用户态模拟，支持双 ABI 测试框架。 |

## 2. 实现的子系统与功能

### 2.1 内核核心（来自 RT-Thread 基线）
- **线程管理**：线程创建/初始化、启动、删除/分离、挂起/恢复、延时、主动让出、控制接口，完整的线程状态机（INIT→READY→RUNNING→SUSPEND/CLOSE），互斥锁嵌套持有链表与退出自动释放。
- **调度器（UP）**：32 优先级位图加速的抢占式调度，同优先级时间片轮转，调度器锁与临界区保护。
- **IPC 机制**：计数信号量（FIFO/PRIO 等待）、互斥锁（优先级继承与天花板协议）、事件标志（AND/OR 触发与可选清除）、邮箱（固定大小消息，支持紧急消息）、消息队列（可变长度消息，支持紧急消息）。
- **内存管理**：SLAB 分配器（作为堆后端，配置 `RT_USING_SLAB_AS_HEAP`）、memheap 多堆管理、内存池固定块分配，支持内存追踪。
- **定时器与时钟**：软件定时器（单次/周期）、系统节拍（100 Hz）、硬件定时器（RISC-V mtime/mtimecmp 通过 SBI 设置）。
- **信号处理**：POSIX 风格线程间信号（安装、屏蔽、等待、发送、默认终止处理）。
- **对象管理**：基于 `rt_object` 的内核对象容器，按类型链表管理，支持按名称查找。
- **组件自动初始化**：基于链接器 section 的分级自动初始化框架。

### 2.2 架构层（RISC-V64）
- **Trap 处理**：S-Mode 完整分发（外部中断、定时器中断、用户态 ecall/异常），通过弱引用钩子 `os_user_ecall_handle` / `os_user_fault_handle` 与竞赛代码桥接。
- **上下文切换**：汇编实现的 callee-saved 寄存器保存/恢复，支持 `rt_hw_context_switch`。
- **MMU（SV39）**：三级页表管理，支持 ASID、内核重映射、TLB 刷新，用于竞赛用户态页表。
- **PLIC 驱动**：中断优先级设置、使能/禁用、claim/complete 流程。
- **SBI 封装**：OpenSBI 0.2+ 兼容层，自动探测扩展与 SBI 实现，封装定时器、IPI、RFENCE、HSM 等调用。

### 2.3 BSP 与驱动
- **NS16550 UART**：基于 MMIO（0x10000000），波特率 115200 8N1，挂接 RT-Thread serial 设备框架。
- **VirtIO 驱动**：MMIO 传输层，自动枚举 blk/net/console/gpu/input 设备并调用对应 init handler。
- **板级初始化**：堆初始化、PLIC 初始化、中断初始化、UART 初始化、时钟节拍配置。

### 2.4 竞赛应用层（核心自研）
- **EXT4 只读扫描器**：直接读取 virtio-blk0 设备，解析 superblock、组描述符、inode、extent tree 与 legacy 间接块，支持目录遍历、路径解析、文件缓存（最多 96 个缓存文件）。
- **ELF 加载器**：支持 ELFCLASS64、小端、ET_EXEC/ET_DYN、EM_RISCV/EM_LOONGARCH、PT_LOAD/PT_DYNAMIC/PT_INTERP，最多 8 个 LOAD 段。
- **用户镜像管理器**：将加载的 ELF segment 转换为页对齐映射，处理 BSS 清零，提供 entry、aux 向量、phdr 信息。
- **用户态执行与 syscall 处理**：
  - 完整 SV39 用户页表构造（按需分配三级表，支持大页恒等映射）。
  - 进入/退出用户态流程（保存/恢复 runner 上下文，使用 sret）。
  - 约 80+ syscall 实现（详见下文），包括文件 I/O、文件管理、内存管理（mmap/brk）、进程管理（clone/execve/exit_group/wait4）、时间、系统信息、资源限制、信号（骨架）、socket（模拟）、管道、伪文件系统等。
  - syscall 预算机制（20000 次限制）。
  - 简化 clone（页面快照，单级子进程）、execve（通过返回状态码协作式重新加载）。
  - 内建 in-memory 文件系统（支持 128 个文件，目录、读写、伪文件如 /proc/...）。
- **测试运行器**：支持 smoke、basic、busybox、libctest、lua、直接 ELF 执行等多种模式，脚本解析（跳过包装器、提取参数、重定向检测、管道支持），ABI 探测顺序 glibc→musl，限制每 ABI 测试用例数。

### 2.5 LoongArch64 最小入口（独立 stub）
- 完全自包含的 freestanding 实现（不依赖 RT-Thread），含 UART 驱动、PCI 扫描、VirtIO MMIO 操作、EXT4 解析、ELF 加载、60+ syscall 处理、测试执行，仍在开发中。

### 2.6 被启用但未使用的 RT-Thread 组件
- DFS v2（含 FatFs、DevFS、RomFS）、lwIP 网络栈、Finsh Shell、POSIX 兼容层等，竞赛代码绕过这些组件，自行实现所需功能。

## 3. 各子系统实现细节与优缺点

### 3.1 线程管理
- **实现细节**：基于 `rt_thread` 结构体，栈空间可由静态或动态分配，线程挂起时可能持有互斥锁记录在 `taken_object_list`，退出时自动释放，最终进入 defunct 列表由 idle 线程回收。
- **优点**：实现完整，具备可靠的反向互斥锁清理机制；API 丰富，满足实时系统常见需求。
- **缺点**：无多核支持（UP 编译），无线程亲和性等高级特性。

### 3.2 调度器
- **实现细节**：优先级位图 `rt_thread_ready_priority_group` 使用 `__rt_ffs()` 快速定位最高就绪优先级，每个优先级维护一个就绪链表，同优先级内按 FIFO 轮转（通过 `rotor` 指针）。调度器锁为原子变量，支持嵌套。
- **优点**：确定性高，响应快速；O(1) 调度决策。
- **缺点**：仅 32 个优先级，无抢占阈值或 deadline 调度；未同 RT-Thread SMART 模式集成。

### 3.3 IPC
- **实现细节**：所有 IPC 对象继承自 `rt_ipc_object`，等待线程按优先级或 FIFO 挂起，支持带超时等待。互斥锁实现完整优先级继承，记录持有者原始优先级。
- **优点**：种类丰富，正确性高；优先级继承可缓解优先级反转。
- **缺点**：消息队列的紧急消息可能破坏 FIFO 顺序，实现复杂但功能边界清晰。

### 3.4 内存管理（内核堆）
- **实现细节**：当前启用 SLAB 作为堆后端，在 `board.c` 中通过 `rt_system_heap_init(RT_HW_HEAP_BEGIN, RT_HW_HEAP_END)` 将一段连续物理内存交给分配器。memheap 可管理多段不连续内存，但此时未被激活（仅 SLAB）。
- **优点**：SLAB 分配器对固定大小对象分配极快，内部碎片少。
- **缺点**：无虚拟内存支持用于内核本身（内核直操物理地址），无 page cache 等。

### 3.5 定时器与时间管理
- **实现细节**：系统节拍由 RISC-V mtime 比较器通过 SBI 产生 100Hz 中断。软件定时器以节拍为单位，挂载在定时器列表中，由单独线程（优先级 4）处理超时。`clock.c` 提供节拍与毫秒转换、系统运行时间等。
- **优点**：基本功能完整，满足周期性任务需求。
- **缺点**：定时器精度受 10ms 节拍限制，未使用高分辨率定时器。

### 3.6 信号处理
- **实现细节**：每个线程有 `rt_sigset_t` 挂起信号集和屏蔽集，`rt_thread_kill` 通过 `_signal_deliver` 尝试在适当时候（如系统调用返回）调用信号处理函数。仅支持少数信号，大多数默认终止线程。
- **优点**：提供了线程间信号通知的基本框架。
- **缺点**：竞赛代码未完整利用，实际信号不传递给用户程序，仅维护屏蔽位，限制了依赖信号的用户程序。

### 3.7 用户态执行（竞赛自主实现）
- **实现细节（SV39 页表）**：为每个用户镜像动态构造三级页表，使用 `alloc_aligned_page` 分配 4KB 对齐页，`ensure_next_table` 按需创建中间页表。映射时遵循段权限。进入用户态前切换页表，退出时切回裸机。
- **优点**：完整实现了 SV39 地址翻译，可运行标准 RISC-V 用户态程序。
- **缺点**：无 ASID 管理（每次切换需全量 TLB 刷新），无 Copy-on-Write，内存开销大；恒等映射内核代码段为 1GB 大页，未细粒度保护。
- **实现细节（syscall 层）**：通过 `a7` 分发 80+ 系统调用，涵盖文件、内存、进程、时间、系统信息等。文件 syscall 基于 in-memory 文件系统和 EXT4 只读后端。内存 syscall（mmap/brk）在用户地址空间内分配物理页并建立映射，无 swap。进程管理 syscall 使用页面快照模拟 fork，协作式 execve。管道实现为环形缓冲区（2048 字节）。
- **优点**：覆盖面广，能运行 busybox、libc-test 等复杂用户负载；syscall 预算机制增强了资源控制。
- **缺点**：fork 语义不完整（页面快照而非 CoW，不支持嵌套 fork），execve 递归深度限制为 4，可能失败于深层调用链；网络 syscall 为极简模拟，无法真正通信；信号相关 syscall 大多返回成功但不递送信号。

### 3.8 EXT4 扫描器
- **实现细节**：从 `rt_device_read` 以 512 字节扇区读取，逐结构解析。支持 extent 树递归遍历和 legacy 间接块，可处理多级路径查找。文件数据按块缓存，避免重复 I/O。
- **优点**：摆脱了对 VFS 和块缓存的依赖，代码紧凑，能在内核未挂载文件系统的情况下读取文件。
- **缺点**：仅支持只读，不支持写回；未实现哈希树目录索引，大目录查找效率低；extent 深层树未能进行完整性校验。

### 3.9 ELF 加载器
- **实现细节**：解析 ELF header 和 program header table，为每个 PT_LOAD 段分配内存并读取文件内容，BSS 段填充零。返回段映射信息供页表构建。
- **优点**：支持动态链接可执行文件和解释器路径，可加载 glibc/musl 测试程序。
- **缺点**：不支持重定位处理，动态链接器依赖用户态 ld.so；未处理 PT_TLS、PT_GNU_STACK 等高级段。

### 3.10 测试运行器
- **实现细节**：根据脚本内容确定运行模式，跳过 `env`、`timeout` 等包装器，提取真实命令和参数；对于 busybox 管道，创建管道 fd 对并串联执行。通过 libctest 列表文件批量执行测试用例。
- **优点**：可驱动多种评测任务，具有较好的脚本兼容性。
- **缺点**：复杂 shell 构造（如 `$(...)`、`&&`、`||`）会被跳过，可能遗漏测试用例；重定向实现有限。

## 4. OS内核整体实现完整度

评估基准：以支持 RISC-V64 用户态 Linux 兼容可执行文件运行为核心目标，考察自研部分的功能覆盖度与健壮性。RT-Thread 基线功能不计入缺失。

- **用户态执行环境核心**：80%。SV39 页表、syscall 分发、内存映射、文件系统后端等关键路径已齐备。缺失部分主要包括 CoW、完整信号递送、真正的多级 fork、execve 深度限制等。
- **syscall 覆盖度**：75%。核心文件/进程/内存/时间 syscall 完整，能执行 busybox 和 libc-test 基础用例。网络、高级信号、资源控制等部分为 stub。
- **文件系统支持**：70%。EXT4 只读扫描可满足读需求，in-memory 文件系统支持读写但无持久化，无法在重启后保留数据。
- **进程模型完整性**：50%。clone/fork 语义简化，execve 采用协作式重载，缺乏完整 task_struct 式进程描述符。
- **整体内核实现完整度**（以竞赛目标衡量）：约 70%，可运行大部分评测负载，但在极端语义场景下可能受限。

## 5. 动态测试设计与结果

### 5.1 动态测试设计
项目未提供独立的单元测试或内核自测程序。动态验证依赖 EXT4 磁盘镜像中的测试脚本，由测试运行器（`os_test_runner.c`）解析并执行。主要测试模式：
- **smoke**：运行 `oskernel_user_smoke.sh`，验证基本启动和用户态进入。
- **basic**：运行 `basic_testcode.sh`，对每个 ABI（glibc、musl）执行基础系统调用测试，限制 48 个用例。
- **busybox**：执行 `busybox_testcode.sh`，使用 busybox 命令进行功能验证，支持简单管道。
- **libctest**：解析 libc-test 用例列表，逐用例加载执行，限制每 ABI 128 个测试用例。
- **lua**：执行 Lua 测试脚本。
- **直接执行**：直接加载 libcbench、iozone 等基准工具。

### 5.2 测试结果
由于环境限制，本次评估未在 QEMU 中动态运行。根据源代码审查，测试运行器设计合理，脚本解析逻辑可覆盖大部分常见调用模式，但无实际通过率数据。报告中无动态测试结果可供分析。

## 6. 细则评价表格

以下表格基于静态代码审查，对每个条目给出实现与否、关键发现和评价。

| 条目 | 是否实现及完整度 | 关键发现 | 评价 |
|---|---|---|---|
| 内存管理（内核） | 是，90% | 使用 SLAB 作为堆分配器，支持 memheap、mempool；通过边界地址初始化堆，含内存追踪。未使用 MMU 保护内核自身。 | 成熟稳定，满足内核运行时需求。 |
| 内存管理（用户态） | 是，80% | 实现 SV39 三级页表，支持 mmap/brk 构建用户虚拟地址空间，按需分配物理页。无 CoW、无 swap、无 ASID 优化。 | 核心映射功能完善，可正确加载多段 ELF；性能和资源节省方面有提升空间。 |
| 进程管理 | 部分，50% | 实现了 clone/fork（页面快照，单级）、execve（协作式重载）、exit/wait4；没有完整的进程控制块和优先级继承，缺乏独立进程映像管理。 | 虽能执行用户程序并创建子进程，但简化模型限制了复杂多进程应用，可靠性在深层嵌套或频繁 fork 场景下存疑。 |
| 文件系统 | 部分，70% | 绕过 RT-Thread DFS，直接从 virtio-blk 读取 EXT4 结构，支持超级块、inode、extent/间接块、目录遍历、路径解析。in-memory 文件系统提供读写和伪文件。无写回、无大目录优化、无持久化。 | 读取路径实现精巧，满足竞赛评测的读取需求。写支持和持久化缺失，对需要修改文件系统的测试不利。 |
| 交互设计 | 部分 | 内核通过串口控制台可交互（Finsh shell 启用但未在测试中使用），用户态程序通过标准 fd 与 console 交互，管道支持基础。无图形、无网络终端。 | 满足竞赛测评的基本输入输出需求。用户交互能力限于文本流。 |
| 同步原语 | 是，95% | 内核提供信号量、互斥锁（含优先级继承）、事件、邮箱、消息队列；用户态 syscall 不映射这些 IPC（用户态依赖 Linux syscall 语义，未提供 futex）。 | 内核级同步原语丰富且正确，但用户态多线程同步未提供原生 futex。 |
| 资源管理 | 部分，70% | 内核对象容器管理线程、IPC 等资源；文件描述符表管理文件、管道、socket；syscall 资源限制通过 syscall 预算实现；用户内存通过分配释放管理。无 cgroup 或 quota 机制。 | 基本资源追踪与回收有效，syscall 预算是一种简单而有效的防耗尽手段。 |
| 时间管理 | 是，85% | 系统节拍 100 Hz，软件定时器，虚拟时间 syscall（clock_gettime/nanosleep 等）基于自维护的虚拟时钟。无高精度定时器或 tickless 模式。 | 用户态时间相关 syscall 实现充分，可运行 sleep、time 等命令；虚拟时钟与 wall-clock 可能不同步。 |
| 系统信息 | 是，85% | 实现 uname、sysinfo、getpid/ppid、资源限制查询等 syscall，并提供 /proc 伪文件（mounts、meminfo、version、self/maps 等）。 | 用户态信息获取途径较多，可满足大多数系统信息查询需求。 |
| 构建与可移植性 | 是，75% | SCons 构建 RISC-V64，Makefile 封装；LoongArch64 stub 独立编译。代码依赖清晰的架构层和 BSP 层。 | 构建流程简洁，便于竞赛评测。LoongArch64 端口尚未与主核融合，可视为初步。 |

## 7. 总结评价

OSKernel2026-X 采用了“在成熟 RT-Thread 实时内核之上构建用户态兼容层”的竞赛策略，避免了从零开发底层硬件抽象和调度器等繁重工作，将主要精力集中在用户态执行环境的实现上。项目自研的 EXT4 只读扫描器、ELF 加载器、SV39 页表管理器以及约 80+ Linux 兼容系统调用共同构成了一个可运行标准用户程序的轻量级“OS on OS”环境，展现了清晰的分层设计和务实的工程实现。

项目的突出优点包括：syscall 覆盖面较广，能驱动 busybox 和 libc-test 等真实用户负载；利用弱引用钩子将自研用户态处理无缝集成进内核 trap 路径，无需修改 RT-Thread 核心代码；设计 syscall 预算机制增强鲁棒性；支持双 ABI 测试框架，评测适应性强。

主要不足体现在进程模型的简化（clone 快照、协作式 execve）和高级功能的缺失（真正的信号递送、网络栈、文件持久化等），这些可能在复杂评测场景中导致部分测试失败。此外，大量 RT-Thread 组件被配置却未发挥作用，说明竞赛方案选择性地绕过而非利用已有基础设施，虽减低了集成难度，但也放弃了复用成熟子系统的机会。

整体而言，OSKernel2026-X 在竞赛约束下是一个设计合理、实现扎实的作品，其核心技术创新在于“OS on OS”双层架构和轻量级兼容层实现，具有明确的工程价值和实践意义。
