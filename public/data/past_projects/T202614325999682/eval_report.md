# Sustcore OS 内核技术画像与评估报告

## 一、项目基本信息

- **项目名称**：Sustcore
- **架构**：RISC-V64（Sv39）、LoongArch64
- **实现语言**：C++（GNU++23）、C（GNU18）、汇编
- **生态归属**：独立内核，能力型混合内核（Capability-based Hybrid Kernel），不基于 Linux/BSD 等现有内核
- **代码规模**：约 513 个源码文件，合计约 114,009 行
- **构建系统**：GNU Make + 自定义构建框架，依赖 GCC 15+
- **许可证**：未在分析范围内明确，但假设为开源项目
- **核心特点**：
  - 以 Capability 作为所有内核资源的唯一访问句柄
  - 两级索引的能力空间（CSpace），支持细粒度权限与 CLONE/MIGRATE 传递语义
  - 完整的 Buddy→SLUB→GFP 内存分配链，搭配 VMA 管理和写时复制（COW）
  - 从零实现的 ext4 文件系统读写支持，辅以 tmpfs、procfs、tarfs、devfs
  - 多级调度类（RT/INIT/RR/FCFS/IDLE）和可扩展调度框架
  - 双架构对称实现，通过 C++20 Concepts 约束架构接口
  - 异步 IPC 模型（Endpoint）与 RPC 模式
  - 丰富的用户态模块（30 个）用于功能验证

## 三、各子系统实现完整度与优缺点

所有完整度评估均以 **“可运行多用户进程并具备基础文件/进程/内存管理能力的操作系统内核”** 为统一基准，基于源码静态审查确定。

### 3.1 能力系统 — 完整度：90%
**优点**：
- 两级索引的 CSpace 设计清晰，每个进程最多 1,048,576 个能力槽位。
- 15 种 Payload 类型覆盖所有主要内核对象。
- 细粒度权限模型（64 位位图），支持能力派生、降权、传递，语义正确实现 CLONE/MIGRATE/MIGRATE_ONCE。
- Payload 生命期通过引用计数管理，构造/析构安全。

**缺点**：
- 当前未发现明显的功能性缺失；可在能力回收、跨进程能力审计等方面进一步增强。

### 3.2 内存管理 — 完整度：90%
**优点**：
- Buddy 分配器支持 order 0~15，分裂与合并逻辑完整，伙伴节点查找高效。
- SLUB 分配器按对象大小自适应（small/large），Slab 状态管理（EMPTY/PARTIAL/FULL）和统计信息完善。
- GFP 在 Buddy 之上叠加引用计数，为 COW 提供基础。
- VMA 支持五种类型和三种增长方式，与 Memory Payload 配合实现懒分配与文件后端映射。
- COW 完整覆盖：fork 时拆分享页、缺页异常（`on_np`）分配新页、写保护异常（`on_wp`）复制物理页。
- 双架构页表操作实现完整，Sv39 支持 4K/2M/1G 页面，LA64 页表管理同样详尽。

**缺点**：
- 未发现 per-CPU 页缓存或快速路径优化。
- 缺少更激进的内存回收策略（如页交换）。

### 3.3 进程与线程管理 — 完整度：85%
**优点**：
- PCB 与 TCB 分离，支持进程父子关系和线程组。
- 线程状态机覆盖 8 种状态，生命周期管理合理。
- 信号系统支持 64 个信号，可设置信号处理动作和掩码，兼容 SIGKILL、SIGSEGV 等关键信号。
- 等待/唤醒子系统通过 WaitReasonManager、WaitContext、Promise/Future 提供统一的同步原语，支持超时和信号中断。

**缺点**：
- 缺少进程组、会话、作业控制等高级 POSIX 特性。
- 线程 join 等同步语义未明确实现。

### 3.4 调度器 — 完整度：85%
**优点**：
- 多级调度类设计，优先级从 RT（实时）到 IDLE 覆盖不同场景。
- RR 类实现时间片（5 ticks），调度核心 `schedule()` 逻辑完备，包含 NEED_RESCHED 和抢占禁用检查。
- 上下文切换流程通过 `prepare_switch()` 切换地址空间和架构相关状态，浮点/向量上下文惰性保存。
- 调度类接口统一（enqueue/dequeue/pick_next/put_prev/check_preempt_curr/on_tick），易于扩展。

**缺点**：
- 仅支持单核，无负载均衡或 CPU affinity。
- 无优先级继承或死锁检测机制。

### 3.5 虚拟文件系统 — 完整度：80%
**优点**：
- 清晰的分层接口（IFsDriver → ISuperblock → IINode → IFile/IDirectory/ISymlink）。
- ext4 实现从超级块解析到 extent 树遍历、目录项解析、inode 分配释放，涵盖读写与符号链接等操作。
- 页缓存采用类 Linux 的双链表 LRU（inactive/active），支持脏页回写和统计，最大 1024 页。
- 额外提供了 tmpfs（内存文件系统）、procfs（进程信息）、tarfs（initramfs）、devfs（设备文件），覆盖多种场景。
- 路径解析完善，支持相对/绝对路径、符号链接跟随、挂载点切换和 `.`/`..` 处理。

**缺点**：
- ext4 日志（journal）不支持，无法保证崩溃一致性。
- 块位图与 inode 位图管理细节未全覆盖；部分 extent 深度大于 1 的复杂场景需进一步验证。
- 页缓存淘汰策略较简单，无主动预读。

### 3.6 块 IO 层 — 完整度：70%
**优点**：
- 抽象出 IBlockDeviceOps、BlockRequestQueue 和 BufferCache。
- BlkManager 为每个块设备分配独立缓冲区和内核工作线程。
- BufferCache 支持基本的读/写/同步和状态标记（CLEAN/DIRTY/LOCKED）。

**缺点**：
- 请求队列缺少 IO 调度（如电梯算法），不进行请求合并或重排。
- 无异步预读机制。
- 对存储错误的处理较简单。

### 3.7 IPC 与同步原语 — 完整度：85%
**优点**：
- Endpoint 提供完整的消息传递：同步/异步 send/recv，以及 call/reply 构成的 RPC 模式，ReplyPayload 作为一次性能力。
- Notification 支持 24 个独立信号位，每信号位可挂接等待者，原子修改。
- Pipe 实现经典的环形缓冲区（默认 4096 字节），读写端分离为独立能力。
- 等待机制与调度器深度集成，满足多种同步需求。

**缺点**：
- 不提供消息优先级或顺序保证（如 FIFO 仅靠链表顺序）。
- 缺少跨进程的共享内存抽象（仅通过 Memory Payload 传递能力间接实现）。

### 3.8 系统调用接口 — 完整度：80%
**优点**：
- 约 80 个系统调用，覆盖能力操作、进程/线程管理、内存、文件 IO、IPC、信号、时间、挂载、管道、procfs 等主要领域。
- 每个调用均经过能力查找和权限校验，保障安全。
- UAccess 设计对用户态内存提供边界检查，避免内核越界访问。

**缺点**：
- POSIX 兼容性有限，缺少 poll/select、mmap/munmap 的变体、更完整的信号集等。
- 部分系统调用标记为“不稳定”，表明接口仍在演化。

### 3.9 设备驱动 — 完整度：65%
**优点**：
- VirtIO MMIO/PCI 驱动框架完整，支持块设备。
- 具备串口、时钟、RTC、系统关机等基础驱动。
- 中断控制器抽象可适配 PLIC/CLINT。

**缺点**：
- 无网络驱动、图形驱动、USB 驱动。
- 整体驱动数量有限，仅支撑基本运行。

### 3.10 架构抽象层 — 完整度：85%
**优点**：
- RISC-V64 和 LoongArch64 异常/中断处理、页表操作、上下文切换均完整实现。
- 通过 C++ Concepts 定义架构 Traits，强制接口一致性，降低移植风险。
- 上下文结构、页表位定义、系统寄存器操作均通过静态断言验证。

**缺点**：
- 未提供 ARM64/X86 等其他架构支持。

### 3.11 构建与可移植性 — 完整度：60%
**优点**：
- 构建系统较清晰，支持两种架构。
- 用户态模块与内核分离编译。

**缺点**：
- 构建强依赖 GCC 15+（`flags.mk` 要求主版本号 >= 15），当前主流发行版极少满足，极大限制可复现性。
- 用户态模块的编译依赖目标架构交叉工具链，配置灵活度一般。

### 3.12 动态测试
**测试设计与结果**：
- 代码库中包含 30 个用户态测试模块（`module/`），覆盖 fork/exec、线程、信号、内存、procfs、ext4 读写与权限、文件回写内存、页缓存、RPC、IPC 等。每个模块可独立编译为可执行文件，由内核加载运行。
- `contest-runner` 模块暗示存在针对比赛场景的自动化测试流程。
- 然而，由于构建环境当前无法提供 GCC 15+ 交叉编译器，**本次评估未能执行动态测试**。因此无法提供实际的运行结果或性能数据。
- 基于源码结构的分析，测试套件的设计是合理的，能够验证核心功能的正确性。

## 四、OS内核整体实现完整度

以 **“可运行多用户进程、提供文件持久化存储、进程间通信和基本系统管理的通用操作系统内核”** 为标准，Sustcore 的总体实现完整度约为 **75–80%**。

内核具备从启动、内存管理、进程调度、文件系统到 IPC 的完整闭环，且在能力安全与 ext4 支持上超出许多同类项目的深度。主要缺失在于多核支持、网络协议栈和更广泛的 POSIX 兼容性，这些限制了其作为“通用内核”的实用性，但作为技术原型和教学竞赛作品，已达到较高水准。

## 六、总结评价

Sustcore 是一个设计导向明确、实现扎实的能力型混合内核。项目以 Capability 作为全系统统一的资源访问抽象，从内存管理、进程调度、IPC 到文件系统均围绕该模型构建，展现出良好的安全设计理念和系统一致性。其内存管理子系统（Buddy+SLUB+GFP+COW）和从零实现的 ext4 文件系统在同类竞赛项目中尤为突出，说明开发者在底层基础设施上投入了大量精力。同时，双架构（RISC-V64 与 LoongArch64）的对称实现及 C++20 Concepts 的使用，体现了对可移植性和类型安全的前沿探索。

存在的不足主要集中在：仅支持单核、缺失网络协议栈、POSIX 兼容性有限，以及构建依赖 GCC 15+ 导致可复现性差。这些问题使其尚不能作为一个可直接投产的通用操作系统，但考虑到作品定位于技术展示与竞赛，这些缺陷多数属于“未实现”而非“实现有误”。

整体而言，Sustcore 在架构设计、安全模型和文件系统深度上具有显著创新价值，代码质量和工程组织也达到了较高水准，是一个优秀的内核技术作品。
