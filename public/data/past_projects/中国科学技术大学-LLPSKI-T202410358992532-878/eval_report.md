# StarsOS 内核项目技术画像与评估报告

## 一、 项目基本信息

- **项目名称**：StarsOS
- **目标架构**：RISC-V 64位 (rv64imafd)
- **实现语言**：C语言、RISC-V 汇编语言
- **生态归属**：类 Unix 操作系统，系统调用接口兼容 Linux
- **代码规模**：内核 C/汇编代码约 14,645 行，头文件约 5,588 行
- **核心特点**：支持 SMP 多核启动、实现 Sv39 虚拟内存与 Demand Paging、具备完整的 FAT32 文件系统与 VFS 抽象、提供本地 Socket 与 Futex 支持、采用被动映射机制优化内存分配。

## 二、 实现的子系统与功能

该项目实现了以下核心子系统及其对应功能：
1. **引导启动**：SMP 多核启动协议、内核页表初始化、子系统依次初始化。
2. **设备驱动**：16550 UART 串口轮询驱动、VirtIO MMIO 块设备驱动、PLIC 中断控制器配置、SBI 定时器驱动。
3. **内存管理**：物理页帧链表管理、Sv39 三级页表管理、分级对象池内核堆分配器、写时复制 (COW)、按需调页 (Demand Paging)。
4. **进程与线程管理**：进程与线程生命周期管理、FIFO 调度器、COW fork、clone 线程创建、睡眠与唤醒机制、定时睡眠。
5. **异常与中断处理**：用户态/内核态 Trampoline 机制、系统调用分发、缺页异常处理、时钟与外部中断处理。
6. **文件系统**：FAT32 文件系统驱动、VFS 抽象层、两层文件描述符管理、管道 (Pipe)、设备文件系统 (DevFS)、proc 文件系统 (ProcFS)。
7. **系统调用**：约 85 个 Linux 兼容系统调用，覆盖文件、进程、内存、信号、Socket 等领域。
8. **信号与定时器**：POSIX 信号处理机制、信号 Trampoline、间隔定时器 (itimer)。
9. **同步与 IPC**：自旋锁与睡眠锁、Futex (wait/wake/requeue)、System V 共享内存、本地 Unix Domain Socket (TCP/UDP)。

## 三、 子系统实现完整程度、优缺点及实现细节

### 1. 引导启动子系统
- **完整程度**：基础功能完备。
- **实现细节**：汇编入口为每个 hart 分配独立的初始栈，C 语言入口 `kernel_start` 由 hart 0 完成全局初始化并通过 SBI 唤醒其他核心，其他核心自旋等待后完成本地初始化。
- **优点**：多核启动流程清晰，使用了内存屏障保证同步。
- **缺点**：缺少对启动失败核心的优雅处理，核心数硬编码为 2，缺乏动态检测机制。

### 2. 设备驱动子系统
- **完整程度**：覆盖核心基础设备，缺乏网络与高级设备支持。
- **实现细节**：UART 采用 MMIO 轮询模式；VirtIO 块设备驱动实现了描述符表、可用环与已用环，采用忙等策略等待 I/O 完成；PLIC 配置了 S-mode 中断使能。
- **优点**：VirtIO 驱动结构符合规范，支持链式描述符。
- **缺点**：UART 未启用中断驱动模式；VirtIO 仅支持块设备，未实现 QEMU 参数中指定的网络设备驱动；I/O 采用忙等，并发性能受限。

### 3. 内存管理子系统
- **完整程度**：核心机制完备，缺乏高级内存回收机制。
- **实现细节**：物理内存采用空闲页链表管理；虚拟内存实现 Sv39 页表遍历与映射；内核堆采用分级对象池设计，最大支持 33 页分配。引入“被动映射”机制，在页表项中仅记录权限位而不设置有效位，触发缺页时再分配物理页。
- **优点**：支持 COW 和 Demand Paging，被动映射设计在竞赛级内核中具有较高的工程价值；内核堆分配器针对特定大小对象进行了优化。
- **缺点**：物理内存上限硬编码为 128MB；无页面置换 (Swap) 算法，内存耗尽时直接触发 panic。

### 4. 进程与线程管理子系统
- **完整程度**：生命周期管理完备，调度策略过于基础。
- **实现细节**：采用全局进程池与线程池，通过空闲链表分配。调度器采用全局单运行队列的 FIFO 策略。支持 `fork` (COW)、`clone` (共享地址空间)、`exec` 和 `wait`。睡眠机制分为基础睡眠和基于有序链表的定时睡眠。
- **优点**：进程与线程模型分离清晰，COW 机制与页表管理结合紧密，定时睡眠事件队列检查时间复杂度为 O(1)。
- **缺点**：纯 FIFO 调度无时间片轮转和优先级机制，易导致 CPU 密集型进程饿死其他进程；缺少孤儿进程重父化到 init 进程的完整实现。

### 5. 异常与中断处理子系统
- **完整程度**：用户态与内核态处理机制完备。
- **实现细节**：使用 Trampoline 页技术处理用户态中断，保存 32 个整数寄存器和 32 个浮点寄存器。缺页异常处理分发至 COW 处理函数或 Demand Paging 处理函数。
- **优点**：浮点寄存器上下文保存完整，Trampoline 机制符合 RISC-V 特权架构规范。
- **缺点**：内核态陷阱处理仅支持定时器和外部中断，其他异常直接 panic，缺乏内核态缺页处理能力。

### 6. 文件系统子系统
- **完整程度**：功能丰富，是该项目最复杂的子系统。
- **实现细节**：实现了完整的 FAT32 驱动，支持长文件名 (LFN) 和目录树构建。VFS 层抽象了 `FileSystem` 和 `Dirent`。文件描述符采用用户层与内核层两层设计。实现了 `/dev` 和 `/proc` 伪文件系统，以及基于环形缓冲区的管道。
- **优点**：FAT32 实现细节丰富，支持挂载和硬链接；两层文件描述符设计提高了 `fork` 时的 fd 复制效率；使用 `initcall` 机制自动注册 proc 文件。
- **缺点**：文件系统全局使用粗粒度的 `mtx_file` 睡眠锁，并发 I/O 性能受限；缺乏 ext4 等日志文件系统支持。

### 7. 系统调用子系统
- **完整程度**：以 Linux 核心系统调用子集（约 120 个）为基准，实现了约 85 个，覆盖率约 70%。
- **实现细节**：通过包含 1024 项的函数指针表进行分发，包含性能分析宏。
- **优点**：覆盖面广，涵盖了文件、进程、内存、信号、Socket 等核心领域。
- **缺点**：部分系统调用（如 `sched_getaffinity`、`prlimit64`）为桩实现，仅返回固定值；缺乏 `epoll`、`inotify` 等高级 I/O 多路复用机制。

### 8. 信号与定时器子系统
- **完整程度**：核心信号机制完备。
- **实现细节**：每线程维护信号队列与信号掩码，在返回用户态前检查信号。使用独立的 `sigSec` 段放置信号返回代码。实现了 `setitimer`/`getitimer` 周期性与一次性定时器。
- **优点**：支持 `siginfo` 和 `ucontext`，信号 Trampoline 与主 Trampoline 分离，设计合理。
- **缺点**：缺少 `sigaltstack` 支持，无法处理栈溢出时的信号。

### 9. 同步与 IPC 子系统
- **完整程度**：基础同步与本地通信完备，缺乏网络与高级 IPC。
- **实现细节**：实现了基于中断开关与原子操作的自旋锁，以及基于睡眠队列的睡眠锁，均支持可重入。Futex 支持 wait、wake 和 requeue。实现了 System V 共享内存和本地 Unix Domain Socket。
- **优点**：Futex 实现覆盖了核心操作，本地 Socket 支持 TCP 和 UDP 语义及 `socketpair`。
- **缺点**：缺少读写锁和条件变量；共享内存缺少 `shmdt` 分离操作和引用计数管理；Socket 仅限本地通信，无 TCP/IP 网络协议栈。

## 四、 OS 内核本身的实现完整度

以“类 Unix 教学与竞赛级操作系统核心功能”为基准，StarsOS 内核具备了完整的进程管理、虚拟内存、文件系统和系统调用骨架。其核心功能完整度较高，能够支撑多进程、多线程并发运行及复杂的文件系统操作。但在高级特性方面存在明显缺失，如网络协议栈、页面置换算法、高级调度策略和细粒度并发控制。整体而言，该内核是一个功能覆盖面广但部分高级机制停留在基础实现或桩实现阶段的竞赛级作品。

## 五、 动态测试的设计和结果

### 测试设计
项目在用户态设计了自动化测试驱动 `main.c`，该程序作为 init 进程的替代者，依次通过 `fork` 和 `exec` 运行 `riscv-syscalls-testing` 目录下的 33 个独立测试程序。测试用例覆盖了 `brk`、`clone`、`execve`、`mmap`、`pipe`、`mount` 等核心系统调用。每个测试程序运行完毕后，驱动通过 `wait` 系统调用收集其退出状态，以验证内核实现的正确性。此外，项目还集成了 busybox、lmbench、iozone 等性能测试工具的构建脚本。

### 测试结果
由于当前沙箱环境缺乏 `sudo` 权限，无法执行 `make` 流程中涉及的 `mount` 命令以完成 FAT32 磁盘镜像 (`sdcard.img`) 的制作与文件拷贝。虽然环境中存在预编译的 `kernel-qemu` (30MB) 和 `sdcard.img` (512MB) 产物，但受限于镜像挂载与验证环境的缺失，QEMU 动态运行测试未能实际执行。因此，无法提供基于动态运行的通过率或性能基准数据。

| **异常与中断** | 用户态处理完备，内核态处理基础 | 实现 Trampoline 机制，保存完整整数与浮点上下文；内核态仅处理定时与外部中断。 | 用户态陷阱处理规范且完整，但内核态异常处理过于简单，缺乏内核态缺页等高级异常处理能力。 |

## 七、 总结评价

StarsOS 是一个面向操作系统内核比赛的中等规模 RISC-V 64 位类 Unix 内核。该项目在文件系统、内存管理和系统调用覆盖面方面表现出较高的完成度。其 FAT32 文件系统实现详尽，VFS 抽象与伪文件系统的设计体现了良好的架构思维；内存管理子系统引入的“被动映射”机制和 Demand Paging 展现了较强的工程实现能力；约 85 个系统调用的实现为上层应用提供了较好的 Linux 兼容性基础。

然而，该项目在部分核心机制的设计上存在明显的局限性。全局单队列 FIFO 调度器无法应对复杂的多任务负载；文件系统的全局粗粒度锁削弱了 SMP 多核架构的性能优势；缺乏页面置换和网络协议栈限制了内核的实际应用场景。此外，部分系统调用的桩实现和孤儿进程处理的不完善表明项目在细节打磨上仍有提升空间。总体而言，StarsOS 是一个结构清晰、核心功能完备、在特定子系统上有深入探索的竞赛级内核作品，适合作为操作系统教学与研究的参考案例。
