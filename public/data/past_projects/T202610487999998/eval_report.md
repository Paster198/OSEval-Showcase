# 技术画像与评估报告

## 一、项目基本信息

- **项目名称**: oskernel2026‑x
- **架构**: RISC‑V 64 (rv64gc), LoongArch 64 (LA64)
- **实现语言**: C (GNU11), 汇编 (RISC‑V / LoongArch)
- **代码规模**: 约 8 000 行 (不含空行/注释；22 个源文件)
- **生态归属**: 独立实现的 monolithic 内核，面向 OS 竞赛赛道
- **主要特点**:
  - 双架构支持 (RISC‑V 用户态较完整，LoongArch 仅启动和输出)
  - 面向竞赛测试套件的最小化实现，大量测试采用“合成输出”绕过真实执行
  - 基础的 VirtIO‑blk 驱动 + 只读 EXT4 (extent 树) 文件系统
  - 合作式多进程调度，约 140 个系统调用分发框架
  - 统一文件描述符抽象 (15 种类型，包含内存文件、管道、虚拟设备等)

## 二、子系统功能与完整度分析

### 2.1 架构引导层 (Architecture Boot)

**RISC‑V 64**  
- 实现：多核启动检测（副核 `wfi` 自旋）、BSS 清零、栈初始化、`kmain()` 调用；用户态上下文进入/退出 (`user_enter` / `user_resume` / `user_trap_entry`)；基于 `stvec` / `sscratch` 的陷阱处理；SBI 字符输出与关机。  
- 完整度：中高。缺少中断控制器初始化，副核不支持 SMP，浮点寄存器延迟保存未完善。  
- 优缺点：陷阱入口的 `sscratch` 原子交换设计简洁高效；但仅处理 `scause=8` 的 ecall，其他异常直接 `panic` 返回。

**LoongArch 64**  
- 实现：DMW 直接映射窗口配置、BSS 清零、MMIO UART 输出、GED 关机。  
- 完整度：低。无用户态陷阱处理、无页表、无系统调用，所有用户态代码路径返回‑1。  
- 优缺点：启动流程清晰，但功能极度受限，本质为“输出框架”。

### 2.2 块设备驱动——VirtIO‑blk

- 文件：`kernel/common/virtio.c` (431 行)  
- 实现功能：Legacy MMIO (RISC‑V) 与 Legacy PCI IO (LoongArch) 两条路径；设备发现、协商、单队列配置；描述符链实现扇区读取；`block_read` 跨扇区封装。  
- 完整度：中等。仅支持只读，纯轮询无中断，单队列，不支持现代 VirtIO 规范。  
- 优缺点：双传输方式适配体现了驱动层次的设计，但缺少写入能力限制了文件系统完整性。

### 2.3 EXT4 文件系统

- 文件：`kernel/fs/ext4.c` (708 行)  
- 实现功能：超级块解析 (magic 校验、块大小/索引节点大小/组描述符大小)；extent 树递归遍历 (深拷贝保护)；线性目录项解析；路径查找 (`ext4_lookup_path`)，支持 `.`/`..` 与相对/绝对路径。  
- 完整度：中低。只读访问，仅处理 extent 格式，不支持间接块、日志、权限检查、扩展属性，目录缓冲区固定 4 KB。  
- 优缺点：extent 树遍历实现正确，支持 64 位文件大小；但大量现代 EXT4 特性缺失，无法写入，仅满足读取 ELF 和测试数据的基本需求。

### 2.4 文件描述符与虚拟文件系统层

- 文件：`kernel/fs/fd.c` (78 行) + `user_fd.h`  
- 实现功能：15 种 FD 类型统一管理 (包括普通文件、目录、管道、内存文件、`/dev/null`、`/dev/zero`、`/dev/random`、eventfd、timerfd、epoll 等)；FD 分配/释放；管道环形缓冲 (1 KB) 读写。  
- 完整度：中高。抽象层清晰，类型分派通过 `switch` 在 `read/write/close` 中实现；但资源硬编码（最大 64 个 FD、16 个管道），`/dev/random` 为简单伪随机序列。  
- 优缺点：统一接口极大简化了系统调用实现，但未与真正的 VFS 分离，扩展性有限。

### 2.5 进程/用户态管理核心

- 文件：`kernel/proc/user.c` (4 492 行，占总代码量的 67%)  
- 包含子模块：  
  - **物理内存分配**：64 MB 固定页池，位图管理，O(n) 首次适配。  
  - **Sv39 页表**：三级页表 walk、按需创建、恒等/用户映射、深拷贝 (用于 fork)、按标志位释放。  
  - **用户栈构建**：栈顶布局 argc/argv/auxv，遵循 Linux 惯例。  
  - **mmap 管理**：最多 32 个区域，支持 `MAP_SHARED` 脏页同步。  
  - **ELF 加载器**：静态/动态 ELF 加载，动态链接器路径映射，`PT_LOAD` 段权限设置。  
  - **进程控制**：进程结构 (`user_proc`)，状态机 (RUNNABLE/RUNNING/WAITING/ZOMBIE)，合作式调度，`clone` (仅 fork 语义) / `wait4` / `exit` 实现。  
  - **系统调用分发**：约 140 个系统调用，大量存根；真正的文件 I/O、内存管理、进程管理、时间查询等有实际实现。  
- 完整度：整个模块为内核的核心，实现度“中高” (真实执行路径)。  
- 优缺点：ELF 加载器与页表管理实现水平较高；但调度无抢占、无时间片，同步原语仅为存根 (futex 不阻塞)，信号全部返回 0，网络、定时器等完全模拟。

### 2.6 公共库与测试调度

- **公共库**：基本格式化输出 (`kprintf`, `put_char`)、字符串操作 (`memset`, `memcpy`, `strcmp`, `strlen`)，功能完整。  
- **测试调度 (main.c)**：系统启动后自动执行预定义测试组 (basic‑musl, basic‑glibc, ltp, lua, busybox 等)。  
  - **合成输出策略**：除少量 musl‑basic 测试在 RISC‑V 上实际执行外，其他所有组均直接打印硬编码的“期望输出”。  
  - **结果判定**：通过字符串匹配评分脚本，而非真正验证程序行为。  
- 完整度：测试框架本身完整（发现了测试文件、调度逻辑存在），但多数测试不反映 OS 的真实能力。

## 三、动态测试的设计与结果

### 3.1 测试执行模式

| 测试组 | 真实执行 | 合成输出 | 条件 |
|-------|----------|----------|------|
| basic‑musl | 部分 (25/32) | 剩余 7 个 (clone, execve, fork, pipe, wait, waitpid, yield) | RISC‑V 且 EXT4 挂载成功 |
| basic‑glibc | 无 | 全部合成 | - |
| ltp‑musl | 部分 (排除 mmap01, munmap01, nanosleep01, getdents01 等) | 其余 | RISC‑V 且 EXT4 挂载成功 |
| ltp‑glibc | 无 | 全部合成 | - |
| lua / busybox / libctest / libcbench / iozone / lmbench / cyclictest / iperf / netperf | 无 | 全部合成 | - |
| LoongArch 所有组 | 无 | 全部合成 | 强制 `force_synthetic_lib` |

### 3.2 真实执行路径的测试表现

- 实际运行的 basic 测试覆盖了文件描述符操作、内存分配、目录操作、stat、时间、uname 等基础系统调用。  
- 所有真实执行均依赖内核的 EXT4 读取、ELF 加载、页表映射、陷阱处理等子系统，这些测试能够通过证明了核心路径的功能正确性。  
- 由于没有记录具体测试输出（报告中仅提到“通过”合成输出），无法定量衡量真实执行时的稳定性或边界条件表现。  
- 临界情况：`clone`/`execve`/`pipe` 等使用了合成输出，表明其 fork 语义或管道实现尚不足以支持一般程序（如 `fork` 测试）。

### 3.3 测试结果可靠性评价

- 合成输出保证了评分脚本上的“全绿”，但与真实 OS 能力脱钩。  
- 动态测试的实际意义仅限于基础 musl 测试那 25 个用例，它们验证了最小化的用户态运行路径。其余测试结果不具有内核能力证明力。

## 四、细则评价表格

| 条目 | 是否实现及完整度 | 关键发现 | 评价 |
|------|----------------|----------|------|
| **内存管理** | 部分实现，完整度中 | 物理内存为静态 64 MB 位图管理；Sv39 三级页表，支持用户映射、恒等映射、fork 深拷贝、mmap；无 demand paging，无 swap，无共享内存优化 | 满足简单 ELF 加载和进程运行的基本需求，但无法支持内存超用或高级内存策略 |
| **进程管理** | 部分实现，完整度中低 | 支持 fork (无 COW)、execve、exit、wait4；合作式调度，最多 64 进程；无优先级，无真实时间片，clone 不支持线程 | 能运行多进程测试程序，但调度简陋，同步原语缺失，无法支撑真正的并发应用 |
| **文件系统** | 部分实现，完整度低 | 只读 EXT4 (extent only)，不支持间接块、日志、写入；虚拟 /proc、/dev 仅为固定内容模拟；路径查找支持符号链接一次 | 可作为比赛测试的 ELF/数据源，但绝非一个可用的文件系统，写入能力完全缺失 |
| **交互设计** | 未真正实现，完整度极低 | 所有“交互”通过合成输出模拟，内核本身不提供 shell 或命令行交互；控制台输出仅通过 `kputs` 打印预定义字符串 | 高度面向竞赛评分的取巧设计，不具备通用 OS 的交互界面 |
| **同步原语** | 存根实现，完整度极低 | futex 的 WAIT 不阻塞，WAKE 返回 0；信号全部返回 0；无实际的锁、信号量、条件变量等 | 无法支持多线程或进程间同步，对依赖正确同步的用户程序无效 |
| **资源管理** | 基础实现，完整度中低 | FD (64)、管道 (16)、进程 (64)、mmap 区域 (32) 均静态分配，无动态扩展或回收策略；无资源限额 (rlimit 固定返回 1 GB) | 边界明确，足以应对比赛用例，但缺乏健壮性，资源泄漏风险高 |
| **时间管理** | 存根实现，完整度低 | `gettimeofday` 每次调用 +100 µs 虚拟时间；无 RTC、无定时器中断、无 nanosleep/定时器回调 | 仅用于提供单调递增的时间值，不具备真实计时功能 |
| **系统信息** | 基础实现，完整度中 | `uname` 返回固定串；`/proc/cpuinfo`/`/proc/meminfo`/`/proc/self/status` 等硬编码；缺少真实硬件探测 | 提供比赛脚本所需的最小信息集，无实际系统监控能力 |

## 五、总结评价

`oskernel2026‑x` 是一个在 OS 竞赛约束下展现出高效工程判断的内核作品。其最核心的成功策略在于**精确识别测试套件的最小通过路径**：仅对一小部分 musl‑basic 测试真正实现用户态运行，其余通过预定义的“合成输出”绕过实际执行。这一策略使得项目能够在代码量受限的前提下获得评分上的完全通过。

从单纯的技术角度看，该内核具备一个可工作的 monocore 系统的基本骨架：VirtIO‑blk 驱动、只读 EXT4 解析器、Sv39 虚拟内存、ELF 动态链接加载、约 140 个系统调用的分发框架以及协作式多进程。在 RISC‑V 架构上，上述子系统的组合足以运行静态或动态链接的 ELF 程序，并响应用户态的系统调用。

然而，该内核的**局限性极为显著**：几乎所有非 basic 的测试组均未经过真实执行验证；同步原语、中断、磁盘写入、定时器、线程支持等现代 OS 的核心能力几乎全部缺失或仅为存根。内核的行为很大程度上由硬编码的预期输出定义，而非由真实的系统行为决定。

因此，`oskernel2026‑x` 应当在**竞赛项目**的语境下被评价：它是一个精心设计、高度工程化的竞赛解决方案，在给定的评分体系下达到了高效得分的目的。若作为教学或原型研究，它的代码简洁、分层清晰仍有一定参考价值；但若以通用操作系统内核的标准衡量，其完整性与健壮性均处于较低水平。
