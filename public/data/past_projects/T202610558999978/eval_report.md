## 一、项目基本信息

| 项目     | 内容                                 |
|----------|--------------------------------------|
| 项目名   | BirdOS（参赛名：OSKernel2026_mini）  |
| 所属队伍 | KernelTrap（中山大学）               |
| 架构     | RISC-V 64-bit（RV64），Sv39 分页    |
| 实现语言 | C（内核约 8,565 行，用户程序约 7,261 行，头文件约 1,507 行） |
| 内核类型 | 宏内核（Monolithic Kernel）          |
| 生态归属 | 基于 xv6-riscv 基线扩展的自有内核    |

## 二、子系统实现与功能概览

BirdOS 在 xv6 基线之上实现了以下子系统和功能：

- **进程管理**：完整进程生命周期（创建、执行、等待、退出），53 个系统调用入口，动态优先级调度器，clone/join 线程支持，定时器信号（sigalarm/sigreturn）。
- **内存管理**：COW（写时复制）fork，每 CPU 物理页分配器（引用计数），mmap/munmap 懒加载文件映射，共享内存区域，缺页异常处理（COW/lazy/mmap/shm）。
- **文件系统**：混合三级索引（直接、一级间接、二级间接），最大文件约 67 MB，文件权限（rwx）、符号链接、文件恢复辅助系统调用（geti/recoveri），日志系统。
- **缓冲区缓存**：13 个哈希桶的细粒度锁缓冲区管理层，采用时间戳 LRU 回收策略。
- **同步原语**：自旋锁、睡眠锁、System V 风格信号量（创建、P 操作、V 操作、释放）。
- **进程通信（IPC）**：管道（512 字节循环缓冲）、消息队列（4 KB 内存池，变长消息，类型匹配接收）、共享内存（8 个全局区域，支持引用计数）。
- **网络子系统**：UDP/IP/ARP 协议栈，e1000 网卡驱动（发送环和接收环各 16 个描述符），简单 socket 接口（UDP 套接字、阻塞收发），PCI 枚举。
- **设备驱动**：UART 16550A（中断驱动发送）、virtio-blk 块设备驱动、RAM 磁盘基础框架。
- **中断管理**：PLIC 中断控制、时钟中断（含用户态定时器回调）、缺页异常多路径分发。

## 四、各子系统优缺点及实现细节

### 4.1 进程管理
- **优点**
  - 实现了**动态优先级调度**：优先级公式 `dyn = prio + wait_time/5 - cpu_time/5`，能自动平衡 I/O 密集与计算密集进程的 CPU 占用，一定程度防止饥饿。
  - 支持 **clone/join** 线程模型：子线程共享页表，通过 `pthread` 字段记录父子关系，等待回收逻辑改造较小。
  - 定时器信号机制完整：支持 `sigalarm` 设置间隔回调，`sigreturn` 恢复上下文，且采用 `alarm_goingoff` 防止重入。
- **缺点与局限**
  - 调度器缺乏时间片轮转：同一最高优先级进程可能长期占据 CPU，低优先级进程仍有饥饿风险。
  - clone 实现未提供独立线程 ID、线程退出通知等机制，与 POSIX 线程模型差距较大。
  - 进程控制块中多个新增字段增加了内存占用，但尚未触及 `NPROC`=64 限制。

### 4.2 内存管理
- **优点**
  - **COW fork** 完整度较高：利用 PTE 保留位（`PTE_F`）标记共享页，结合 `krefcnt` 全局引用计数实现写时分配。`cowalloc` 处理单引用直接恢复写权限的优化路径。
  - **mmap 懒加载**设计较系统：通过 VMA 结构记录映射区域，缺页时读取文件内容，支持 MAP_SHARED/MAP_PRIVATE 语义，`munmap` 可回写共享页。
  - 每 CPU 物理页分配器 + 跨 CPU 窃取机制有效降低全局锁竞争。
- **缺点与局限**
  - 所有缺页异常（COW、mmap、sbrk、共享内存）均复用 `usertrap` 同一分支，处理逻辑较长且嵌套锁较多。
  - mmap 实现假设 `addr=0` 和 `offset=0`，未实现用户指定地址或文件偏移映射。
  - 物理内存固定 128 MB，无内存回收/换出机制。

### 4.3 文件系统
- **优点**
  - **三级混合索引**显著扩展了文件容量上限（约 268 KB → 67 MB），`bmap`/`itrunc` 逻辑清晰完整。
  - **符号链接**跟随深度限制达 10 层，并在 `sys_open` 路径中自动解析。
  - 引入 ACL 权限位（`mode`），配合 `chmod` 系统调用实现基本读写保护。
  - **geti/recoveri** 提供了一种应用级文件恢复辅助手段，有一定创意。
- **缺点与局限**
  - 权限仅区分 r/w/x 三位，未实现用户/组/其他分离。
  - 大文件支持虽达 67 MB，但受限于日志区和 virtio 块设备可能的大小（实际磁盘镜像限制），且大文件随机读写时二级间接块开销较大。
  - 没有索引节点缓存逐出策略的显式优化。

### 4.4 缓冲区缓存
- **优点**
  - 采用**13 个哈希桶 + 每桶自旋锁**的设计，相比 xv6 全局锁大幅降低多核 I/O 冲突。
  - 使用全局 `ticks` 时间戳进行 LRU 替换，跨桶搜索未使用缓冲，提高利用率。
- **缺点**
  - 缓冲区总数固定（编译时 `NBUF`），没有动态伸缩。

### 4.5 同步原语
- **优点**
  - 自旋锁和睡眠锁划分清晰：自旋锁用于短临界区，睡眠锁用于 inode 等长时操作。
  - 信号量实现简洁且支持阻塞等待与唤醒，配合进程锁使用避免了部分死锁场景。
- **缺点**
  - 未实现更复杂的同步结构（如读写锁、条件变量），使得某些并发场景（如大量读共享数据）必须退化为互斥访问。

### 4.6 IPC
- **优点**
  - 消息队列支持变长消息，`reloc` 碎片整理函数尝试解决内部碎片，设计有一定巧思。
  - 共享内存支持 fork 继承和引用计数管理，使用位图 `shmkeymask` 跟踪 8 个段。
- **缺点**
  - 消息队列单区仅 4 KB，容量较小，碎片整理可能无法回收任意顺序释放的碎片，导致大消息无法发送。
  - 共享内存每个区域最多 4 页（16 KB），限制了大数据共享场景。

### 4.7 网络子系统
- **优点**
  - 从协议栈（Ethernet → ARP → IP → UDP）到底层 e1000 驱动均为独立实现，展示了良好理解。
  - mbuf 数据包缓冲区支持头空间（headroom）操作，便于协议头高效添加。
  - socket 层设计简洁，阻塞读写语义基本正确。
- **缺点**
  - 仅支持 UDP，无 TCP 状态机，无法提供可靠传输。
  - 未实现 IP 分片重组，不能处理大于 MTU 的 UDP 数据报。
  - ARP 表项无老化机制，长期运行可能缓存失效条目。
  - 无 DHCP 等地址配置手段，IP 地址可能硬编码。

### 4.8 中断与异常处理
- **优点**
  - 缺页异常分发顺序清晰（COW → mmap → sbrk lazy → shm lazy），且各自有对应的回退方案（无匹配则杀死进程）。
  - 定时器信号设计完整，支持用户态回调。
- **缺点**
  - 多个懒分配处理路径共享同一个异常号时，需要依赖地址范围判断，可能因竞态或逻辑错误导致误判。
  - 没有统计各类异常的发生频率等监测机制。

## 五、动态测试的设计和结果

**测试环境与构建情况**：
- 使用 RISC-V GNU 工具链（riscv64-unknown-elf-gcc 13.2.0）成功编译内核，修正一处数组参数声明不匹配的 `-Werror` 问题后生成 `kernel/kernel` ELF（326,576 字节）。
- 用户程序全部编译成功（46 个用户可执行程序）。
- 因仓库未提供 `mkfs/mkfs.c` 源码，无法生成 `fs.img` 文件系统镜像，导致 QEMU 启动过程因 virtio-blk 缺失磁盘镜像而中止，未能执行动态功能测试。

**测试设计观察**：
- 项目提供了大量用户态测试程序，覆盖多数新增系统调用，例如：
  - `cowtest`：测试 COW fork 的正误行为。
  - `mmaptest`：测试 mmap/munmap 映射与共享写回。
  - `pgrptest`/`threadstest`：测试调度与线程创建。
  - `sharememtest`、`mqtest1/2`、`semtest`、`nettest`、`alarmtest` 等专项测试。
- 各测试程序通常采用 `printf` 断言形式输出成功或失败信息，具备一定的自动化测试框架雏形。
- 测试结果的客观评估需待完整环境恢复运行后确认，当前仅能肯定内核构建流程正确，用户态测试程序可链接。

## 七、总结评价

BirdOS 是一个以 xv6-riscv 为起点、在多个维度上进行实质性扩展的操作系统内核。项目在调度、内存管理、文件存储、并发性能优化和网络协议栈等方面均展现了较强的系统工程能力和理论基础。新增的 32 个系统调用覆盖广泛，使得内核拥有的功能集合远超原始教学基线，且多数功能有配套的用户态测试程序验证设计正确性。

主要亮点集中在动态优先级调度算法的平衡设计、COW 和 mmap 懒加载的完整实现、文件系统三级索引与符号链接的引入，以及针对多核场景的物理页分配器和缓冲区缓存锁细化优化。网络协议栈虽仅支持 UDP 且缺少许多生产级特性，但从零构建的实践价值较高。消息队列、共享内存和信号量等 IPC 机制的实现让内核具备了更丰富的进程间通信能力。

不足之处主要体现为以下方面：调度器缺乏时间片轮转保障；mmap 接口未完整实现用户指定地址和偏移；网络子系统无 TCP 可靠传输；消息队列和共享内存容量偏小且硬编码上限较多；部分子系统（如线程、权限）仍停留在最小可行版本，距离通用 POSIX 兼容有较大距离。

综合来看，BirdOS 在竞赛/教学内核范畴中是一项完成度高、创新点明确、工程实现较为扎实的作品。尽管部分功能仍处于基础可用阶段，但整体体现了团队对操作系统核心概念的深入理解和较完整的工程实践能力。
