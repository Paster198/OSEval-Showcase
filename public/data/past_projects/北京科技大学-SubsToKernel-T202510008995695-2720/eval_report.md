# SubsToKernel OS 内核项目技术评估报告

## 1. 项目基本信息

- **项目名称**：SubsToKernel
- **支持架构**：RISC-V 64位、LoongArch 64位
- **实现语言**：Rust（主体，91个源文件）、汇编（6个源文件，用于启动与上下文切换）、C（外部文件系统库绑定）
- **生态归属**：基于 rCore-Tutorial-v3（ch8 分支）开发，为 2025 春秋季开源操作系统训练营参赛作品（北京科技大学）
- **核心特点**：
  - 双架构支持，包含 LoongArch 架构下的 TLB 快速重填与 DMW 窗口配置。
  - 内存管理集成 COW（写时复制）、Lazy Allocation（延迟分配）与跨 fork 共享组机制。
  - 同步子系统引入银行家算法进行死锁避免，并实现了完整的 Linux 风格 Futex。
  - 进程管理支持 ELF 动态链接器加载与辅助向量（Auxiliary Vector）设置。
  - 设备驱动支持 VirtIO 块设备与网络设备的 MMIO 及 PCI 两种传输方式。

## 2. 子系统与功能实现概述

该项目实现了一个具备基础 Linux 兼容性的宏内核，主要包含以下子系统与功能：
- **启动与硬件抽象层**：双架构汇编启动、BSS 段清理、Trap 异常与中断处理、UART 控制台输出。
- **内存管理**：SV39/48位分页、物理页帧栈式分配、COW、mmap/munmap/mprotect、System V 共享内存、缺页异常处理。
- **进程与任务管理**：进程/线程控制块、fork（含 clone 语义）、exec、waitpid、FIFO 调度、Stride 调度数据结构。
- **文件系统**：VFS 抽象层、基于 lwext4 的 ext4 文件系统、管道、设备文件系统（devfs）、挂载管理。
- **系统调用**：实现 100 余个系统调用，覆盖文件 I/O、进程控制、内存映射、网络、信号与同步。
- **网络通信**：基于 smoltcp 的 TCP/UDP 套接字、DNS 解析、本地回环接口。
- **同步与信号**：互斥锁、信号量、条件变量、银行家算法、33 种 POSIX 信号处理。
- **设备驱动**：VirtIO 块设备与网络设备驱动、PCI 总线枚举与 BAR 分配。

## 3. 子系统实现完整度与细节分析

*注：以下各子系统的“完整度”评估均以 **rCore-Tutorial-v3 ch8 分支的教学功能要求** 为基准（100% 代表完全覆盖并达到该教学大纲的扩展要求）。*

### 3.1 启动与硬件抽象层 (HAL)
- **完整度**：85%
- **实现细节**：启动分为汇编与 Rust 两阶段。LoongArch 架构通过汇编手动配置直接映射窗口（DMW）实现高低地址跳转，并在汇编层实现了三级页表的 TLB 快速重填。RISC-V 采用标准的 Trampoline 机制。Trap 上下文保存 32 个通用寄存器与 32 个浮点寄存器。
- **优点**：LoongArch 的 TLB 重填直接在汇编中完成，减少了异常处理的上下文切换开销；双架构代码通过条件编译实现了较好的复用。
- **缺点**：仅支持单核启动，`cpu` 参数在 `main` 函数中未被使用；中断处理仅实现了定时器中断，缺乏外部中断控制器的支持。

### 3.2 内存管理
- **完整度**：90%
- **实现细节**：采用栈式物理页帧分配器与 RAII 模式的 `FrameTracker`。支持 SV39（RISC-V）与 48 位三级页表（LoongArch）。实现了基于 PTE 第 9 位的 COW 机制、堆栈的 Lazy Allocation，以及通过 `GroupManager` 实现的 `MAP_SHARED` 跨 fork 页面共享。支持 System V 风格的 `shmget/shmat` 共享内存。
- **优点**：内存映射区域（`MapArea`）通过枚举区分 ELF、Stack、Heap、Mmap 等类型，便于 fork 时进行差异化处理；缺页异常处理逻辑覆盖了延迟分配、COW 与文件映射。
- **缺点**：未实现 Swap（交换空间）机制与物理内存回收机制，在物理内存耗尽时无法进行页面置换。

### 3.3 进程与任务管理
- **完整度**：85%
- **实现细节**：区分进程控制块（PCB）与任务控制块（TCB）。`fork` 支持 `CLONE_VM/FS/FILES/THREAD` 等 Linux 语义。`exec` 支持检测 `PT_INTERP` 段以加载动态链接器，并设置 AT_PHDR、AT_BASE 等 11 种辅助向量。调度器实际采用 FIFO 就绪队列，虽实现了 Stride 调度算法的数据结构，但未集成至主调度循环。
- **优点**：进程与线程模型清晰，支持完整的线程组管理与 TID 设置；动态链接支持使得 musl libc 动态链接程序具备运行条件。
- **缺点**：调度器仅为简单的 FIFO，Stride 调度未实际启用；缺乏 SMP（多核）支持，无法利用多核算力。

### 3.4 文件系统
- **完整度**：80%
- **实现细节**：VFS 层定义了 Inode、File、Sock 三组 Trait。底层通过 `lwext4_rsut` 绑定 C 库实现 ext4 文件系统。实现了 64KB 环形缓冲区的管道，以及包含 zero、null、rtc、random 等节点的 devfs。支持最多 16 个挂载点的 mount/umount 操作。
- **优点**：VFS 抽象层设计合理，通过 `FileClass` 枚举统一了普通文件、抽象文件与套接字的操作接口；管道支持批量读写优化与 SIGPIPE 信号发送。
- **缺点**：procfs 与 sysfs 仅为启动时创建的静态文件，不具备动态反映系统状态的能力；不支持硬链接，符号链接支持有限。

### 3.5 系统调用与交互设计
- **完整度**：85%（系统调用），60%（交互设计）
- **实现细节**：实现了 100 余个系统调用，涵盖文件、进程、内存、网络、信号等。用户态仅提供 4 个基础程序（initproc、user_shell、usertest、test_waitpid）。
- **优点**：系统调用覆盖面广，参数解析通过 `bitflags` 宏定义了完善的标志位类型；支持 `pselect6`、`ppoll` 等高级 I/O 多路复用调用。
- **缺点**：部分系统调用（如 `madvise`、`setsid`、`sched_getaffinity`）仅为桩实现；缺乏 `epoll` 和 `inotify` 机制；用户态 Shell 功能极简，缺乏作业控制与管道重定向等高级交互功能。

### 3.6 网络通信
- **完整度**：75%
- **实现细节**：基于 smoltcp 协议栈，实现了 TCP 与 UDP 套接字。TCP 支持阻塞/非阻塞模式及完整的状态机流转；UDP 支持数据报收发与超时接收。实现了基础的 DNS 查询与本地回环接口。
- **优点**：套接字状态机模型清晰，监听表（Listen Table）有效管理了端口复用与连接队列。
- **缺点**：缺乏 IPv6 支持；Socket 选项（setsockopt/getsockopt）处理有限；DNS 实现较为基础，缺乏缓存与重试机制。

### 3.7 同步与信号处理
- **完整度**：85%（同步），80%（信号）
- **实现细节**：同步原语包含自旋/阻塞互斥锁、计数信号量、条件变量，并引入了银行家算法进行死锁避免。实现了支持 WAIT/WAKE/REQUEUE/WAKE_OP 的 Futex。信号子系统支持 33 种 POSIX 信号及自定义处理函数。
- **优点**：银行家算法在资源分配前进行安全性检查，在教学 OS 中具备特色；Futex 使用物理地址作为键，正确区分了进程内与跨进程共享。
- **缺点**：缺乏内核级读写锁（RwLock）；信号机制缺乏实时信号的排队处理（信号队列）与备用信号栈（sigaltstack）的完整支持。

### 3.8 设备驱动
- **完整度**：75%
- **实现细节**：基于 `virtio-drivers` crate 实现了块设备与网络设备驱动。支持 RISC-V 的 MMIO 传输与 LoongArch 的 PCI 传输。PCI 驱动实现了总线枚举、BAR 地址分配与命令寄存器配置。
- **优点**：双架构下的设备传输方式适配完整，PCI 枚举逻辑具备较高的工程完成度。
- **缺点**：仅支持块设备与网络设备，缺乏 VirtIO GPU、输入设备等其他外设支持；缺乏设备热插拔与电源管理。

## 4. OS 内核整体实现完整度

以 **rCore-Tutorial-v3 ch8 分支的教学功能要求** 为基准，SubsToKernel 的整体实现完整度约为 **85%**。该项目在教学大纲的基础上进行了大量实质性扩展，特别是在双架构适配、内存映射优化、Linux 兼容系统调用以及同步死锁避免方面达到了竞赛级 OS 的标准。但在多核并行处理、高级调度算法、内存页面置换以及动态文件系统（procfs）方面仍存在明显的功能缺失。

## 5. 动态测试设计与结果

**测试结果：缺失**

**原因说明**：
在当前环境中尝试构建 RISC-V 与 LoongArch 目标时，构建流程中断，未能生成内核镜像，导致无法在 QEMU 中进行动态测试。具体阻碍因素如下：
1. 项目依赖的 `lwext4_rsut` 库需要通过 `cc` crate 编译 C 源码，其构建脚本对 RISC-V/LoongArch 交叉编译器的路径有特定要求，与当前环境中的工具链路径不匹配。
2. 项目运行依赖的 `sdcard-rv.img` 文件系统镜像未包含在代码仓库中，导致即使内核编译成功也无法挂载根文件系统。

## 7. 总结评价

SubsToKernel 是一个在教学 OS 基础上进行了深度扩展的内核项目。其核心技术优势在于对 RISC-V 与 LoongArch 双架构的良好适配，特别是 LoongArch 架构下汇编级 TLB 重填与 DMW 窗口配置的实现，展现了扎实的底层硬件理解能力。在内存管理方面，COW、延迟分配与共享组机制的组合有效优化了内存使用效率；在同步子系统引入银行家算法与完整的 Futex 实现，提升了系统的并发控制能力与 Linux 兼容性。

然而，该项目在工程完整性与高级特性上仍存在局限。首先，单核架构与简单的 FIFO 调度器限制了系统的吞吐与响应能力；其次，缺乏 Swap 机制与内存回收策略，使得系统在长时间运行或高内存负载下存在稳定性风险；最后，由于构建环境依赖与镜像缺失导致动态测试未能执行，其实际运行时的稳定性与系统调用的正确性缺乏运行时数据的验证。总体而言，该项目在功能广度与特定技术深度上表现良好，具备较高的学术与工程实践价值。
