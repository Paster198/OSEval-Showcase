# NoAxiom-OS 内核项目技术画像与评估报告

## 一、 项目基本信息

- **项目名称**：NoAxiom-OS
- **支持架构**：RISC-V64、LoongArch64
- **实现语言**：Rust（主体逻辑）、汇编（底层入口、陷阱处理与上下文切换）
- **内核类型**：宏内核（Monolithic Kernel）
- **生态归属**：独立自研，深度集成 Rust 异步生态（如 `async_task`、`smoltcp`）
- **核心特点**：
  - 基于 Rust 无栈协程（`async/await`）的异步调度与 IO 架构。
  - 通过硬件抽象层（HAL）实现 RISC-V64 与 LoongArch64 双架构支持。
  - 采用细粒度并发数据模型，按访问模式对任务控制块字段进行分类锁保护。
  - 面向对象的虚拟文件系统（VFS）设计与多文件系统支持。

## 二、 实现的子系统与功能

该项目实现了现代操作系统所需的核心子系统，具体包括：
1. **调度子系统**：多级优先级调度（实时 FIFO + 普通 Expired 双队列）、时间片轮转、CPU 亲和性。
2. **进程与任务管理**：进程/线程创建（fork/clone）、程序加载（execve）、进程等待（wait4）、进程组与线程组管理、Futex 同步。
3. **内存管理**：虚拟地址空间管理、页表操作、写时复制（COW）、按需分页（懒分配）、mmap/munmap、System V 共享内存。
4. **文件系统**：VFS 抽象层、EXT4/FAT32/RamFS/ProcFS/DevFS 实现、页缓存、块缓存、管道、符号链接与硬链接。
5. **网络子系统**：基于 smoltcp 的 TCP/UDP 协议栈、IPv4/IPv6 支持、Socket API。
6. **信号子系统**：64 个标准与实时信号、信号处理动作、信号掩码、可中断系统调用。
7. **IO 多路复用**：ppoll、pselect6。
8. **时间管理**：系统时钟、高精度定时器、间隔定时器（itimer）、时间统计。
9. **硬件抽象与驱动**：双架构 HAL、virtio-blk/virtio-net 异步驱动、AHCI 驱动、PLIC 中断控制器。
10. **中断与异常处理**：用户态/内核态陷阱分发、页面错误处理、异步系统调用分发。

## 三、 各子系统实现完整程度

以下完整度评估均以 Linux 内核对应子系统的功能覆盖范围为基准（100% 代表完全覆盖 Linux 对应子系统的核心功能）：

- **调度子系统**：约 75%。实现了多级调度与时间片轮转，但更复杂的 CFS 调度与多核负载均衡未实际启用。
- **进程与任务管理**：约 85%。覆盖了核心的 clone/exec/wait 语义及 futex，部分边缘进程控制接口缺失。
- **内存管理**：约 80%。实现了 COW、懒分配与 mmap，但缺乏显式的 msync 支持及更高级的内存回收策略。
- **文件系统**：约 80%。VFS 与 5 种具体文件系统实现完整，但 sync/fsync 等持久化同步接口为空实现。
- **网络子系统**：约 70%。TCP/UDP 核心功能完备，缺乏 Unix Domain Socket 及高级套接字选项。
- **信号子系统**：约 75%。支持 64 个信号与可中断系统调用，信号备用栈（SigAltStack）未完全实现。
- **IO 多路复用**：约 60%。实现了 ppoll 和 pselect，缺失 epoll 机制。
- **时间管理**：约 80%。覆盖了主要的时钟与定时器系统调用。
- **硬件抽象层**：约 90%。双架构的启动、中断、内存与陷阱抽象高度完整。

## 四、 各子系统优缺点及实现细节

### 1. 调度子系统
- **实现细节**：基于 `async_task` 构建协程运行时，将用户任务包装为 `UserTaskFuture`。采用 `MultiLevelScheduler`，包含实时队列（FIFO）和普通队列（O(1) 风格的 current/expire 双队列）。
- **优点**：异步抽象使得 IO 等待与调度自然结合，避免了复杂的回调与状态机管理；零成本抽象保证了调度路径的低开销。
- **缺点**：代码中包含了完整的 CFS（完全公平调度器）实现，但被明确标记为废弃且未投入使用；多核负载均衡代码存在但自评性能极差，未实际完善。

### 2. 进程与任务管理
- **实现细节**：`Task` 结构体按并发访问模式将字段划分为 `Mutable`（锁保护）、`ThreadOnly`（无锁）、`Immutable` 和 `SharedMut`。完整实现了 `clone` 的各类标志位（如 `CLONE_VM`、`CLONE_VFORK`）。
- **优点**：数据结构的并发设计严谨，有效降低了锁竞争；Futex 实现区分了私有与共享队列，支持 `FUTEX_WAIT`、`FUTEX_WAKE` 等核心操作。
- **缺点**：部分高级进程控制功能（如复杂的资源限制 enforcement）实现较为基础。

### 3. 内存管理
- **实现细节**：`MemorySet` 管理地址空间，支持 SV39 与 LoongArch 4 级页表。通过 `MappingFlags::COW` 实现写时复制，在 `StorePageFault` 时触发物理页复制。栈、堆、mmap 均支持懒分配。
- **优点**：用户指针校验（`UserPtr`）封装了安全的异步跨空间内存访问；mmap 支持文件映射并在 `Drop` 时自动回写。
- **缺点**：`msync` 系统调用标记为 `unimplemented!`，缺乏显式的内存同步机制；物理帧分配器使用全局自旋锁，在多核高并发下可能成为瓶颈。

### 4. 文件系统
- **实现细节**：VFS 采用 `Dentry`、`Inode`、`File`、`SuperBlock` 等 trait 抽象。页缓存采用 MSI（Modified-Shared-Invalid）状态协议，块缓存采用 LRU 策略。
- **优点**：文件系统种类丰富（5 种），ProcFS 提供了详细的进程与系统信息；管道实现基于物理帧环形缓冲区，支持异步等待与 poll。
- **缺点**：`sync`、`fsync`、`fdatasync` 等确保数据落盘的系统调用为空实现，存在数据丢失风险；页缓存清理策略较为简单（超限全量遍历）。

### 5. 网络子系统
- **实现细节**：基于 `smoltcp` 协议栈，通过 `SocketFile` 将网络套接字适配至 VFS `File` trait，使其支持标准的文件描述符操作。
- **优点**：网络 IO 与内核异步调度器无缝集成，避免了阻塞式网络等待；支持 IPv4/IPv6 及丰富的 Socket API。
- **缺点**：缺失 `epoll` 导致高并发网络 IO 多路复用效率受限；Unix Domain Socket 标记为 `todo!`，不支持本地进程间高效网络通信。

### 6. 信号与中断处理
- **实现细节**：用户态陷阱处理函数（`user_trap_handler`）本身是 `async` 的，允许在异常处理中直接进行异步 IO。信号传递通过修改用户态上下文（EPC/RA）并构建 `UContext` 实现。
- **优点**：异步陷阱处理是该架构的一大亮点，简化了缺页异常等需要等待 IO 的处理流程；支持 `SA_RESTART` 自动重启被信号中断的系统调用。
- **缺点**：实时信号未实现严格的排队机制；信号备用栈（`sigaltstack`）功能未完善。

## 五、 OS 内核整体实现完整度

NoAxiom-OS 内核整体实现完整度较高。从代码规模来看，包含 221 个内核源文件与 135 个内部库源文件。从接口覆盖来看，实现了 115 个系统调用，涵盖了文件操作（约 45 个）、进程管理（约 20 个）、网络（约 18 个）、内存（约 10 个）等核心领域。内核具备运行 BusyBox 及复杂用户态测试程序的能力，整体功能完整度以通用宏内核为基准评估约为 80%。

## 八、 总结评价

NoAxiom-OS 是一个架构设计极具创新性且工程质量较高的 Rust 宏内核项目。其最大的技术亮点在于将 Rust 的无栈协程（`async/await`）机制深度融入内核调度与 IO 路径，使得系统调用、页面错误处理及驱动交互均能以同步的代码风格实现异步的非阻塞执行。这种设计不仅大幅降低了内核状态管理的复杂度，还在 IO 密集型场景下展现了卓越的性能。

在功能实现上，该项目展现了较高的完整度，双架构支持、5 种文件系统、完整的进程克隆语义以及 115 个系统调用的实现，证明了团队扎实的系统编程能力。代码结构模块化清晰，并发数据模型设计严谨。

主要的改进空间在于：部分高级功能（如 CFS 调度、epoll、msync、fsync）处于废弃、未实现或空实现状态；多核环境下的负载均衡与全局锁优化仍有提升潜力。总体而言，NoAxiom-OS 在架构探索与性能优化方面表现突出，是一个优秀的 OS 内核竞赛作品。
