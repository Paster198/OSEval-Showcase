## 三、各子系统详细分析

### 3.1 内存管理

**实现完整度：较高**

该子系统实现了 Sv39 三级页表管理、物理页分配器（带引用计数）、COW（Copy-on-Write）和惰性分配（Lazy Allocation）。

**优点：**

1. **COW 实现成熟**：使用 RSW 位（PTE_COW）标记写时复制页，在 `uvmcopy()` 中将可写页共享给父子进程。页面错误处理 `uvm_handle_page_fault()` 在触发写操作时检查引用计数，若为 1 则直接恢复可写，否则分配新页。该设计避免了不必要的内存复制。

2. **双页表架构**：每个进程维护 `pagetable`（用户态）和 `kpagetable`（内核态副本），内核可在进程上下文直接访问用户内存。`copyin2()`/`copyout2()` 中通过 `ensure_user_page()` 透明处理 COW 和惰性分配的缺页异常，避免了传统 xv6 中临时切换页表的开销。

3. **引用计数集成**：`kalloc.c` 中的 `kaddref()`/`kgetref()`/`kfree()` 通过空闲链表加引用计数数组管理物理页生命周期，为 COW 提供基础支撑。

**缺点：**

1. **无页面替换**：`vm.c` 中未实现任何页面替换算法（FIFO/LRU），系统在物理内存耗尽时将直接返回分配失败，无换页能力。

2. **惰性分配范围受限**：`lazy_alloc_page()` 仅检查 `va >= PGROUNDUP(p->brk)` 且 `va < PGROUNDDOWN(p->trapframe->sp)`，不覆盖 mmap 区域（mmap 实现通过 `growproc` 直接分配）。

3. **sbrk 无回收**：`growproc()` 仅支持向上增长堆，`brk` 系统调用不允许收缩地址空间。

**关键实现细节：**

- 物理页分配器使用全局空闲链表 `kmem.freelist` 加自旋锁保护
- 引用计数索引通过 `pa2idx(pa)` 基于物理地址偏移计算
- COW 判断条件：`scause == 15`（Store page fault）且 PTE_COW 置位
- 页表遍历使用递归 `freewalk()` 释放页表自身物理页

### 3.2 进程管理

**实现完整度：较高**

该子系统实现了完整的进程生命周期管理（创建、执行、等待、退出）、三种调度算法和上下文切换。

**优点：**

1. **调度器丰富**：实现了 RR（时间片轮转）、Priority（优先权调度）和 MLFQ（多级反馈队列）三种算法，通过全局变量 `sched_algo` 切换。MLFQ 动态调整基于 CPU 密集/IO 密集启发式评估，每 `MLFQ_WINDOW=5` 个 tick 更新一次。

2. **fork/clone 支持**：`fork_at()` 实现通用含义的进程创建，`clone_proc()` 允许指定子进程栈指针，为 Linux ABI `clone` 系统调用提供基础。

3. **exec 的 Linux 兼容增强**：`exec.c` 构建 auxv 向量（AT_PHDR、AT_PHENT、AT_PHNUM、AT_PAGESZ、AT_ENTRY、AT_PLATFORM、AT_RANDOM、AT_EXECFN），支持 glibc/musl 动态链接器。

4. **测试框架内嵌**：init 进程的 `exit()` 路径中嵌入测试状态机，按序执行 basic/lua/busybox/libctest 等多组测试用例，自动推进。

**缺点：**

1. **进程数量固定**：进程表 `proc[NPROC]` 为静态数组（NPROC=50），无法动态扩展。

2. **无优先级继承**：优先级调度和 MLFQ 均不支持优先级继承，可能在锁竞争场景下导致优先级反转。

3. **调度算法全局化**：`sched_algo` 为全局变量，影响所有进程，不支持按进程组或进程独立配置调度策略。

4. **MLFQ 启发式粗糙**：仅使用 `mlfq_cpu_ticks` 和 `mlfq_sleep_ticks` 两个计数器，无更细粒度的 I/O 等待时间或历史行为建模。

**关键实现细节：**

- `swtch.S` 保存/恢复 14 个 callee-saved 寄存器（ra, sp, s0-s11）
- CPU 结构 `cpus[NCPU]` 中 NCPU=2
- RR 使用 `rr_cursor` 循环游标避免饥饿
- 时间片可通过 `set_timeslice` 系统调用配置

### 3.3 文件系统

**实现完整度：中等**

该子系统通过 VFS 层统一 FAT32（读/写）和 EXT4（只读）两种文件系统。

**优点：**

1. **VFS 调度干净**：`fat32.c` 中的每个 VFS 入口函数（`eread`、`ewrite`、`ename`、`ealloc`、`eremove` 等）在 `ext4_active()` 时优先调用 EXT4 实现，否则走 FAT32 路径，不侵入上层调用代码。

2. **FAT32 实现完整**：支持 BPB 解析、FAT 表遍历、簇分配/释放、短文件名和 VFAT LFN、目录创建/删除、文件读写和截断。

3. **EXT4 extent 树解析**：`ext4.c` 递归读取 extent 索引/叶节点，支持深度大于 0 的 extent 树，从超级块到 inode 到 extent 树的完整读取路径。

4. **设备抽象成熟**：`/dev/null`（读返回 0，写丢弃）、`/dev/zero`（读返回零字节，写丢弃）通过 `devsw` 数组统一分发。

**缺点：**

1. **EXT4 只读**：`ext4.c` 无任何写入操作实现（除内存虚拟文件外），`ext4_ealloc`、`ext4_eremove` 等函数未提供写入能力，实际仅为预录制基准输出提供数据读取。

2. **无日志支持**：FAT32 和 EXT4 均无日志或事务机制，异常断电时文件系统完整性无法保证。

3. **缓冲区缓存策略简单**：LRU 回收使用双向链表，但缺乏脏页写回策略或周期性同步。

4. **EXT4 虚拟文件硬编码**：`ext4.c` 中为 `test_mmap.txt` 和 `test_chdir`/`test_mkdir` 等提供内存中虚拟文件，这些文件在真实 EXT4 镜像中不存在，是一种测试适配手段。

**关键实现细节：**

- 缓冲区缓存每 buffer 有睡眠锁保护
- FAT32 簇分配使用 `read_fat()`/`write_fat()` 操作 FAT 表
- EXT4 魔数验证 `0xEF53`，块大小、inode 大小从超级块动态读取

### 3.4 系统调用

**实现完整度：中等偏上**

该子系统实现了 26 个 xv6 原生系统调用和 50+ Linux RISC-V 系统调用的兼容层。

**优点：**

1. **双层分发架构**：`syscall()` 先通过 `is_linux_abi_proc()` 判断进程类型，优先走 `linux_dispatch()` 映射，回退到 xv6 系统调用表，非 Linux 进程也可尝试 Linux 系统调用作为最后手段。

2. **Linux ABI 覆盖广**：文件操作（openat/read/write/readv/writev/pread64/lseek/getdents64）、文件系统（mkdirat/unlinkat/mount/umount2/statfs）、进程管理（clone/execve/exit/wait4）、内存管理（brk/mmap/munmap/mprotect）等核心调用均有实现。

3. **返回格式转换**：`linux_sys_wait4()` 将 xv6 的退出状态码重排为 Linux 格式（状态码左移 8 位），确保测试二进制能正确解释子进程退出状态。

**缺点：**

1. **进程名匹配脆弱**：`is_linux_abi_proc()` 通过字符串比较进程名与已知列表（"times", "execve", "busybox", "lua", "runtest.exe" 等），无法处理未预注册的 Linux 二进制。

2. **信号系统调用为存根**：`rt_sigaction`、`rt_sigprocmask`、`rt_sigtimedwait` 等均返回 0 或硬编码值，无实际信号处理能力。

3. **mmap 语义非标准**：`linux_sys_mmap()` 通过 `growproc()` 扩展进程大小，对文件描述符有效时读取文件内容，不支持 MAP_SHARED/MAP_PRIVATE 的完整语义，无页面级保护。

4. **部分系统调用特殊处理**：`statfs` 返回硬编码 EXT4 超级块信息，`getrandom` 可能返回伪随机值，并非通用实现。

**关键实现细节：**

- Linux 系统调用号定义在 `sysnum.h`
- 每个 Linux 系统调用由独立的 `linux_sys_*()` 函数实现
- 参数通过 `argraw()` 从 trapframe 中提取（`a0`-`a5`）

### 3.5 中断与异常处理

**实现完整度：较高**

该子系统实现了内核态和用户态的完整 trap 路径，支持多平台设备中断分发。

**优点：**

1. **trap 路径清晰**：内核态通过 `kernelvec.S` 保存全部 32 个通用寄存器，调用 `kerneltrap()` 处理设备中断或定时器中断后 `sret` 返回。用户态通过蹦床页（TRAMPOLINE）实现 `uservec` → `usertrap()` → `usertrapret()` → `userret` 的完整往返。

2. **页面错误集成处理**：`usertrap()` 中 `scause==13`（Load page fault）或 `scause==15`（Store page fault）统一调用 `uvm_handle_page_fault()` 处理 COW 和惰性分配，用户态无感知。

3. **多平台中断分发**：`devintr()` 通过编译时条件（`#ifdef QEMU`/K210/VF2）适配不同平台的中断触发方式。

**缺点：**

1. **无 NMI 支持**：`kernelvec.S` 和 `trampoline.S` 未处理不可屏蔽中断。

2. **K210 中断路径特殊**：使用 supervisor software interrupt（`scause` bit63 + 异常码 1 + `stval=9`）模拟外部中断，与标准 PLIC 路径不一致。

3. **定时器分辨率固定**：`INTERVAL = 390000000 / 2000`（约 200Hz），不支持高精度定时器。

**关键实现细节：**

- 蹦床页在内核和用户空间映射到相同虚拟地址
- `sscratch` CSR 用于在用户态和内核态之间交换 trapframe 地址和内核栈指针
- 定时器使用 SBI `set_timer` 接口

### 3.6 同步原语

**实现完整度：较高**

该子系统实现了自旋锁和睡眠锁两种同步机制。

**优点：**

1. **自旋锁实现规范**：使用 GCC 原子内置函数 `__sync_lock_test_and_set`/`__sync_lock_release`，配合 `push_off()`/`pop_off()` 嵌套中断禁用，避免死锁。

2. **睡眠锁实用**：基于自旋锁加睡眠/唤醒机制，适用于可能长时间持有锁的场景（如文件系统操作）。

3. **调试辅助**：`holding()` 函数在持有锁时断言当前 CPU 确实持有该锁，有助于发现锁使用错误。

**缺点：**

1. **无更高级同步原语**：缺少读写锁、信号量、条件变量或 RCU 等机制。

2. **无死锁检测**：自旋锁无超时机制或死锁检测能力。

**关键实现细节：**

- 自旋锁包含 `locked` 字段、持有 CPU 信息、中断禁用计数
- 睡眠锁包含自旋锁和 `locked` 标志

### 3.7 时间管理

**实现完整度：中等**

该子系统通过全局 `ticks` 计数器提供基本时间服务。

**优点：**

1. **时间接口完整**：`sleep()`、`uptime()`、`wait()` 等系统调用均基于 `ticks` 实现。

2. **定时器中断处理直接**：`timer_tick()` 递增 `ticks`，调用 `wakeup(&ticks)` 唤醒等待进程，然后设置下一次超时。

**缺点：**

1. **无高精度时间**：缺少微秒/纳秒级定时器接口，`nanosleep` 实现基于 tick 转换。

2. **无 RTC 支持**：系统时间从启动开始计数，无实时时钟（RTC）读取或设置能力。

3. **`gettimeofday` 为存根**：返回固定值或启动秒数，无真实的时间戳。

### 3.8 系统信息

**实现完整度：较低**

该系统调用提供基本系统标识和统计信息。

**优点：**

1. **`uname` 返回标准字段**：sysname、nodename、release、version、machine 均有填充。

2. **`sysinfo` 返回内存统计**：`kallocated_pages()` 计算已用物理页数。

**缺点：**

1. **无 CPU 信息**：缺少 CPU 数量、频率、型号等信息的查询接口。

2. **无挂载表**：`mount` 系统调用为空操作，无挂载信息查询。

3. **`syslog` 为存根**：返回固定值 0。

### 3.9 资源管理

**实现完整度：中等**

该子系统通过文件描述符表和进程表管理资源。

**优点：**

1. **文件描述符引用计数**：`filedup()` 在 fork/clone 时增加引用计数，`fileclose()` 在引用归零时释放。

2. **进程资源回收完整**：`exit()` 关闭所有文件描述符，`freeproc()` 释放页表、内核栈和 trapframe。

**缺点：**

1. **无资源限制**：无 `rlimit`/`prlimit64` 的实际实现（存根），无法限制进程的文件描述符数量、内存使用或 CPU 时间。

2. **无配额机制**：文件系统无磁盘配额。

3. **物理内存无预留**：无法为特定进程预留物理页。

### 3.10 交互设计

**实现完整度：中等**

该子系统通过控制台层提供基本的用户交互。

**优点：**

1. **控制台编辑功能**：支持 Backspace（^H）、行删除（^U）、EOF（^D）、进程列表（^P），使用 128 字节环形缓冲区。

2. **串口多平台支持**：QEMU 使用 SBI 控制台，K210 使用 UARTHS，VF2 使用独立 UART 驱动。

**缺点：**

1. **无终端控制**：不支持 ANSI 转义序列、光标移动或颜色输出。

2. **无输入历史**：控制台无命令历史记录功能。

3. **用户态 Shell 简单**：`sh.c` 仅支持基本命令执行、重定向和管道，无脚本能力。

### 3.11 LoongArch64 探针

**实现完整度：极低**

LoongArch64 侧不是真实的操作系统内核，而是一个内核探针。

**实现的功能：**

- 早期启动（CSR 初始化、串口配置）
- 受控异常处理（仅处理 PLV3 下的 write(64) 和 exit(93)）
- PLV3 用户态入口（用于遍历预录制输出数据段）
- UART 输出（16550 兼容，地址 0x1FE001E0）

**未实现的功能：**

- 进程管理、内存管理、文件系统、中断处理（中断保持禁用）、调度器、页表/MMU 配置

**预录制输出机制：**

- 22 个数据段（1 条欢迎消息 + 21 组基准输出）
- 用户态探针 `la_user_probe` 遍历段表，通过 `write` 系统调用逐段打印
- 数据来自 `la-minimal/*_output.txt` 文件，嵌入在 `basic_output.S` 中

## 四、OS 内核整体实现完整度

**RISC-V64 内核**：约 65%，实现了进程管理、内存管理、文件系统、系统调用、中断处理、同步原语和设备驱动等核心子系统，具备运行多用户进程和竞赛公开测试二进制的能力。主要不足在于无网络栈、EXT4 只读、无页面替换、信号子系统为存根。

**LoongArch64 探针**：约 3%，仅实现启动、异常处理骨架和 UART 输出。无任何真实内核功能。

## 五、动态测试设计与结果

### 5.1 RISC-V64 侧测试设计

项目包含以下测试机制：

1. **init 进程内嵌测试框架**：`proc.c` 中维护测试用例状态机，按序执行多组测试（basic/lua/busybox/libctest，各含 glibc 和 musl 版本），在 `exit()` 路径中自动推进。

2. **用户态回归测试**：`xv6-user/usertests.c` 提供标准 xv6 回归测试套件。

3. **专项测试程序**：
   - `test_mem_cow` — COW 行为验证
   - `test_mem_lazy_allocation` — 惰性分配验证
   - `test_proc_rr` / `test_proc_priority` / `test_proc_mlfq` — 调度算法验证
   - `test_ipc_producer_consumer` / `test_ipc_philosopher` — IPC 验证
   - `test_vm_fifo` / `test_vm_lru` — 页面替换测试桩

4. **公共基准测试**：通过 `rv_public_output.c` 和 `rv_public_output_data.S` 输出预录制基准测试结果（libctest、lmbench、libcbench、iozone、ltp、iperf、netperf、cyclictest，各含 glibc/musl 版本，共 14 组）。

### 5.2 LoongArch64 侧测试设计

LoongArch64 侧无动态测试能力。所有基准测试输出均为预录制数据，通过 PLV3 用户探针逐段打印。

## 七、总结评价

Oblivion 是一个面向 OS 内核赛道的双架构提交作品。其核心价值集中于 RISC-V64 侧——一个基于 xv6-riscv 框架的深度扩展内核，具备以下竞争力特征：

**主要优势**：

1. **内存管理设计精巧**：COW 和惰性分配的实现完整且经过良好的错误路径集成（通过 `uvm_handle_page_fault()` 统一处理），双页表架构使 `copyin/copyout` 操作能透明处理缺页异常，避免了传统的临时页表切换开销。

2. **调度器多样性**：三种可全局切换的调度算法（RR/Priority/MLFQ）提供了丰富的调度策略，MLFQ 的动态优先级调整虽然是简单的 CPU/IO 密集启发式，但足以展示调度器可扩展性。

3. **Linux ABI 兼容层务实有效**：通过进程名匹配实现的 50+ Linux 系统调用兼容层覆盖了竞赛公开测试二进制（Busybox、Lua、libc-test）的系统调用需求，尽管在通用性上存在局限。

4. **文件系统 VFS 集成干净**：FAT32+EXT4 双文件系统通过统一 VFS 接口无缝切换，EXT4 extent 树解析实现到位。

**主要不足**：

1. **LoongArch64 侧非真实内核**：该侧的"内核"实际上是一个预录制输出探针，不具备进程管理、内存管理、文件系统或中断处理能力。项目文档对此坦率承认，但评分价值取决于竞赛规则对双架构提交的评判标准。

2. **核心功能缺失**：无网络子系统、EXT4 仅支持只读、信号机制完全为存根、无页面替换算法。这些缺失意味着内核在面对需要真正功能完整性的隐藏测试时将面临严重挑战。

3. **部分设计选择偏竞赛化**：Linux ABI 进程检测通过名字符串匹配、预录制基准测试输出、EXT4 虚拟文件硬编码等设计在通用场景下不可用，但在竞赛语境中具有明确的目的性。

**综合评估**：

Oblivion 的 RISC-V64 内核体现了将教学框架（xv6-riscv）扩展至可运行标准化测试二进制（glibc/musl 工具链编译的 BusyBox、Lua 和 libc-test）的工程能力。COW 内存管理、MLFQ 调度器和 Linux ABI 兼容层是三个值得关注的实现点。但 LoongArch64 侧的探针设计以及若干核心子系统的存根化处理，表明该项目在竞赛功能的广度覆盖与内核功能的深度实现之间存在策略性的取舍。作为一个竞赛提交作品，其技术得分基于预录制输出的基线测试而非运行时功能验证，这在竞赛规则的"公共基准测试兼容 runner"定义下构成一个需要规则制定者审慎评判的设计范式。
