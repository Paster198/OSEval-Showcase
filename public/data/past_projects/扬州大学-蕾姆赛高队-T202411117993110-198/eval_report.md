# REMOS 内核项目技术画像与评估报告

## 一、 项目基本信息

- **项目名称**：REMOS
- **目标架构**：RISC-V 64位 (RV64)
- **实现语言**：C语言（含少量汇编语言）
- **生态归属**：基于开源教学操作系统内核框架深度改造，集成轻量级 EXT4 文件系统库
- **代码规模**：总计约 49,200 行代码（其中 EXT4 文件系统移植代码约 18,047 行，自主编写及改造的内核代码约 13,000 行）
- **核心特点**：引入 Buddy 物理内存分配器与 Per-process 内核页表、支持 EXT4 文件系统、实现约 50 个系统调用、支持多核 SMP 启动。

## 二、 子系统与功能实现概述

- **内核启动与多核初始化**：支持 SBI HSM 扩展的多核启动，完成内存、中断、设备初始化。
- **中断与异常处理**：实现用户态与内核态陷阱分发，支持 PLIC 外部中断与定时器中断。
- **进程与线程管理**：实现进程创建、执行、等待、退出，以及基于标志位的线程创建（简化实现）。
- **内存管理**：实现 Buddy 物理内存分配、Sv39 三级页表虚拟内存管理、mmap/munmap 内存映射。
- **文件系统**：实现 EXT4 文件系统（含日志、Extent、目录索引）、FAT32 文件系统（遗留）、LRU 缓冲区缓存、VFS 抽象层。
- **系统调用接口**：实现约 50 个系统调用，覆盖进程、文件、内存、时间等类别。
- **信号机制**：定义标准与实时信号，实现信号掩码与处理函数注册（部分实现）。
- **设备驱动**：实现 VirtIO 块设备驱动、SBI 控制台驱动、VisionFive 平台 SD 卡/SPI 驱动。
- **同步机制**：实现自旋锁、睡眠锁、管道。

## 三、 子系统实现完整度与详细分析

*注：以下完整度百分比以“通用宏内核基本功能集及 OS 竞赛标准”为 100% 基准。*

### 1. 内核启动与核心初始化 (完整度：85%)
- **实现细节**：通过 OpenSBI 将控制权转移至入口汇编，为每个 hart 分配 16KB 独立栈。hart 0 执行完整初始化序列，其他 hart 通过 SBI HSM 扩展启动。链接脚本将内核加载于 `0x80200000`，段 4KB 对齐。
- **优点**：多核启动流程规范，利用 SBI HSM 扩展实现多核管理，初始化顺序逻辑清晰。
- **缺点**：BSS 段未显式清零，依赖固件行为；`boot_stack` 总大小仅 32KB（2核），在多核高并发场景下存在栈溢出风险。

### 2. 中断与异常处理 (完整度：80%)
- **实现细节**：区分用户态（trampoline）与内核态（kernelvec）陷阱路径。定时器中断间隔设为 1,950,000 个时钟周期（约 5ms，200Hz）。PLIC 配置 UART 与 VirtIO 中断优先级。在陷阱处理中记录用户态与内核态时间戳。
- **优点**：Trampoline 机制正确实现用户/内核态隔离与切换，时间统计功能为 `times` 系统调用提供支撑。
- **缺点**：内核态异常处理过于简单，遇到未预期异常直接触发 panic，不支持缺页异常等可恢复异常的处理。

### 3. 进程管理 (完整度：70%)
- **实现细节**：最大进程数 50，支持 5 种状态。实现 fork、exec、waitpid、exit。调度器采用轮转调度（Round-Robin）。`clone` 系统调用支持 `CLONE_VM`、`CLONE_FILES` 等标志位，但内部实现简化为创建新进程并选择性共享资源。
- **优点**：进程生命周期管理基本完整，exec 正确解析 ELF 并处理内核页表切换，waitpid 支持 `WNOHANG` 选项。
- **缺点**：调度器缺乏优先级与时间片概念；线程支持为半成品，未使用线程链表，`gettid` 硬编码返回 0。

### 4. 内存管理 (完整度：75%)
- **实现细节**：物理内存采用 Buddy 分配算法，最大 order 为 13（32MB），划分为每 CPU 独立的内存池，支持跨池窃取。虚拟内存采用 Sv39 页表，引入 Per-process 内核页表，将用户内存映射至内核空间（无 `PTE_U` 标志）。实现 `do_mmap`/`do_munmap`，支持文件映射与段记录管理。
- **优点**：Buddy 分配器支持大块内存分配与合并，Per-process 内核页表避免了 `copyin`/`copyout` 时的页表遍历，提升内存访问效率。
- **缺点**：mmap 的匿名映射未实际分配物理页；`brk` 系统调用仅支持增大不支持缩小；Copy-on-Write 仅预留引用计数接口，未实现核心逻辑。

### 5. 文件系统 (完整度：80%)
- **实现细节**：核心为 EXT4 文件系统，包含高层操作、Extent 管理、JBD2 日志、htree 目录索引等 23 个源文件。通过适配层桥接至内核 VFS。块设备接口固定为 512 字节。保留 FAT32 遗留代码。缓冲区缓存采用 LRU 双向链表，容量 30 个块。
- **优点**：EXT4 子系统功能覆盖全面，支持日志恢复与复杂目录索引，具备企业级文件系统的基本特征。
- **缺点**：`blockdev_bwrite` 中未调用 `bwrite` 写回磁盘，存在数据丢失风险；`filekstat` 硬编码返回 0，`fstat` 无法获取有效状态；块大小固定 512 字节限制了 I/O 性能。

### 6. 系统调用 (完整度：65%)
- **实现细节**：通过函数指针数组分发，注册约 50 个系统调用。包含针对不存在路径（如 `/etc/localtime`）返回伪文件的兼容性处理。实现 `writev` 向量写与 `fcntl` 文件控制。
- **优点**：接口数量丰富，覆盖了主流用户态程序运行的基本需求，向量 I/O 与文件控制功能完整。
- **缺点**：多个调用为 stub 或硬编码实现（如 `getcwd` 返回 `"/"`，`ppoll` 直接返回 `nfds`）；`nanosleep` 时间单位计算错误；`clock_gettime` 仅支持 `CLOCK_REALTIME`。

### 7. 信号机制 (完整度：30%)
- **实现细节**：定义 31 个标准信号与 32 个实时信号。实现 `sigaction` 与 `sigprocmask`。`sighandle` 仅处理 `SIGALRM`，通过修改 trapframe 的 epc 跳转至处理函数。
- **优点**：信号数据结构定义完整，掩码操作支持三种基本模式。
- **缺点**：仅支持 `SIGALRM` 一种信号的实际递送；`rt_sigreturn` 未注册；`sighandle` 未接入 `usertrapret` 导致信号无法实际触发；`sigprocmask` 中阻止不可屏蔽信号的位运算存在逻辑错误。

### 8. 设备驱动与同步机制 (完整度：85%)
- **实现细节**：VirtIO 块设备驱动基于描述符环与中断通知实现。控制台驱动通过 SBI 调用实现字符 I/O 与行编辑。同步机制包含基于原子操作的自旋锁（支持中断状态保存）与基于 sleep/wakeup 的睡眠锁。
- **优点**：驱动与同步原语实现规范，自旋锁的 `push_off`/`pop_off` 机制有效防止了中断死锁，管道支持向量读写。
- **缺点**：VisionFive 平台相关驱动（SPI/SD卡）未在 QEMU 环境中验证；控制台输入缓冲区较小（128字节）。

## 四、 OS内核整体实现完整度

以“通用宏内核基本功能集及 OS 竞赛标准”为 100% 基准，REMOS 内核的整体实现完整度约为 **72%**。
项目在文件系统与物理内存分配方面达到了较高的工程实现水平，但在进程高级调度、线程管理、信号递送机制以及部分系统调用的严谨性上存在明显缺失。内核具备运行基础用户态程序的能力，但距离稳定运行复杂多任务环境仍有差距。

## 五、 动态测试设计与结果

### 1. 编译测试
- **设计**：使用 RISC-V 交叉编译工具链执行 Make 构建。
- **结果**：原始 Makefile 缺失标准 C 库头文件路径，添加 `-I/usr/riscv64-linux-gnu/include` 后编译成功。编译过程产生若干隐式函数声明警告（如 `push_off`、`kcalloc`），最终生成约 1.1 MB 的 `kernel-qemu` ELF 文件。

### 2. QEMU 启动测试
- **设计**：使用 QEMU 8.2.2 (riscv64) 配合 OpenSBI v1.3，配置 2 核、128MB 内存、VirtIO 块设备启动内核。
- **结果**：内核成功完成所有核心初始化阶段，包括 Buddy 内存分配器初始化（每核 15104 页）、内核页表创建、定时器与 PLIC 配置、VirtIO 设备探测、EXT4 文件系统挂载与日志恢复。随后因磁盘镜像中缺失测试用例文件（`time-test`、`busybox`）而停止执行用户程序。启动流程稳定，无内核崩溃或死锁现象。

## 七、 总结评价

REMOS 是一个面向 RISC-V 架构的宏内核操作系统项目。该项目的核心工程价值在于将复杂的 EXT4 文件系统与 Buddy 物理内存分配器成功集成至基础教学内核框架中，并引入了 Per-process 内核页表以优化内存访问路径。其代码规模达到约 49,200 行，展现了较强的工程集成与代码改造能力。

在优势方面，项目的文件系统子系统功能覆盖全面，具备日志恢复与高级索引能力；内存分配器设计合理，多核环境下的锁竞争控制较好；同步原语与中断处理机制实现严谨，保障了内核底层运行的稳定性。

在不足方面，项目在进程高级特性（如多线程实质支持、高级调度）、信号机制递送逻辑以及部分系统调用的严谨性上存在明显短板。多个系统调用采用硬编码或 Stub 实现，信号掩码运算与块设备写回逻辑存在事实性 Bug，导致内核在运行复杂多任务或进行高强度 I/O 时存在可靠性风险。

总体而言，REMOS 在文件系统与内存管理维度达到了较高的实现水平，但在系统调用完备性、信号机制与线程管理等维度仍需深化。该项目具备运行基础用户态程序的能力，是一个具有显著工程工作量且在特定子系统上表现突出的竞赛级内核作品。
