# yungekc OS 内核技术画像与评估报告

## 1. 项目基本信息

- **项目名称**：yungekc (云客)
- **架构**：RISC-V 64、LoongArch 64 双架构
- **实现语言**：C、汇编（RISC-V / LoongArch）
- **代码规模**：约 72,860 行（209 个源文件）
- **生态归属**：面向 Linux ABI 兼容的自研宏内核，遵循 Linux RISC-V/LoongArch 系统调用规范
- **构建工具**：GCC、GNU Make、自建链接脚本
- **应用领域**：OS 内核实现赛道（2026 年全国大学生计算机系统能力大赛），目标是运行 LTP、lmbench、iozone、libc-test 等测试套件
- **项目定位**：SC7 项目的升级版本，强调 SC7 兼容入口与独立入口双模式
- **架构层次**：HAL（硬件抽象层）→ HSAI（架构无关陷阱/中断/平台服务层）→ Kernel（内核逻辑层）

## 2. 子系统与功能概览

该项目实现了操作系统的核心子系统，主要覆盖：

- **启动与多核管理**：BSS 清零、栈初始化、设备树物理内存探测、多核乱序启动协调、定时器与中断控制器初始化。
- **中断与异常处理**：完整的用户态/内核态 trap 处理，缺页异常按需分页，信号检视点集成，PLIC 中断控制器驱动。
- **系统调用层**：144 个系统调用，涵盖进程、文件、目录、内存、信号、时钟、同步、网络、IPC 及系统信息等类别，严格遵循 Linux ABI 调用号。
- **进程管理**：六状态进程模型，进程池静态分配，PID 分配，线程管理（独立线程池），fork/clone/clone3/execve 完整流程，轮询调度器。
- **内存管理**：伙伴系统物理分配器 + Slab 分配器双层架构，SV39/LA64 四级页表，VMA 环形双向链表管理虚拟地址空间，mmap/munmap/mprotect/mremap 及共享内存，缺页按需加载与懒分配策略。
- **文件系统**：VFS 层 + ext4 完整实现（含 extent 树、HTree 目录索引、JBD2 日志、扩展属性），另支持 VFAT 适配层；提供管道、FIFO、块 I/O 缓冲、块设备抽象、loop 设备接口。
- **设备驱动**：RISC-V VirtIO MMIO 块设备，LoongArch VirtIO-PCI 块设备（含 PCI 枚举与 BAR 分配），NS16550/LS7A UART 驱动。
- **同步机制**：自旋锁（基于原子指令）、睡眠锁（sleep/wakeup）、futex 系统调用（支持 FUTEX_WAIT/WAKE/BITSET/REQUEUE、futex_waitv 及 robust list）。
- **信号处理**：支持标准信号与实时信号，每线程独立信号掩码/待处理/处理器，sigaction/sigprocmask/sigreturn 完整流程，双架构 sigtrampoline，可中断系统调用。
- **时钟管理**：RISC-V mtimecmp/SBI 定时器及 LoongArch 倒计时定时器，tick 计数、唤醒等待进程、获取绝对/相对时间。
- **命名空间**：UTS 命名空间（hostname 隔离），支持引用计数与 clone(CLONE_NEWUTS)。
- **procfs**：支持 /proc/mounts、/proc/meminfo、/proc/<pid>/stat 等节点。
- **网络子系统**：socket 接口框架，但无 TCP/IP 协议栈实现（仅为桩代码）。
- **控制台与服务进程**：行缓冲控制台输入，可选的服务进程输出仲裁模式，/dev/null 与 /dev/zero 设备。

以下完整度以 Linux 内核相应功能集为基准（100% 表示功能完全覆盖该子系统的生产级语义，而非代码成熟度）。

| 子系统 | 完整度 | 基准说明 |
|--------|--------|----------|
| 启动流程 | ~85% | 包含多核启动、设备树内存探测、BSS 清零、栈初始化、定时器与中断设置；缺少完整的 ACPI/设备树外设枚举。 |
| 异常/中断处理 | ~80% | 用户/内核态 trap 分离，缺页处理完整，PLIC 驱动可用；LoongArch 用户态 trap 仍依赖 trampoline。 |

| 信号处理 | ~80% | 标准+实时信号框架，sigaction/sigprocmask/sigreturn、altstack、可中断系统调用均已实现；某些实时信号扩展未覆盖。 |

| 同步机制 | ~80% | spinlock/sleeplock/futex/futex_waitv/robust_list/membarrier 均已实现；适用于用户态与内核态的同步需要。 |
| 网络 | ~10% | 仅有 socket 框架桩，无 TCP/IP 协议栈、无网络设备驱动。 |

## 4. 各子系统优缺点与实现细节

### 4.1 启动流程
- **优点**：RISC-V 与 LoongArch 启动路径清晰分离，M-mode 初始化、委派中断、PMP 配置、DMW 直接映射窗口设置均正确；多核乱序启动通过自旋标志 `started` 实现简单有效的同步。
- **缺点**：未充分利用设备树进行全面硬件枚举（仅解析内存节点），缺少 ACPI 支持；错误处理路径在缺盘时会导致反复缺页直至 panic，鲁棒性不足。
- **细节**：内核 BSS 清零仅由 hart 0 执行；`sc7_start_kernel` 通过 filter-out 与独立入口共存。

### 4.2 中断/异常处理
- **优点**：trampoline 页面与 sscratch 机制实现高效的用户态/内核态 trap 分发；缺页处理涵盖 VMA 查找、权限检查、写时复制预留钩子、文件映射页面加载；PLIC 中断控制器驱动完整。
- **缺点**：LoongArch 用户态 trap 仍依赖 trampoline 页，对 TLB 重填的软件处理可能存在边界情况；缺少对高级异常（如机器检查）的完整处理。
- **细节**：`pagefault_handler` 在 VMA 链表无匹配且地址小于进程地址空间上界时，将地址视为堆扩展并赋予默认权限，该策略可能导致非法访问被错误批准。

### 4.3 系统调用
- **优点**：系统调用号严格遵循 Linux 规范，便于移植用户态工具；144 个调用覆盖数量远超同类项目，为运行标准测试套件奠定基础。
- **缺点**：部分调用（如 sendfile、copy_file_range、splice）仅提供桩实现或极简化版本；缺少对 32 位兼容调用 / 时间结构兼容层的全面处理；网络相关系统调用虽然存在但后端无实际功能。
- **细节**：`sys_execve` 内嵌了特殊路径硬编码（如 `/tmp/hello` 直接替换为 busybox sh），这种临时措施会降低通用性。

### 4.4 进程管理
- **优点**：进程与线程两级模型、六状态机、进程组/会话、UID/GID 与资源限制已实现，接近 Linux 语义；fork 时 VMA 链表深拷贝、文件描述符表引用计数复制等操作逻辑正确。
- **缺点**：fork 依赖 `uvmcopy` 进行完整的页面复制，未启用写时复制，内存开销较大；调度器仅支持轮询，无优先级与公平性策略；`yield` 基于时钟中断触发，时钟 tick 频率单一。
- **细节**：进程池固定 128 个控制块，线程池独立固定大小，合理限制了大赛场景下的资源消耗。

### 4.5 内存管理
- **优点**：伙伴系统与 Slab 双层分配器在物理内存管理中实现了高效的小粒度分配与回收；Buddy System 支持最大 2^11 页的大块分配；VMA 环形链表配合懒分配策略体现了现代内核设计思想。
- **缺点**：COW 未实现，导致 `fork` 时物理页复制开销高，进程创建性能受影响；`pagefault_handler` 中 COW 钩子预留但未激活，存在非匿名映射的写时复制缺陷；内核页表未采用 ASID 或 PCID 加速。
- **细节**：Slab 通过 `simple_alloc` 自举，规避了“元数据分配需要分配器”的循环依赖；Buddy 使用位图标记每页使用状态，合并/分裂算法正确，但未提供内存碎片统计接口。

### 4.6 文件系统
- **优点**：ext4 实现是项目最大亮点，约 22,000 行代码覆盖了 extent 树、HTree 目录索引、JBD2 日志、扩展属性等高级特性，展现了对现代文件系统结构的深入理解；VFS 层通过 `filesystem_op_t` 支持多文件系统扩展。
- **缺点**：JBD2 日志在某些写路径下存在死锁风险（开发者已通过将部分目录操作改为只读模式规避），日志恢复稳健性未充分验证；ext4 块分配器缺乏延迟分配等优化；VFS 缺少全功能 dcache，目录缓存效率受限于 inode 缓存。
- **细节**：`ext4_mkfs.c` 提供了格式化工具，方便构建测试用磁盘镜像；`vfs_ext4.c` 作为 VFS 适配器衔接了 ext4 内部函数与通用文件操作 API。

### 4.7 同步机制
- **优点**：spinlock 实现了中断禁用嵌套跟踪（push_off/pop_off），有助于死锁诊断；futex 支持多等待、位掩码筛选和 robust list，足以支撑 pthread 的大部分同步需求。
- **缺点**：spinlock 在 VirtIO 中断处理中，若磁盘初始化失败，会导致锁状态不一致而 panic（已在动态测试中触发）；缺少读写锁、RCU 等更高级的同步工具。
- **细节**：futex 队列采用静态数组而非动态哈希，埋点数量有限，高负载下可能触及上限。

### 4.8 设备驱动
- **优点**：RISC-V VirtIO MMIO 与 LoongArch VirtIO-PCI 块设备驱动展示了跨总线类型的硬件操作能力；LoongArch PCI 枚举通过 ECAM 扫描与 BAR 探测，较为规范。
- **缺点**：网络设备驱动完全缺失，UART 仅支持轮询模式，无中断驱动；块设备 I/O 请求采用轮询等待中断完成的方式，在 I/O 并发时效率受影响。
- **细节**：VirtIO MMIO 驱动每次 I/O 使用 3 个描述符链，释放锁后自旋等待中断处理程序置标志，此模式在无磁盘镜像时会无限循环。

### 4.9 信号处理
- **优点**：每线程独立信号数据结构，与 Linux 线程语义一致；sigaction 支持包括 SA_SIGINFO 在内的标志；信号处理路径中执行栈权限检查，避免信号帧写入非法区域。
- **缺点**：SA_SIGINFO 标志虽被接受，但 `siginfo_t` 填充不完整，部分字段硬编码；实时信号队列为每个线程一个待处理位图，未实现排队。
- **细节**：双架构的 `sigtrampoline.S` 实现对信号处理函数返回后的上下文恢复至关重要；可中断系统调用（如 clock_nanosleep）在接收到信号后正确返回 EINTR。

### 4.10 其他子系统（时钟、网络、procfs 等）
- **时钟**：RISC-V 使用 mtimecmp 或 SBI 设置定时器，LoongArch 使用倒计时定时器，tick 驱动调度与定时唤醒；获取时间基于启动时间戳 + 时钟计数推算，精度依赖 CLK_FREQ 宏，未利用 RTC 硬件。
- **网络**：仅定义 socket 接口，sendto/recvfrom 等返回空数据，缺乏任何协议栈实现。
- **procfs**：硬编码部分内存信息，不支持动态统计，大部分标准节点缺失。
- **控制台**：服务进程输出模式是一个创新尝试，但依赖于特定的配置宏和内核态服务进程存在，通用性受限。

## 5. 内核整体实现完整度

以 Linux 内核功能集为参照，yungekc 实现了约 **65-70%** 的核心 OS 功能。该项目已经具备一个可运行标准多线程用户程序的宏内核雏形，能够支持 busybox + musl 等轻量级用户态环境，并在文件系统方面达到接近真实操作系统的容量。但关键缺失（COW、网络协议栈、完善的调度器）限制了其作为通用 OS 的能力；部分子系统虽然功能齐全，但鲁棒性和错误处理尚需打磨。

## 6. 动态测试设计与结果

### 6.1 测试设计
测试采用 QEMU RISC-V virt 平台，使用仓库预编译的 ELF 镜像 `kernel-rv`，搭配 OpenSBI 固件，配置为：
- `qemu-system-riscv64 -machine virt -m 256M -nographic -smp 1`

测试目的为验证：
- 内核能否从 OpenSBI 接力启动
- 字符设备初始化与彩色输出
- 物理内存管理伙伴系统初始化
- 后续调用链中各子系统初始化的顺序与异常处理

### 6.2 测试结果
- **启动阶段正常**：OpenSBI 成功加载内核至 `0x80200000`，内核打印彩色 ASCII 启动横幅，26 个字母逐一输出，表明 UART 初始化和字符设备链工作正常。
- **物理内存初始化成功**：伙伴系统报告检测到可用内存范围 `0x81196000` – `0xbf996000`（约 1012 MB），并输出 “pmem init success (buddy system)”。
- **后续初始化失败**：在 `virtio_disk_init` 阶段，因未提供磁盘镜像，VirtIO MMIO 寄存器访问引发连续缺页异常（scause=7 或 5），反复重试后 spinlock 的 `pop_off` 检测到不一致状态并触发 panic。
- **测试结论**：内核的基础启动流程、串口输出、物理内存管理功能得到验证；但异常处理路径在无磁盘场景下的鲁棒性不足，内存访问错误未能被优雅降级，需要增强设备初始化失败时的回退机制。

| 交互设计 | 部分实现。提供控制台行缓冲、特殊按键处理、服务进程输出仲裁模式、/dev/null 和 /dev/zero。缺少 TTY 子系统的完整分层。 | 服务进程输出模式避免了多进程控制台输出交错，是一个实用创新；但控制台依赖编译宏，灵活性不足。 | 基本的用户交互环境已经具备，但在终端抽象与虚拟控制台方面尚显薄弱。 |
| 同步原语 | 已实现，完整度较高。自旋锁、睡眠锁、futex（含等待/唤醒/BITSET/REQUEUE）、futex_waitv、robust list、membarrier 均已支持。 | futex 实现支持高级特性（位掩码、多 futex 等待），可支撑 pthreads 的大部分原语；spinlock 的嵌套深度跟踪有助于诊断；无读写锁、RCU。 | 同步原语覆盖面优于多数大赛项目，但自旋锁的错误恢复机制需要加强。 |
| 资源管理 | 部分实现。文件描述符引用计数、VMA 管理、共享内存段限制、系统资源限制（rlimit）可用；命名空间仅 UTS；无 cgroup、无全功能配额。 | 进程资源限制通过 `rlimits` 数组保存，但大部分限制未在实际分配中强制；VMA 合并逻辑有助于减少元数据开销。 | 基础的资源跟踪已具备，但距离细粒度资源隔离与审计尚有较大差距。 |
| 时间管理 | 已实现基础功能。定时器中断驱动 tick 计数、定时唤醒、绝对/相对时间获取、clock_gettime/clock_nanosleep 等系统调用可用。 | RISC-V 与 LoongArch 分别使用各自的定时器硬件，tick 处理正确；时间获取基于启动时间戳 + 计数器，精度受 CLK_FREQ 宏影响，未使用 RTC。 | 时间子系统能支撑基本的调度与定时需求，但精度和稳定性有待提升。 |
| 系统信息 | 部分实现。uname、sethostname、sysinfo、syslog 等调用部分可用；/proc/mounts、/proc/meminfo、/proc/<pid>/stat 可读。大量 /proc 节点缺失。 | /proc/meminfo 数据为硬编码，无法反映真实内存使用状况；syslog 为桩实现。 | 提供了最低限度的系统信息接口，但远未达到可观测性工具的需求。 |

## 8. 总结评价

yungekc 是一个工程量庞大、架构格局清晰的自研 OS 内核项目。其最突出的成就在于：在约 7.3 万行代码内实现了双 ISA（RISC-V、LoongArch）支持、144 个 Linux 兼容系统调用，以及超过 2.2 万行代码的 ext4 文件系统实现，后者囊括 extent 树、HTree 目录索引及 JBD2 日志等复杂特性，表现出很强的系统编程能力和对文件系统内部机制的深刻理解。三层架构（HAL-HSAI-Kernel）的设计有效解耦了架构相关代码，使得代码主体能够复用，为多平台移植奠定良好基础。

项目中体现的若干设计细节，如服务进程输出仲裁、Slab 自举初始化、VMA 环形哨兵链表、futex_waitv 的多等待支持等，均显示出超出简单课程设计的工程意识。

然而，项目仍存在若干关键不足：

- **写时复制（COW）缺失**，造成 fork 导致的物理内存拷贝开销，影响多进程场景下的性能；
- **网络协议栈完全缺失**，限制了与外界通信的能力；
- **调度器仅为轮询**，无优先级与公平调度策略，不能满足交互或实时场景；
- **部分功能为桩实现或硬编码**（如网络系统调用、/proc 信息、特殊执行路径），削弱了通用性和可观测性；
- **错误处理路径的鲁棒性不足**，在无磁盘等异常情况下容易从缺页循环恶化为不可恢复的 panic。

综合来看，yungekc 在操作系统内核竞技赛道上具备鲜明的技术深度和工程广度，其文件系统与系统调用覆盖已达到可运行标准测试套件的基础条件。后续若能在 COW、调度器、网络接入等方面持续完善，并加强异常恢复与测试覆盖，将有机会成为一个非常出色的学生内核作品。
