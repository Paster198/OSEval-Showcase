# AddddOS 内核项目技术画像与评估报告

## 一、 项目基本信息

- **项目名称**：AddddOS
- **目标架构**：RISC-V 64位 (QEMU virt), LoongArch 64位 (QEMU virt)
- **实现语言**：C（内核主体），汇编（架构相关底层代码）
- **生态归属**：独立开发的操作系统竞赛项目（基于教学内核 xv6 扩展）
- **构建系统**：CMake (>= 3.21) + GNU Make
- **代码规模**：内核代码约 13,244 行（不含第三方 lwext4 库）
- **核心特点**：双指令集架构支持、EXT4 文件系统支持、Linux 兼容系统调用接口、伙伴系统物理内存管理。

## 二、 实现的子系统与功能

该项目实现了操作系统的核心子系统，具体包括：
1. **启动引导子系统**：多核启动框架、架构相关的 MMU 与中断控制器初始化。
2. **内存管理子系统**：物理页分配（伙伴系统）、虚拟内存管理（多级页表）、用户空间内存映射（mmap）。
3. **进程管理子系统**：进程生命周期管理、ELF 加载执行、轮转调度、线程支持（clone）。
4. **文件系统子系统**：VFS 抽象层、EXT4 文件系统实现、虚拟文件系统（/proc, /dev）。
5. **设备驱动子系统**：块 I/O 缓冲、virtio-blk MMIO/PCI 驱动、PCI 总线枚举。
6. **中断与异常处理子系统**：系统调用分发、时钟中断、外部中断、基于 VMA 的缺页异常处理。
7. **进程间通信与网络子系统**：管道、本地 Socket 通信、信号机制。
8. **同步与锁子系统**：自旋锁、睡眠锁、信号量、Futex。

## 三、 各个子系统的实现完整程度

- **启动引导**：实现了 RISC-V 和 LoongArch 的单核完整启动流程。LoongArch 架构下的非主核初始化代码缺失，多核启动不完整。
- **内存管理**：实现了伙伴系统和虚拟内存管理。Slab 分配器代码存在但未在初始化中启用，未实现写时复制（COW）和延迟分配（Lazy Allocation）。
- **进程管理**：实现了基础的轮转调度和进程管理，支持通过 clone 创建线程。未实现优先级调度和完全公平调度（CFS）。
- **文件系统**：实现了 VFS 层和 EXT4 适配，支持常规文件与目录操作。VFS 层当前仅挂载 EXT4 一种文件系统类型。
- **设备驱动**：实现了双架构的 virtio-blk 块设备驱动和 LoongArch PCI 总线扫描。未实现网络或显示设备驱动。
- **中断与异常**：实现了双架构的 Trap 处理和缺页异常处理。缺页异常仅处理 mmap 区域，未覆盖常规用户内存。
- **IPC 与网络**：实现了管道、信号和本地 Socket。Socket 未实现网络协议栈，仅限本地进程间通信。
- **同步原语**：实现了自旋锁、睡眠锁、信号量和 Futex。信号量 V 操作存在惊群效应。

## 四、 各个子系统的优缺点及实现细节

### 4.1 启动引导子系统
- **实现细节**：RISC-V 通过 SBI 获取 hartid 并切换至 S 模式；LoongArch 配置直接映射窗口（DMWIN）和 CRMD 寄存器。
- **优点**：双架构初始化流程清晰，通过目录隔离了架构相关代码。
- **缺点**：LoongArch 非主核初始化代码缺失，多核支持不完整。

### 4.2 内存管理子系统
- **实现细节**：伙伴系统采用类线段树结构，元数据存储在物理内存起始处；虚拟内存支持 Sv39（RISC-V）和四级页表（LoongArch）。
- **优点**：伙伴系统分配与回收时间复杂度为 O(log n)，非侵入式设计节省管理空间。
- **缺点**：物理页分配器的锁被注释，存在多核竞态风险；`kcalloc` 参数计算存在逻辑错误；Slab 分配器未启用。

### 4.3 进程管理子系统
- **实现细节**：使用全局静态进程表，轮转调度（RISC-V 每 5 个时钟中断让出，LoongArch 每 10 个让出），ELF 加载支持辅助向量（auxv）。
- **优点**：支持 Linux 风格的 clone 标志位，ELF 加载器支持动态链接器解析。
- **缺点**：调度算法单一，缺乏优先级和公平性保障；进程表大小固定，无法动态扩展。

### 4.4 文件系统子系统
- **实现细节**：VFS 层定义统一文件操作接口，通过适配层调用第三方 lwext4 库实现 EXT4。
- **优点**：支持硬链接、符号链接、文件截断等丰富的文件操作，提供 `/proc` 和 `/dev` 虚拟文件。
- **缺点**：EXT4 操作使用单一信号量互斥，并发性能受限；VFS 层仅支持单一文件系统类型。

### 4.5 设备驱动子系统
- **实现细节**：块 I/O 使用 LRU 双向链表管理，RISC-V 使用 MMIO 访问 virtio，LoongArch 通过 PCI 总线枚举并使用 MMIO 映射。
- **优点**：实现了完整的 PCI 总线扫描和 virtio 队列初始化。
- **缺点**：仅支持块设备，缺乏其他外设驱动。

### 4.6 中断与异常处理子系统
- **实现细节**：RISC-V 使用 PLIC，LoongArch 使用 APIC+EXTIOI。缺页异常遍历进程 VMA 进行按需映射。
- **优点**：双架构中断控制器适配完整，Trap 处理路径清晰。
- **缺点**：缺页异常仅处理 mmap 区域，未实现非 mmap 区域的缺页处理。

### 4.7 IPC 与网络子系统
- **实现细节**：Socket 使用全局数组和位图管理，消息使用预分配池，支持 TCP/UDP 语义的本地通信。
- **优点**：提供了较为完整的 Socket API（bind, listen, accept 等）。
- **缺点**：无网络协议栈，UDP 分支存在死代码，锁粒度较粗。

### 4.8 同步与锁子系统
- **实现细节**：自旋锁支持嵌套禁用中断，Futex 支持超时等待。
- **优点**：提供了 Futex 以支持用户态高级同步原语。
- **缺点**：信号量 V 操作唤醒所有等待进程，存在惊群效应。

## 五、 OS 内核本身的实现完整度

基于项目声明的功能目标（双架构支持、EXT4 文件系统、Linux 兼容系统调用），内核在功能广度上覆盖了启动、内存、进程、文件、设备等核心模块。系统调用接口实现了 80 余个，涵盖了进程控制、文件操作、内存映射等基础类别。但在功能深度上，部分高级特性（如 COW、网络协议栈、高级调度算法）未实现，且多核环境下的并发安全性存在缺陷（关键锁被注释）。整体而言，内核具备了运行基础用户态程序的静态结构，但在高并发场景下的稳定运行能力尚有欠缺。

## 六、 动态测试的设计和结果

- **测试设计**：项目提供了 51 个 RISC-V 预编译测试二进制文件，覆盖 brk、clone、execve、mmap、pipe 等系统调用。同时包含 BusyBox 二进制以测试综合系统调用兼容性。
- **测试结果**：由于构建环境中缺乏 `mount` 权限，无法将用户态程序写入 EXT4 磁盘镜像，导致 QEMU 启动测试未能完成。内核二进制文件（约 278KB）编译链接成功，但在 `userinit()` 阶段因无法加载 init 进程而未能进入用户态。因此，动态运行测试结果缺失，仅验证了静态编译与链接的正确性。

| :--- | :--- | :--- | :--- |
| **内存管理** | 已实现。基于伙伴系统管理物理页，基于多级页表管理虚拟内存。未实现 Slab、COW 和延迟分配。 | 伙伴系统采用 O(log n) 复杂度的线段树结构；物理页分配器的锁被注释；`kcalloc` 存在参数计算错误。 | 物理内存分配算法设计合理，但多核并发安全性存在严重隐患，部分内存管理高级特性缺失。 |
| **进程管理** | 已实现。基于全局进程表和轮转调度，支持 clone 创建线程。未实现优先级调度。 | 调度器基于固定时钟中断次数让出 CPU；ELF 加载器支持 auxv 和动态链接器。 | 具备基础的进程与线程管理能力，但调度策略过于简单，无法满足复杂负载下的公平性与实时性需求。 |
| **文件系统** | 已实现。实现了 VFS 和 EXT4，覆盖常规文件与目录操作。未实现多文件系统挂载。 | 通过适配层集成 lwext4 库；EXT4 操作使用单一信号量互斥；支持 `/proc` 和 `/dev` 虚拟节点。 | 文件系统功能覆盖面广，支持复杂的文件操作，但全局互斥锁设计限制了 I/O 并发性能。 |
| **交互设计** | 已实现。通过 80+ 个 Linux 兼容系统调用提供用户态接口，支持 BusyBox 运行。 | 系统调用号采用 Linux 标准编号；部分系统调用（如 `set_robust_list`, `prlimit64`）仅为返回 0 的存根。 | 系统调用接口设计注重兼容性，能够支撑常规用户态工具链，但存根实现可能影响复杂应用的运行。 |
| **同步原语** | 已实现。提供自旋锁、睡眠锁、信号量和 Futex。 | 自旋锁支持中断嵌套；Futex 支持超时等待；信号量 V 操作唤醒所有等待进程。 | 提供了构建用户态同步机制的基础原语，但信号量实现存在惊群效应，缺乏读写锁等高级原语。 |
| **资源管理** | 部分实现。通过引用计数管理文件描述符，通过全局数组管理进程和 Socket。 | 文件表和进程表大小在编译时固定（`NFILE`, `NPROC`）；Socket 使用位图管理。 | 资源分配机制简单直接，但缺乏动态扩展能力和严格的资源配额限制（如 ulimit）。 |
| **时间管理** | 已实现。支持时钟中断、时间片轮转及时间查询系统调用。 | 实现了 `gettimeofday`, `clock_gettime`, `nanosleep` 等系统调用；RISC-V 与 LoongArch 时间片长度不同。 | 提供了基础的时间统计与睡眠功能，能够满足常规进程的时间管理需求。 |
| **系统信息** | 已实现。提供 `uname`, `sysinfo`, `/proc/interrupts` 等信息查询接口。 | `uname` 返回固定的内核版本信息；`/proc/interrupts` 动态展示中断统计。 | 具备基础的系统和硬件信息暴露能力，有助于用户态程序进行环境探测。 |
| **设备与驱动** | 部分实现。实现了双架构 virtio-blk 驱动和 PCI 总线枚举。 | LoongArch 实现了完整的 PCI 配置空间读取和 BAR 初始化；仅支持块设备。 | 块设备驱动实现完整，能够支撑文件系统运行，但缺乏网络和显示等外围设备驱动。 |

## 八、 总结评价

AddddOS 是一个在功能广度上进行了大规模扩展的操作系统内核项目。其核心工程贡献在于实现了 RISC-V 与 LoongArch 双指令集架构的底层适配，集成了完整的 EXT4 文件系统，并构建了包含 80 余个系统调用的 Linux 兼容接口层。项目在目录结构组织、构建系统现代化（CMake）以及双架构代码复用方面展现了较好的工程素养。

然而，该项目在实现深度和工程质量上存在明显不足。首先，多核环境下的并发安全性未得到保障，核心内存分配器的锁机制被注释，存在严重的竞态风险。其次，部分已编写的模块（如 Slab 分配器）未被实际启用，且代码中存在逻辑错误（如 `kcalloc` 参数计算）和死代码。最后，调度算法、缺页异常处理等核心机制较为基础，缺乏对高级特性（如 COW、延迟分配、优先级调度）的支持。总体而言，该项目具备运行基础用户态程序的结构能力，但在系统稳定性、并发性能和功能深度方面仍需进一步打磨。
